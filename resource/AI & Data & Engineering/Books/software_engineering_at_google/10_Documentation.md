# 10. Documentation

## 챕터 개요 (3줄 요약)
- 문서화는 작성자에게 즉각적 보상이 없지만, 단 한 번 쓰여 수백·수천 번 읽히므로 시간·규모에 걸쳐 막대한 가치를 낳는다.
- 핵심 통찰은 "문서는 코드와 같다(Documentation is like code)" — 소유자·소스컨트롤·리뷰·버그추적을 갖추고 개발자 워크플로에 통합되면 품질이 개선된다.
- AI 시대에 코드가 폭증할수록, 설계 의도와 사용법을 청중 관점에서 기록하는 문서화 역량이 조직 확장과 유지보수의 레버리지가 된다.

---

## 1. Why Documentation Is Needed (왜 필요한가)
> 문서의 이점은 모두 하류(downstream)에 있어 작성자에게 즉시 돌아오지 않지만, 테스트처럼 시간이 지나며 투자비를 회수한다.

- 이점: 코드·API 이해도 향상, 설계 의도("왜 이렇게 했는가") 기록, 온보딩 비용 절감, 반복 질문 감소.
- 작성자에게도 이득: API 설계 검증("설명 못 하면 잘 설계 못 한 것"), 유지보수 로드맵, 전문성 인상, 질문 감소.
- 문제 인식: 2010년대 후반 문서화 상태는 1980년대 후반 테스트 상태와 유사 — 중요성은 알지만 조직적 인정이 부족.

---

## 2. Documentation Is Like Code (문서는 코드와 같다)
> 문서도 규칙·문법·스타일을 가진 도구이며, 코드처럼 다뤄야 한다.

- 갖춰야 할 것: 내부 정책/규칙, 소스컨트롤, 명확한 소유자, 변경 리뷰, 버그 추적, 주기적 평가.
- 사례 GooWiki: 소유자·계층·프로세스 없는 위키가 노후화·중복·왜곡 → 개발자 설문 1위 불만이 됨.
- 해결: 문서를 소스컨트롤로 이동(canonical location), Markdown 도입, g3doc로 코드 옆에 문서 배치 → 코드와 문서를 같은 변경에서 갱신.
- go/ links로 정규(canonical) 문서 확립을 용이하게 함.

---

## 3. Know Your Audience (청중을 알라)
> 가장 큰 실수는 "자신만을 위해" 쓰는 것 — 먼저 주 청중을 식별하고 그에 맞춰 써라.

- 청중 구분: 경험 수준, 도메인 지식, 목적(고객 vs 제공자).
- Seekers(원하는 것을 아는 사람): 일관성(consistency)이 핵심 — 같은 포맷으로 빠르게 스캔 가능.
- Stumblers(불명확한 사람): 명확성(clarity)이 핵심 — 개요/도입부 제공, "TL;DR" 문구로 부적합 독자 안내.
- 잘 쓸 필요는 없다 — "읽을 수 있으면 쓸 수 있다"; 짧게 쓰려면 오히려 더 길게 쓴 뒤 편집해야 함(Pascal: "시간이 더 있었다면 더 짧게 썼을 것").

---

## 4. Documentation Types (문서 유형)
> 문서는 단일 목적을 가져야 하며, 유형을 섞지 말 것.

- Reference documentation(코드 주석): 가장 흔함. API 주석(구현 세부 불필요) vs 구현 주석(도메인 지식 가정 가능).
  - File/Class/Function 주석: 함수 주석은 능동 동사로 시작("Merges...", "Creates...") → 일관성으로 seeker가 빠르게 스캔. 인위적 "Returns:/Throws:" 보일러플레이트보다 단일 산문이 더 명확.
- Design docs: 주요 프로젝트 착수 전 승인 필요 — 목표·구현전략·핵심 설계결정과 트레이드오프·대안 + 보안/i18n/프라이버시 검토; 역사적 기록·성공 측정 기준.
- Tutorials: "Hello World" — 신규 팀 합류 시 작성 적기; 모든 사용자 단계를 명시적으로 번호화(시스템 동작은 번호 안 매김).
- Conceptual documentation: 가장 쓰기 어렵고 가장 방치됨 — 완전성·정확성을 일부 희생하고 명확성·이해 강조("주석이 단위 테스트면 개념 문서는 통합 테스트").
- Landing pages: 목적만 명확히 하고 다른 페이지로 링크만 — 교통경찰 역할 외 일을 하면 실패; 팀 페이지와 고객 페이지를 분리.

---

## 5. Reviews & Philosophy (리뷰와 철학)
> 문서도 리뷰가 필요하며, 좋은 문서는 완전성·정확성·명확성의 균형이다.

- 세 가지 리뷰: 기술 리뷰(정확성, SME), 청중 리뷰(명확성, 비전문가), 작문 리뷰(일관성, 테크니컬 라이터).
- 5W: HOW에만 매몰되지 말고 도입부 두 단락에 WHO(청중)/WHAT(목적)/WHEN(날짜)/WHERE(위치)/WHY(요지)를 다뤄라.
- 모든 문서는 시작·중간·끝(섹션)을 가져야 하며, 문서화에서는 중복(redundancy)이 종종 유용(도입에서 요약 후 본문 상술).
- 좋은 문서의 3요소: 완전성(Completeness)·정확성(Accuracy)·명확성(Clarity) — 셋을 동시에 다 얻기 어려워 문서의 의도된 역할에 맞게 조정.
- Deprecating: "freshness date"(예: reviewed '2019-02-27')와 버그 추적으로 노후 문서를 관리.

```
Quality Trade-off Triangle (a doc rarely maximizes all three)
        Completeness
           /    \
          /      \
   Accuracy ---- Clarity
   * pick focus per document TYPE
```

---

## 6. When Do You Need Technical Writers? (테크니컬 라이터)
> 한정된 자원이므로, 엔지니어가 평소 못 하는 일 — 특히 API 경계를 넘는 문서 — 에 집중해야 한다.

- 팀 내부 문서는 엔지니어가 잘 쓴다(피드백 루프·도메인 지식이 즉각적); 다른 청중용 문서가 어렵다.
- 중요 프로젝트에 라이터를 붙이면 "엔지니어가 문서 안 써도 됨"이라는 역인센티브 발생 — 라이터는 도메인 외부인으로서 가정을 검증하는 역할.

---

## Summary (핵심 정리)
- 문서화는 시간·규모에 걸쳐 매우 중요하며, 문서 변경은 기존 개발자 워크플로를 활용해야 한다.
- 문서는 단일 목적에 집중하고, 자신이 아니라 청중을 위해 써라.
- AI 시대 코드 폭증 속에서 설계 의도·사용법을 청중 관점에서 기록하는 것이 조직 확장·유지보수의 시니어 레버리지다.
