# 13. Microkernel Architecture Style

## 챕터 개요 (3줄 요약)

- 마이크로커널(플러그인) 아키텍처는 코어 시스템과 플러그인 컴포넌트 두 가지로 구성되는 모놀리식 스타일이다.
- 제품 기반 앱(IDE·브라우저)과 커스터마이징이 필요한 비즈니스 앱(보험·세금)에 적합하다.
- 변동성 높은 코드를 플러그인으로 격리해 확장성·유지보수성·테스트성을 높인다.

---

## 1. Topology / Style Specifics (토폴로지·세부)

> 코어 시스템(최소 기능 또는 해피 패스)과 독립 플러그인으로 나뉜다.

### Core System (코어 시스템)

시스템 실행에 필요한 최소 기능, 또는 커스텀 처리가 거의 없는 해피 패스. 애플리케이션의 순환 복잡도를 코어에서 제거해 플러그인으로 옮긴다(예: Eclipse 코어는 기본 텍스트 에디터). 코어는 계층형이나 모듈러 모놀리스, 또는 도메인 서비스로 구현 가능. 플러그인 레지스트리로 적절한 플러그인을 찾아 호출(Going Green 기기 평가 예시).

```
        +----------------+
        |  Core System   |
        +----------------+
        /    |     |     \
    [Plugin][Plugin][Plugin][Plugin]
   (self-contained, no inter-dependencies)
```

### Plug-In Components (플러그인 컴포넌트)

자기완결적·독립적이며 변동성 높은 코드를 격리. 플러그인 간 의존성이 없는 것이 이상적. 통신은 보통 점대점(메서드 호출). 컴파일 기반(전체 재배포 필요) 또는 런타임 기반(OSGi·Jigsaw·Prism 프레임워크로 무중단 추가/제거). 공유 라이브러리(JAR/DLL/Gem)나 네임스페이스(app.plug-in.<domain>.<context>)로 구현. REST·메시징으로 원격 플러그인 호출도 가능하나, 코어가 모놀리식이라 여전히 quantum=1이며 분산 아키텍처가 되어 복잡·비용 증가.

### Spectrum of "Microkern-ality"

플러그인을 지원하는 모든 시스템이 마이크로커널은 아니다. 코어에 독립 기능이 얼마나 있는지가 척도. '순수' 마이크로커널(Eclipse, linter)은 코어 기능이 거의 없고, 웹 브라우저는 플러그인 없이도 완전 작동(오른쪽 끝).

### Registry & Contracts (레지스트리·계약)

레지스트리는 플러그인 이름·계약·접근 프로토콜 정보를 담는다(단순 맵 ~ ZooKeeper/Consul). 계약은 보통 도메인 전반 표준이며 동작·입출력 데이터 포함. 제3자 플러그인은 어댑터로 표준 계약에 맞춘다.

## 2. Data / Cloud / Risks / Governance / Teams

> 보통 단일 관계형 DB를 쓰며, 플러그인은 코어를 통해 데이터를 받는다(디커플링).

데이터: 플러그인이 중앙 DB에 직접 연결하지 않고 코어가 전달(DB 변경이 코어만 영향). 플러그인은 자체 데이터 스토어 가능. 클라우드: 모놀리식이라 거친 옵션(전체 클라우드, 데이터만 클라우드, 코어 온프레미스+플러그인 클라우드 — 후자는 지연 오버헤드 위험). 위험: Volatile Core(코어가 계속 변하면 철학 훼손), Plug-In Dependencies(플러그인 간 의존은 전이적 의존성 충돌 야기 — 가급적 회피). 거버넌스: 코어 변동성·변경률 체크, 계약 테스트. 팀: 코어/플러그인 분리(enabling 팀의 A/B 테스트, complicated-subsystem 팀의 특화 처리에 적합).

## 3. Architecture Characteristics (특성 평가)

> 단순성·비용이 강점, 확장성·내결함성·탄력성이 약점이며, 유일하게 도메인·기술 분할 둘 다 가능하다.

quantum=1(모든 요청이 코어 경유). 대부분 기술 분할이나 강한 도메인-아키텍처 동형성으로 도메인 분할도 가능(위치/클라이언트별 설정, 커스터마이징 제품). 테스트성·배포성·신뢰성은 3점(플러그인 격리). 모듈성·진화성도 3점(플러그인 추가/제거 용이 — 세금 양식 변경 예시). 응답성 3점(작은 규모, 싱크홀 회피, 불필요 기능 제거 가능 — WildFly 예시).

```
  Simplicity/Cost   ***** (high)
  Scalability/FT/Elasticity * (low, quantum=1)
  Testability/Deployability/Modularity *** 
```

## 4. Examples and Use Cases (예시)

> 소프트웨어 개발 도구와 커스터마이징이 필요한 대형 비즈니스 앱에 흔하다.

예: Eclipse IDE, PMD, Jira, Jenkins, Chrome/Firefox 브라우저. 비즈니스 예: 세금 신고 SW(1040 양식=코어, 각 부속 양식·워크시트=플러그인 → 세법 변경 격리), 보험 청구 처리(관할별 규칙=플러그인 → 규칙 엔진의 Big Ball of Mud 회피). 코어+플러그인 구조가 '커스터마이징'이라는 흔한 도메인 문제와 잘 맞는다.

---

## Summary (핵심 정리)

- 마이크로커널은 안정적 코어 시스템과 독립 플러그인으로 변동성을 격리하는 모놀리식 스타일이다.
- 플러그인은 자기완결적이고 상호 의존이 없어야 하며, 점대점 또는 원격 통신으로 코어와 연결된다.
- 단순·저비용이 강점이고 확장성은 약하나, 커스터마이징·확장성이 필요한 제품/비즈니스 앱에 최적이다.
