# 22. Analyzing Architecture Risk

## 챕터 개요 (3줄 요약)
- 모든 아키텍처는 운영적/구조적 리스크를 동반하며, 이를 정량화·평가·식별하는 것은 아키텍트의 핵심 활동이다.
- 리스크 매트릭스(영향 x 발생 가능성)와 리스크 평가표(Risk Assessment)를 통해 주관적 판단을 객관적 수치로 전환한다.
- 리스크 스토밍(Risk Storming)이라는 협업 기법으로 팀 전체가 리스크를 식별·합의·완화하는 과정을 다룬다.

---

## 1. Risk Matrix (리스크 매트릭스)
> 리스크를 영향(Impact)과 발생 가능성(Likelihood) 두 차원으로 평가하여 주관적 판단을 객관적 수치로 변환한다.

각 차원을 low(1), medium(2), high(3)로 평가한 뒤 두 값을 곱해 리스크 수준을 산출한다. 결과값 1~2는 낮음(녹색), 3~4는 중간(노랑), 6~9는 높음(빨강)으로 분류한다. 색 구분이 어려운 경우를 위해 음영(shading)을 함께 쓰는 것이 좋다. 평가 시에는 영향을 먼저, 발생 가능성을 나중에 고려하며, 가능성이 불확실하면 일단 high(3)로 둔다. 예를 들어 중앙 DB(Database) 가용성은 영향이 high(3)이지만 클러스터 구성으로 발생 가능성이 low(1)라면 최종 리스크는 3(중간)이 된다.

```
                 Likelihood
              Low(1) Med(2) High(3)
Impact Low(1)   1      2      3
       Med(2)   2      4      6
       High(3)  3      6      9

1-2: Low (green) | 3-4: Med (yellow) | 6-9: High (red)
```

---

## 2. Risk Assessments (리스크 평가)
> 컨텍스트(서비스/서브도메인/도메인) 기반으로 아키텍처 전체 리스크를 요약 보고하는 평가표이며, 아키텍처 특성이 훌륭한 평가 기준이 된다.

리스크 기준(criteria)은 표 왼쪽에, 컨텍스트는 상단에 배치한다. 시스템에 가장 중요한 아키텍처 특성(예: scalability, elasticity, data integrity)을 평가 기준으로 삼는 것이 효과적이다. 정량화된 리스크는 기준별 누적값(예: data integrity 17점이 최고 리스크)과 컨텍스트별 누적값(예: customer registration이 최고 리스크)을 동시에 보여줘 우선순위 결정에 유용하다. 이해관계자 발표 시에는 저/중 리스크(noise)를 걸러내고 고 리스크(signal)만 강조해 신호 대 잡음비(signal-to-noise ratio)를 높일 수 있다.

### 리스크의 방향(Direction)
스냅샷만으로는 리스크가 개선/악화 중인지 알 수 없다. Chapter 6의 적합성 함수(fitness function)를 통한 지속 측정으로 추세를 파악한다. 위로 향한 삼각형은 악화, 뒤집힌 삼각형은 개선, 원은 변화 없음을 의미하며, 기호 사용 시 항상 범례(key)를 포함한다.

## 3. Risk Storming (리스크 스토밍)
> 단일 차원(컨텍스트 또는 기준)의 아키텍처 리스크를 여러 아키텍트와 개발자가 협업으로 식별·합의·완화하는 활동이다.

아키텍트 혼자서는 리스크 영역을 놓치거나 시스템 전체를 알지 못하므로 협업이 필요하다. 시니어 개발자와 테크 리드를 포함하면 구현 관점의 통찰과 아키텍처 이해도가 높아진다. 모든 단계에서 포괄적/컨텍스트 아키텍처 다이어그램(Chapter 23 참고)을 활용하며, 진행자(facilitator)가 참가자에게 다이어그램을 배포한다. 가능하면 한 번에 단일 기준/컨텍스트만 분석해 집중도를 높인다.

```
Risk Storming Phases
[Phase 1] Identification -> individual, unbiased
[Phase 2] Consensus      -> collaborative agreement
[Phase 3] Mitigation     -> collaborative solutions
```

### Phase 1: Identification (식별)
참가자가 개별적으로, 다른 사람에게 영향받지 않고 리스크를 식별한다. 진행자가 다이어그램·기준·일정이 담긴 초대를 보내고, 참가자는 리스크 매트릭스로 분석해 low/med/high를 색 스티커(녹/노/빨)에 적는다. 여러 차원을 동시에 볼 경우 스티커에 해당 기준을 함께 적는다.

### Phase 2: Consensus (합의)
큰 다이어그램에 스티커를 붙여 팀 전체가 리스크 수준에 합의한다. 의견이 일치하면 추가 논의가 불필요하지만, 차이가 있으면 토론으로 조정한다(예: Elastic Load Balancer를 클러스터링 근거로 medium 합의). 한 명만 식별한 리스크도 중요한 통찰일 수 있다(Push Expansion Servers 경험 기반 high). 모르는 기술은 자동으로 최고 리스크(9)를 부여한다(예: Redis cache를 모르는 개발자 → 9), 이는 개발자 참여의 가치를 보여준다.

### Phase 3: Risk Mitigation (완화)
식별된 리스크를 줄이거나 제거하는 협업 단계로, 보통 추가 비용이 발생하므로 의사결정 권한이 있는 비즈니스 이해관계자가 참여해야 한다. 예: DB 가용성 리스크를 $50,000 클러스터링 대신 $16,000 도메인 기반 분리로 절충해 비용과 리스크의 트레이드오프를 조율한다.

## 4. User-Story Risk Analysis (사용자 스토리 리스크 분석)
> 동일한 리스크 매트릭스를 사용해 이터레이션 내 사용자 스토리 완료 리스크를 분석할 수 있다.

스토리가 이터레이션 내 미완료될 경우의 영향과 미완료 가능성을 평가하여 고리스크 스토리를 식별·추적하고 우선순위를 조정한다.

---

## 5. Risk-Storming Use Case (적용 사례: 간호 핫라인 진단 시스템)
> 콜센터 간호사가 환자를 상담하는 HIPAA 준수 시스템을 사례로, 가용성·탄력성·보안 세 차원의 리스크 스토밍을 진행한다.

시스템은 외부 진단 엔진(초당 500건 처리), 250명 간호사 및 수십만 셀프서비스 환자, 의료기록 교환(간호사만 접근), HIPAA(Health Insurance Portability and Accountability Act) 준수, 독감/COVID 시 고부하 대응이 요구된다.

### Availability (가용성)
중앙 DB는 high(6), 진단 엔진은 미지의 가용성으로 high(9), 의료기록 인터페이스는 low(2)로 식별됐다. DB를 간호사 프로필용(클러스터)과 케이스 노트용(단일)으로 분리해 완화했다. 외부 시스템은 SLA(Service-Level Agreement)/SLO(Service-Level Objective)를 조사해(진단 엔진 99.99%, 의료기록 99.90%) 리스크를 제거했다.

### Elasticity (탄력성)
셀프서비스 부하로 진단 엔진 인터페이스가 high(9)로 식별됐다. API Gateway와 진단 엔진 사이에 비동기 큐(messaging)로 백프레셔(backpressure)를 두고, 간호사와 셀프서비스 요청을 분리하는 Ambulance 패턴(두 채널)을 적용했다. 추가로 발병 관련 질문을 캐싱하는 Diagnostics Outbreak Cache Server를 신설해 진단 엔진 호출을 줄였다.

### Security (보안)
모든 호출(셀프서비스/관리자/간호사)이 단일 API Gateway를 통과하는 점이 high(6) 보안 리스크로 식별됐다. 사용자 유형별로 별도 API Gateway를 두어 비(非)간호사 호출이 의료기록 인터페이스에 절대 도달하지 못하게 함으로써 HIPAA 리스크를 완화했다.

---

## Summary (핵심 정리)
- 리스크 스토밍은 일회성이 아니라 시스템 생명주기 전반에 걸쳐 리스크를 식별·완화하며 프로덕션 전에 문제를 발견하는 지속적 프로세스다.
- 수행 빈도는 변경 빈도, 아키텍처 리팩토링, 점진적 개발 등에 따라 달라지며, 보통 주요 기능 추가 후나 이터레이션 종료 시 특정 차원에 대해 수행한다.
- 매트릭스·평가표·스토밍을 결합하면 리스크 추적, 아키텍처 개선, 이해관계자 협상을 효과적으로 구조화할 수 있다.
