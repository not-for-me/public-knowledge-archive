# 04. Prompt Structure and Formatting

## 챕터 개요 (3줄 요약)

- 프롬프트의 시각적·구문적 구조는 단어 자체만큼이나 모델이 무엇에 어텐션하는지를 좌우하므로, 구성요소를 어떻게 "배열"하느냐가 핵심 다음 과제다.
- 구분자(delimiter)와 정보 순서가 모델 어텐션을 안내하며, 형식의 작은 변화가 명확성과 출력 일관성에 큰 차이를 만든다.
- 프롬프트를 신중히 배열된 구조로 다루고 모든 구조 요소를 모델이 가중할 신호로 취급하면, 건축가가 평면도를 읽듯 어떤 프롬프트의 레이아웃도 읽어낼 수 있다.

---

## 1. Visual Hierarchy and Section Markers

> 잘 설계된 문서에서 눈이 헤더를 훑고 본문에 정착하듯, LLM도 인간 글쓰기 관습에서 학습한 동일한 시각적 위계 감각을 내면화한다.

- 헤더는 새 주제의 시작을, 대문자 줄은 비범한 가중치를, 공백은 한 의미 블록의 끝과 다음의 시작을 모델에게 알린다 — 인간 협업자에게 명확히 쓸 때의 형식 습관이 모델 작업에도 직접 전이된다.
- 적용지침: "Instructions"나 "Examples" 같은 섹션 헤딩을 그 아래 내용과 함께 읽히는 레이블로 활용하라.
- 적용지침: 가장 단순한 규율은 "서두르는 독자"를 상상하는 것이다 — 동료가 2초간 훑어 질문 부분과 데이터 부분을 즉시 알아볼 수 없다면 모델도 똑같이 어려움을 겪는다.
- 적용지침: 대문자·들여쓰기를 절제하라 — 들여쓴 블록은 중첩 컨텍스트를, 전체 대문자는 강조를 신호하지만, 남용하면 단서가 희석되고 모든 줄이 외치는 프롬프트는 모델이 그 관습을 무시하게 가르친다.
- 형식 단서의 절제된 사용이 열성적 사용보다 더 강한 결과를 낳는 경향이 있다.

## 2. Delimiters, Tags, and Structural Syntax

> 위계는 프롬프트에 형태를, 구분자(delimiter)는 경계를 부여한다 — 마커 없이 두 텍스트가 붙으면 모델은 어디서 하나가 끝나는지 추측해야 하고, 그 추측 실패 비용은 클 수 있다.

- 지시가 예시 데이터로 새어 들어가면 모델이 입력의 일부로 취급할 떠도는 내용이 되고, 입력 데이터가 지시로 새어 들어가면 모델이 실행하려는 명령으로 재해석되는데 — 이것이 다수의 프롬프트 인젝션(prompt injection) 공격을 일으키는 메커니즘이다.
- 적용지침: 가장 흔한 두 관습으로 경계를 그어라 — `<context>...</context>` 같은 XML 스타일 태그와, 코드·데이터 섹션을 둘러싸는 트리플 백틱(```) 펜스. 각각 모델이 의존할 명확한 경계를 준다.
- 적용지침: Alpay & Alpay (2025)는 XML 프롬프팅을 명시적 태그가 각 입력에 레이블 역할을 부여해 파싱 가능하고 스키마 준수적인 출력으로 모델을 유도하는 "문법 제약 상호작용"으로 보았다 — 구조화 태그가 실무 일관성을 높인다.
- 적용지침: 프롬프트가 수백 토큰을 넘어 유지보수해야 할 경우, 마크다운보다 XML 스타일 태그를 선호하라 — 명명·짝지음이 가능해 모델과 하방 도구가 장기적으로 일관되게 해석한다(짧은 구조화 지시엔 마크다운 헤더·코드 펜스가 충분).
- 적용지침: 구분자 선택은 모델·작업에 의존한다 — He et al. (2024a)에 따르면 GPT-3.5는 여러 벤치마크에서 JSON을, GPT-4는 마크다운 형식에서 최고 추론 정확도를 보였다.
- 적용지침: 작은 모델은 형식 민감도가 극심해 템플릿에 따라 정확도가 최대 40% 출렁이고, 큰 모델은 더 강건하지만 강건함을 면역으로 혼동하지 말 것 — 즉 대상 모델별로 형식을 검증하라.
- 적용지침: 지시와 데이터를 태그로 명확히 분리하면(예: `<instructions>...</instructions>`와 `<article>...</article>`), 기사 본문이 명령형 언어를 포함하더라도 모델이 본문을 지시로 혼동하지 못한다.

```
[Delimiter pattern]
<instructions> Summarize the article in three sentences. </instructions>
<article> [full article text] </article>
  -> tags create unambiguous walls -> model cannot confuse body with instructions
```

## Summary (핵심 정리)

- 프롬프트의 레이아웃(헤더·공백·대소문자)은 모델 어텐션을 안내하는 신호이므로, 절제된 시각적 위계가 명확성을 높인다.
- XML 태그·트리플 백틱 같은 구분자로 지시와 데이터의 경계를 그으면 일관성이 오르고 프롬프트 인젝션 위험이 줄어든다.
- 형식 효과는 모델 의존적이며(GPT-3.5는 JSON, GPT-4는 markdown 선호; 작은 모델은 최대 40% 변동), 유지보수 프롬프트엔 명명 가능한 XML 태그가 유리하므로 대상 모델별로 검증하라.
