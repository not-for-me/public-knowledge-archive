# 27. Retrieval-Augmented Generation

## 챕터 개요 (3줄 요약)

- 검색 증강 생성(RAG, Retrieval-Augmented Generation)은 검색 엔진과 언어 모델을 결합해 프롬프트가 모델 외부에서 끌어온 정보를 담게 한다 — 훈련 데이터의 제한성과 컨텍스트 윈도우의 유한 크기 두 한계를 동시에 다룬다.
- 아키텍처는 원리상 단순하지만 행동을 지배하는 프롬프팅 결정이 실제 엔지니어링이 일어나는 곳이며, 검색·생성 단계가 신중히 설계된 텍스트로 어떻게 소통하는지를 살핀다.
- 프롬프트 설계가 검색 파이프라인의 모든 계층을 통해 파급되어 사용자에게 추적 가능한 답으로 도달하는 방식을 이해한다.

---

## 1. The RAG Pattern

> 표준 RAG 시스템은 예측 가능한 사이클을 따른다 — 사용자 질문이 검색 질의로 변환되어 벡터 스토어·문서 인덱스로 보내지고 후보 구절을 반환받아, 가장 관련 있는 구절이 모델이 읽고 최종 답을 낼 프롬프트로 조립된다(Figure 5-6: 검색 단계 + 생성 단계).

- 적용지침: 각 핸드오프가 프롬프트 설계가 중요한 지점인데, 텍스트가 모든 컴포넌트 간 연결 조직이기 때문이다. RAG는 모델 가중치의 parametric 지식과 dense retrieval로 접근하는 non-parametric 메모리를 결합하는 방법으로 처음 형식화됐고(Lewis et al. 2020), 주변 도구가 진화해도 그 기본 형태가 유지됐다.
- 적용지침: RAG가 모델 단독 생성을 능가하는 이유는 단순 제약으로 귀결된다 — 모델은 훈련된 것만 알 수 있고 가장 유용한 정보는 훈련 세트 밖에 산다. 내부 문서·최근 사건은 범용 모델이 가질 수 없는 지식의 흔한 예다. 컨텍스트를 동적으로 만들어 시스템이 모델 구축 시 없던 콘텐츠를 답할 수 있고, 동시에 모델 주장을 운영자가 통제하는 특정 증거 베이스에 가둘 수 있다.
- 적용지침: 검증 가능성(verifiability)이라는 동등하게 중요한 이점 — 모델이 검색 구절에서 쓰면 답 검증이 가능해진다. 같은 구절을 읽고 모델이 그것을 충실히 표현했는지 확인할 수 있는데, 순수 생성 답은 그런 핸들이 없고 이 추적 불가능성이 정확성이 중요한 환경에서 무보강 모델 출력이 위험한 이유 중 하나다.
- 적용지침: 운영자는 아무것도 재훈련 않고 기저 말뭉치를 갱신할 수 있어 RAG가 배포 시스템을 현재 정보에 정렬 유지하는 비범하게 반응적인 방법이 된다 — 새 정책 문서 추가·낡은 사실 수정·새 제품 라인 확장이 새 모델 릴리스 없이 인덱싱 문제가 된다.
- 적용지침: 아키텍처 프레이밍에 주목하라 — 비검색 환경에선 프롬프트가 엔지니어가 한 번 쓰는 것이지만, RAG 파이프라인에선 프롬프트가 런타임에 여러 움직이는 부분에서 조립되며 그 부분 대부분이 프롬프트로 형성된다(리트리버로 가는 질의도 프롬프트로 만들어지고, 생성 모델 지시도 프롬프트이며, 각 검색 구절 주위의 경계 태그도 모델에 주변 텍스트 해석법을 알리는 일종의 마이크로 프롬프트).

## 2. Prompting the Retrieval Stage

> 검색 단계 자체가 프롬프팅 문제다 — 실행하는 모델이 최종 답을 쓰는 것보다 작거나 다를 수 있어도 그렇다. 사용자 질문은 그 자체로 좋은 검색 질의가 되는 경우가 드물다.

- 적용지침: 사람은 약어로 말하며 대화 이력에 의존하는 대명사·맥락 참조를 쓰는데 리트리버는 그것을 못 본다. 검색 프롬프트의 일은 그 느슨한 인간 문구를 검색 인덱스가 매칭할 수 있는 것으로 변환하는 것이다.
- 적용지침: 질의 변환(query transformation)에 두 지배 변형 — (1) query rewriting: 사용자 입력을 누락 대화 컨텍스트를 포함한 자기 완결적 요청으로 재구성. 밀접 관련 query expansion은 실제 문서가 사용자와 다른 어휘를 쓸 때 recall을 높이는 관련 용어·가설 답을 더한다. (2) decomposition: 다부분 질문을 작은 질의로 쪼개 각각 한 사실을 타깃하고 생성 전 결과를 병합. 검색 품질을 보상 신호로 훈련한 작은 재작성 모델도 리트리버에 더 잘 정렬되고 하방 답 정확도를 높이는 질의를 낸다.
- 적용지침 예시: "당신은 검색 질의 생성기다. 사용자 메시지와 이전 대화가 주어지면, 사용자 의도를 포착하는 단일 자기 완결적 검색 질의를 생성하라. 질문에 답하지 말라. 질의만 출력하라. Conversation: {history} Latest message: {message} Search query:"
- 적용지침: 이 프롬프트의 출력이 리트리버에 먹인다 — 지시가 얼마나 제한적인지 주목하라. 모델은 답하려는 본능을 억누르고 질의 문자열만 내라고 들었다. 그 제약 없이는 모델이 흔히 도움 설명 모드로 빠져 파이프라인을 깬다. 다른 면에서 견고한 검색 시스템이 재작성기가 출력을 추가 자연어 패딩으로 시작해 하방 검색 엔진이 그것을 질의의 일부로 다뤄 실패하는 것을 봤다 — 이를 고치는 비용은 보통 프롬프트의 단일 문장이고, 배포 전 잡는 가치는 막대하다.
- 적용지침: 분해는 짧은 별도 노트가 필요 — 사용자가 두 독립 사실을 담은 질문을 하면 결합 질의 검색이 어느 아이디어도 강하게 매칭 안 하면서 둘 다에 느슨히 관련된 문서를 흔히 반환한다. 질문을 집중 하위 질의로 쪼개고 중복 제거된 결과를 연결하면 생성 모델에 더 완전한 증거 베이스를 준다.
- 적용지침: 많은 프로덕션 시스템이 dense 벡터 검색과 고전 키워드 검색을 결합하는데, 둘이 다른 분포에서 검색하기 때문이다 — 벡터 검색은 문구가 달라도 의미 유사성을 잡고, 키워드 검색은 임베딩 모델이 뭉갤 수 있는 정확한 식별자·고유명사를 신뢰성 있게 표면화한다. 리트리버를 구동하는 프롬프트가 두 형태를 한 번에 내도록(패러프레이즈된 자연어 질의 + 고가치 키워드 용어 목록) 쓸 수 있고, 하방 시스템이 각 검색 모드를 돌려 다음 단계에 먹이기 전 후보 집합을 융합한다.

## 3. Assembling Retrieved Context

> 검색이 후보를 반환하면 시스템은 생성 모델이 읽을 단일 프롬프트를 구성해야 하며, 이것이 여러 형식 결정을 담는다.

- 적용지침: 재순위(reranking)가 검색 후에도 중요하다 — 흔한 패턴은 관대하게 검색하고(상위 20개) cross-encoder로 재순위해 최종 프롬프트엔 상위 4~5개만 유지하는 것. 프롬프트를 짧게 유지하고 가장 관련 있는 구절을 경계 근처에 두면 lost-in-the-middle 페널티가 줄어든다(Module 02).
- 적용지침: 중복 제거(deduplication)도 중요한데, 벡터 리트리버가 다른 소스의 거의 동일한 구절을 흔히 표면화하기 때문이다 — 같은 사실을 모델에 네 번 보여주면 증거 추가 없이 컨텍스트 예산을 낭비한다(Figure 5-7: lost-in-the-middle을 막는 구절 재순위).
- 적용지침: 청킹(chunking)은 검색 실행 훨씬 전 많은 결정이 구워지는 곳이다 — 문서를 임베딩 모델이 정확히 요약할 만큼 작고 각 조각이 의미 있는 컨텍스트를 담을 만큼 큰 조각으로 나눠야 한다. 너무 작은 청크는 원 논증을 단편화하고, 너무 큰 청크는 의미 신호를 희석한다. 어림 규칙 — 소스 문서의 구조적 이음새를 따라 청킹하고 청크를 약간 겹쳐 핵심 문장이 경계에서 잘리지 않게 하라. 올바른 크기는 임베딩 모델·소스 말뭉치 평균 밀도에 의존하며 추측보다 측정할 가치가 가장 큰 하이퍼파라미터 중 하나다.
- 적용지침: 형식 관습(Module 04, 07)이 모델이 검색 증거를 주변 지시·질문과 분리하는 데 도움 — 구조 단서 없이는 모델이 지시와 인용 구절의 경계를 흐려, 증거를 지시로 다루거나 지시를 요약할 콘텐츠로 다룬다. `<passage id="P1">` 같은 단순 래퍼도 모델이 역할을 똑바로 유지하는 능력을 실질적으로 향상시킨다.

## 4. Grounding and Hallucination Control

> 검색이 완벽해도 생성 모델은 여전히 방황할 수 있다 — 훈련 시점 지식을 검색 구절과 섞거나 공급된 소스 어느 것도 뒷받침 않는 세부를 발명할 수 있다. 가장 신뢰할 방어는 모델의 일을 명시적으로 만드는 프롬프트다.

- 적용지침: 두 절이 대부분 일을 한다 — (1) grounding clause(접지 절): 모든 주장이 공급 구절에서 나오도록 요구. (2) refusal clause(거절 절): 그 구절이 불충분하면 모델이 답을 거절하게 허용. 답 전반에 짜인 인용(citation)이 사용자에게 각 주장을 소스에 대조 검증할 방법을 준다.
- 적용지침: 각 절이 특정 실패 양식을 다룬다 — grounding clause는 모델이 prior 지식으로 간극을 채우는 것을 막고, refusal clause는 모델이 사용자를 실망시키는 대신 추측하는 것을 막으며, citation은 사용자 검증을 돕고 모델 자체의 soft check로 작동한다(증거를 가리키게 요청하면 증거가 없는 순간이 흔히 표면화).
- 적용지침 예시: "아래 제공된 구절만 사용해 사용자 질문에 답하라. 각 구절에 ID가 있다. 답의 모든 사실 진술에 [ID] 형태로 인용을 붙여라. 구절이 답에 필요한 정보를 담지 않으면 '제공된 소스가 이 질문에 답할 충분한 정보를 담지 않습니다'라고 답하라. 외부 지식을 쓰지 말라. Passages: [P1] {chunk_1} [P2] {chunk_2} Question: {user_question} Answer:"
- 적용지침: 이런 프롬프트가 크기 대비 많은 일을 한다 — "아래 구절만"이라는 문구가 닫힌 범위를 만들고, 문자적 거절 문자열이 증거가 떨어질 때 허용된 출구를 주며, 구조화 인용 형식이 설계를 완성한다(하방 코드가 답을 파싱하고 각 주장을 소스로 추적 가능).
- 적용지침: grounding 작업의 핵심 긴장 — 어떤 프롬프트도 환각을 혼자 완전히 제거하지 못한다. 모델이 답 주장과 다른 말을 하는 구절을 여전히 인용할 수 있고, 그 실패 양식 평가는 검색 품질 평가보다 어렵다(인용이 답을 뒷받침 안 해도 답이 그럴듯해 보일 수 있으므로). 따라서 grounding 프롬프트는 추적성을 높이는 강력한 도구이되 인용-주장 일치를 검사하는 별도 검증과 함께 다뤄야 한다.

## Summary (핵심 정리)

- RAG는 검색 단계(질의→벡터 스토어→후보 구절)와 생성 단계(조립 컨텍스트→접지된 답)로 작동하며, parametric+non-parametric 메모리를 결합해 최신성·검증 가능성·재훈련 없는 갱신을 제공한다(Lewis 2020).
- 검색 단계는 query rewriting/expansion/decomposition으로 느슨한 인간 문구를 변환하되, "질의만 출력" 같은 제한적 지시로 모델이 답하지 않게 하고 dense+키워드 하이브리드를 활용하라.
- 컨텍스트 조립은 재순위(상위 4~5개를 경계 근처)·중복 제거·구조적 청킹·`<passage>` 래퍼로 하고, grounding/refusal 절과 인용으로 환각을 통제하되 인용-주장 일치 검증을 별도로 두라.
