# 18. Multi-Agent Systems

## 챕터 개요 (3줄 요약)

- 일부 문제는 잘 프롬프트된 단일 모델이 낼 수 있는 것을 넘어선다 — 여러 종류의 전문성이 필요하거나 독립적 검토가 이득일 때다. 멀티 에이전트 시스템은 별개 역할의 여러 언어 모델 인스턴스를 조율해 이 한계를 다룬다.
- 협업하는 에이전트를 설계·프롬프트하는 법을 다루며, 지배적 조율 패턴(orchestrator-worker, peer-to-peer, debate)과 멀티 에이전트 설정 고유의 프롬프팅 요구를 살핀다.
- 멀티 에이전트는 특정 문제 부류에서 실질적 능력 향상을 주지만 개별 에이전트의 실패 양식도 배가하므로, 규율 있는 프롬프트 설계가 특히 중요하다.

---

## 1. When to Use Multiple Agents

> 단일 유능 모델이 프롬프트가 명확하고 컨텍스트가 편안히 들어가면 대부분 작업을 잘 처리한다 — 멀티 에이전트 설계는 단일 연속 추론 트레이스가 혼자 활용 못 할 내부 구조를 작업이 가질 때만 매력적이다.

- 적용지침: 특화(specialization)가 첫 조건 — 하위 작업이 별개 전문성을 요할 때(법률 분석 + 코드 생성), 별도 프롬프트의 별도 에이전트가 둘을 한 번에 저글링하는 단일 에이전트보다 더 타이트하고 관련성 높은 컨텍스트를 유지한다. 각 역할 특정 프롬프트가 에이전트가 필요로 하는 prior를 강화하고 아닌 것을 배제해 정확성·일관성을 높인다(비용은 에이전트 간 핸드오프 복잡성 — 정보가 더 이상 연속 트레이스에 머물지 않아 재패키징·재해석돼야 함).
- 적용지침: 병렬성(parallelism)이 둘째 조건 — 하위 목표가 서로 독립이면 동시 실행이 벽시계 시간을 줄이고 각 분기가 자기 좁은 문제에 전체 토큰 예산을 쓰게 한다(독립 하위 주제를 워커가 각각 조사해 코디네이터에 반환하는 연구 작업이 고전적 예; 총비용보다 지연이 중요할 때 유용 — 사용자가 가장 느린 워커만 기다리므로).
- 적용지침: 적대적 검토(adversarial review)가 셋째 동기 — 작성자 에이전트 출력을 읽는 리뷰어 에이전트가 작성자 혼자 탐지 못 할 오류를 잡는다(새 컨텍스트가 다른 prior를 활성화해 새 실패 양식이 가시화). Du et al. (2024)는 여러 모델 인스턴스가 여러 라운드에 걸쳐 제안 답을 토론하면 여러 벤치마크에서 사실·산술 추론 정확도가 향상됨을 보였다 — 향상은 에이전트가 서로의 추론을 비평한 데서 왔고 모델 업그레이드·파인튜닝 불필요.
- 적용지침: 규모(scale)가 동기를 마무리 — 워크로드가 공격적 압축 후에도 단일 컨텍스트 윈도우에 너무 크면 코디네이터에 보고하는 에이전트들로 일을 분할하는 것이 실용적이 된다(Tran et al. 2025: 협력·경쟁 등 모드를 이 기저 동인에 매핑).
- 적용지침 주의: 멀티 에이전트가 항상 옳은 선택은 아니다 — 동등 컴퓨팅 예산에서 단일 에이전트 베이스라인이 multi-hop 추론에서 멀티 에이전트를 같거나 능가하는 경우가 많아, 보고된 일부 이득은 조율의 진짜 비용을 가린 부풀린 토큰 예산에서 왔음을 시사한다. 좋은 습관 — 단일 에이전트로 시작하고 분할이 고칠 구체 실패를 명명할 수 있을 때만 일을 쪼개라. 둘째 에이전트가 왜 필요한지 명명 못 하면 아직 필요 없는 것이며, 멀티 에이전트 다이어그램은 인상적이나 상호작용 역할이 한둘 넘으면 유지보수 부담이 빠르게 복리된다.

## 2. Coordination Patterns

> 에이전트가 스스로를 조직하는 방식이 프롬프트 내용부터 오류 전파까지 시스템의 나머지를 형성한다 — 몇 패턴이 현재 실무를 지배하며 각각 강점과 프롬프팅 요구가 다르다.

- 적용지침: orchestrator-worker 패턴은 한 에이전트를 최상위에 둔다 — 오케스트레이터가 사용자 요청을 해석해 하위 작업으로 분해하고, 각 하위 작업을 집중 응답을 반환하는 워커 에이전트에 라우팅하며, 출력을 최종 답으로 조립한다. 문제 구조가 초기 추론 후에야 명확해질 때 잘 작동한다(오케스트레이터가 이전 결과 관찰 후 다음 하위 작업 계획). Figure 3-8.
- 적용지침: peer-to-peer 협업은 중앙 권위를 제거한다 — 에이전트가 동등하게 공유 채널에서 차례로 게시하며 합의·해결책이 나올 때까지 소통한다. 원리상 잘 확장되지만 명확한 정지 조건을 요구하는데, 오케스트레이터 없이는 어느 에이전트도 작업 완료 질문을 소유하지 않기 때문이다(Tran et al. 2025: 종료 강제 방식에 상당한 변동). Figure 3-9.
- 적용지침: 토론·비평 구조(debate/critique)는 에이전트를 구조화된 대립에 둔다 — 한 에이전트가 답을 제안하고 다른 에이전트가 한두 라운드에 걸쳐 반박·수정하며, 판사(judge)나 고정 최종 라운드가 남은 불일치를 해결한다. 정답이 원리상 검증 가능하나 도달이 어려울 때 가장 강하다(교차 검증이 각 에이전트의 추론을 정밀 검토에 노출). Figure 3-10.
- 적용지침: 각 패턴의 프롬프팅 함의 — 오케스트레이터 프롬프트는 계획 논리·워커 선택 기준을 인코딩하고, 워커 프롬프트는 오케스트레이터 역할로 표류 안 하도록 좁게 유지하라. peer-to-peer 에이전트는 메시지 형식·종료 조건에 대한 공유 규범이 필요하다. 토론 프롬프트는 진짜 불일치를 장려해야 하는데, LLM은 다른 인스턴스의 인지된 권위에 굴복해 그럴듯하나 미검증 합의로 수렴하는 경향이 있기 때문이다 — "이전 에이전트와 의견이 다르면 반드시 그렇게 말하고 이유를 설명하라" 같은 명시적 지시로 쓰지 않으면 둘째 에이전트가 예의 외 이유 없이 첫 답에 서명한다.

## 3. Role Specialization and Inter-Agent Communication

> 잘 경계 지어진 역할은 두 가지를 동시에 한다 — 에이전트의 권한 안에 무엇이 있고 밖에 무엇이 있는지를 명확히 한다. 두 절반이 동등하게 중요하다.

- 적용지침: "연구 보조"라고만 들은 에이전트는 인접 영역으로 표류하지만, "학술 출처를 조사해 세 불릿 요약을 반환하되 모든 합성 요청은 거절한다"고 들은 에이전트는 오케스트레이터가 기대하는 차선 안에 머문다. 역할 프롬프트는 에이전트의 입력 형식과 출력 형식을 정밀히 명시하고, 에이전트가 요청을 거절하거나 인간·상위 코디네이터로 에스컬레이션하는 조건도 진술해야 한다.
- 적용지침: 에이전트 간 메시지(inter-agent message)는 자체 규율이 필요하다 — 수신 에이전트는 송신 에이전트가 어떻게 추론했는지 기억이 없고 전달된 메시지만 본다. 구조화 메시지가 돕는다 — 결론 + 태그된 지지 증거가 깔끔히 파싱되어 하방 에이전트에 행동·반박할 재료를 준다. 신뢰도 지표(confidence indicator)는 다음 단계가 상류 주장이 얼마나 견고한지 아는 데 의존할 때 도움.
- 적용지침: 두 실무 포인트 — (1) 컨텍스트 비대(context bloat)가 모든 에이전트가 전체 추론 트레이스를 다음으로 전달할 때 스며든다. 메시지를 결론·지지 증거로 다듬으면 대화가 추적 가능하게 유지되고 토큰을 아낀다. (2) 메시지 스키마의 일관성은 작은 2-에이전트 시스템에서도 사람이 알아채기 전 하방 추론을 오염시킬 수 있는 조용한 파싱 실패를 막는다.
- 적용지침 예시(Reviewer Prompt): "당신은 2-에이전트 작성 파이프라인의 REVIEWER 에이전트다. 입력 형식(WRITER가 전송): `<draft>...</draft>`, `<claims>...</claims>`. 책임 — 각 주장을 초안에 대조 검증하고 뒷받침 없는 주장을 표시; 초안의 사실 오류·불명확 참조 식별. WRITER가 모든 수정을 소유하며 당신 일은 검증만. 정확히 이 구조로 출력 — `<unsupported_claims>...</unsupported_claims>`, `<factual_issues>...</factual_issues>`, `<verdict>accept | revise</verdict>`. 주장을 초안만으로 검증 못 하면 `<verdict>revise</verdict>`를 반환하고 `<factual_issues>` 안에 누락 컨텍스트를 기술하라." — 입력·출력 스키마를 한 곳에 고정하고, 초안 재작성에 대한 명확한 경계와 모호 케이스의 폴백 행동을 명시해 리뷰어 기여를 예측 가능하게 만든다.

## 4. Risks and Trade-Offs

> 멀티 에이전트 시스템은 단일 에이전트의 실패 양식을 상속하고 자기 것 여럿을 더한다.

- 적용지침: Cemri et al. (2025)는 1,600개 이상 멀티 에이전트 트레이스에서 14개 구별된 실패 유형을 분류했다 — 대부분 사양 모호성·에이전트 간 불일치(misalignment) 두 큰 클러스터에 속했고, 작은 비중은 하방 검사가 누락·약한 검증 간극이었다. 실무적 읽기 — 대부분 멀티 에이전트 실패는 조율 문제에서 오며 기저 모델 자체는 좀처럼 제한 요인이 아니다.
- 적용지침: 복리 오류(compounding error)가 실무 걱정거리 1위 — 상류 에이전트의 작은 오해를 하방 에이전트가 ground truth로 다루고 그 위에 쌓으면, 시스템이 일관되지만 전적으로 틀린 답을 낸다. 치료는 체인 어딘가에 상류 오류를 너무 많은 하방 투자가 쌓이기 전에 탐지하는 검증 단계를 두는 것 — 전용 체커 에이전트나 외부 진리원에 대한 결정론적 테스트. 두 옵션 간 선택은 모델 속성보다 정확성을 얼마나 싸게 측정할 수 있느냐에 달렸다.
- 적용지침: 폭주 비용(runaway cost)이 뒤따른다 — 모든 추가 에이전트가 프롬프트·출력 토큰과 체인을 흐르는 메시지를 더한다. 토론·peer-to-peer 같은 루프 패턴에서 비용이 라운드 수에 비례하고, 잘못 설계된 종료 규칙이 루프를 일의 가치보다 훨씬 오래 돌릴 수 있다. 오케스트레이터 안에 예산 cap을 설정·강제하는 것이 일주일 뒤 청구서에서 초과를 발견하는 것보다 싼 보험이다.
- 적용지침: 교착(deadlock)은 에이전트가 타이브레이커 없이 의견이 갈릴 때 발생 — 서로 수정을 기다리는 두 리뷰어가 영원히 돌 수 있다. 판사 에이전트나 고정 반복 cap이 대부분 해결하고, 단일 에이전트 초안으로의 폴백이 다른 해결이 실패할 때 최종 안전망이 된다(설계 리뷰 시 동등 컴퓨팅에서 단일 에이전트 베이스라인이 종종 멀티 에이전트를 같거나 능가함을 sanity check로 기억하라).
- 적용지침: emergent misalignment가 더 미묘한 위험 — 토론식 시스템은 순응 동역학(conformity dynamics)을 발달시킬 수 있어, 자신 있어 보이는 틀린 답이 옳은 답을 익사시킨다(에이전트들이 독립적 토대를 결여하고 외견상 합의에 따르는 경향).
- 적용지침: 가드레일이 단일 에이전트 시스템보다 더 중요한데, 문제가 사람이 알아채기 전 증폭될 수 있기 때문이다 — 모든 에이전트 간 메시지를 로깅하면 디버깅 추적이 생기고 메시지 스키마의 구조화 로그는 문제 실행을 오프라인 재생 가능하게 한다. 중간 출력을 주기적으로 검사하는 단일 감독 에이전트가 표류를 일찍 잡으며, 어떤 LLM 호출에도 낭비되기 전 명백한 스키마 위반을 표시하는 작은 결정론적 검사와 짝지으면 더 좋다. 토큰 사용의 하드 cap이 뭔가 잘못될 때 청구를 제한하고, 도구 호출의 유사 cap이 폭주 외부 지출을 막는다.

## Summary (핵심 정리)

- 멀티 에이전트는 특화·병렬성·적대적 검토·규모가 필요할 때 가치가 있으나(Du 2024, Tran 2025), 동등 컴퓨팅에선 단일 에이전트가 종종 비등하므로 분할이 고칠 실패를 명명할 수 있을 때만 쓰라.
- 조율 패턴은 orchestrator-worker(구조가 추론 후 드러날 때)·peer-to-peer(명확한 종료 필요)·debate(진짜 불일치 강제 필요)이며, 역할을 좁게 경계 짓고 에이전트 간 메시지를 결론+증거로 구조화하라.
- 복리 오류·폭주 비용·교착·emergent misalignment(Cemri 2025: 대부분 조율 문제)에 대비해 검증 단계, 예산·도구 호출 hard cap, 메시지 로깅, 감독 에이전트로 가드레일을 두라.
