# 05. Modularization

## 챕터 개요 (3줄 요약)

- 프롬프트의 각 부분을 독립적으로 교체·테스트 가능한 빌딩 블록으로 다루는 컴포넌트 사고(component thinking)를 다루며, 모듈화는 프롬프트를 일회용 텍스트에서 프로젝트·팀에 걸쳐 유지되는 견고한 자산으로 바꾼다.
- 모듈화는 프롬프트 파이프라인과 템플릿 라이브러리 같은 고급 시스템의 개념적 토대이며, 모놀리식(monolithic) 프롬프트가 격리된 채 리뷰 가능한 더 작은 조각으로 나뉘어야 할 시점을 알아보게 한다.
- 사전 비용은 작지만, 모듈 설계를 회피하면 이후 편집에서 페널티가 복리로 불어난다.

---

## 1. The Problem with Monolithic Prompts

> 모놀리식 프롬프트는 모든 관심사가 같은 텍스트 블록 안에 사는 것으로, 페르소나·작업·형식·예시·제약이 한 덩어리로 뒤섞여 내부 이음새가 없다.

- 단일 저자가 소유하고 각 문장의 역할을 기억하는 동안엔 잘 작동하지만, 작은 변경 하나가 필요해지는 순간 취약성이 드러난다.
- 적용지침: 두 줄 떨어진 톤을 부드럽게 하려다 역할 설명이 약화되거나, 출력 스키마 편집이 예시 블록을 교란할 수 있다 — 모든 부분이 경계 없이 함께 살기에 변경 파급이 의도치 않은 행동으로 번진다.
- 비용은 재사용 시 복리로 불어난다 — 전체 텍스트를 복사해 일부만 바꾸면 거의 중복인 프롬프트 두 개를 따로 유지해야 하고, 클론 수가 늘수록 작은 차이가 누적된다.
- 적용지침 사례: 90줄짜리 고객 지원 분류 프롬프트(페르소나+라우팅 규칙+JSON 형식+예시+종료 알림)에서 규칙 섹션을 편집하다 JSON 출력 기술을 무심코 바꿔 무관한 사례의 출력 품질이 떨어지고, 버그는 일주일 뒤 고객에 의해 발견된다.
- 모놀리식 설계의 본질적 비용은 어떤 문장이 핵심 역할을 하는지 아무도 확신하지 못해 모든 변경이 위험하게 느껴지고, 마감이 강제할 때까지 재작성이 미뤄지는 것이다.

## 2. Principles of Prompt Modularization

> 소프트웨어 공학이 수십 년간 정립한 시스템 분해 원리들이 프롬프트에도 잘 전이된다(Figure 1-6).

- 관심사의 분리(separation of concerns): 각 컴포넌트는 프롬프트 작업의 한 차원만 다뤄야 한다 — 지시는 형식 요구와, 페르소나 정의는 입력 데이터와 얽히지 말아야 한다.
- 적용지침: 단일 책임(single responsibility) — 각 컴포넌트는 변경 이유가 하나여야 한다. 무관한 두 이유로 같은 컴포넌트를 편집하고 있다면 그것은 과하게 많은 일을 하고 있으니 분할하라.
- 적용지침: DSPy 프레임워크(Khattab et al. 2024)는 각 프롬프트를 선언된 입출력 시그니처를 가진 모듈로 다뤄 더 큰 파이프라인으로 조합·자동 최적화한다 — 수동 튜닝 문자열을 넘어 각 요소가 명확한 계약(contract)을 갖는 구조화 AI 소프트웨어로 나아간다.
- 적용지침: 재사용성(reusability) — 신중한 증거 기반 분석가 페르소나를 한 번 쓰고 각 프롬프트에서 참조하면 변형 유지 부담이 사라진다(Chen et al. 2025a: 프롬프트를 버전 관리·명시적 모듈 설계의 대상인 선도 자산으로 관리).
- 적용지침: 인터페이스 계약(interface contract) — 각 컴포넌트는 입력으로 무엇을 기대하고 출력으로 무엇을 내는지 (주석으로라도) 선언해야 한다. 계약이 보이면 surrounding 컴포넌트가 내부를 몰라도 안전하게 구현을 교체할 수 있다.

```
[Software principle]      ->  [Prompt application]
Separation of concerns    ->  Persona apart from schema
Single responsibility     ->  Split modules that drift
Reusability               ->  Shared personas, versioned
Interface contracts       ->  Declared inputs and outputs
```

## 3. Refactoring Prompts into Components

> 리팩토링(refactoring)은 동작을 바꾸지 않고 구조를 재배치하는 작업으로, 프롬프트에서는 텍스트 벽 안에 숨은 컴포넌트를 식별해 명명된 교체 가능 부분으로 끌어내는 것이다.

- 적용지침: 첫 단계는 형광펜으로 기존 프롬프트 읽기다 — 역할 설명을 한 색, 작업 지시를 다른 색으로 표시하면 나머지 컴포넌트(예시, 입력 데이터, 제약)가 자연히 드러난다.
- 적용지침: 예시는 명시적 구분자가 있는 레이블된 블록으로, 입력 데이터는 자체 구분 섹션으로, 제약은 "constraints" 같은 헤딩 단락으로 끌어내 각각 독립 리뷰 가능하게 하라(Figure 1-7).
- 적용지침: 명명(naming)이 다음 단계다 — "40번째 줄 근처 부분"보다 `persona`, `output_schema`로 명명하면 논의와 치환이 쉬워진다. Mao et al. (2025)는 효과적 실무 템플릿이 플레이스홀더 슬롯과 명명 변수를 써 정확히 이 치환을 가능케 한다고 보았다.
- 적용지침: "죽은 섹션" 테스트 — 한 컴포넌트를 삭제했을 때 손실이 느껴지는지 물어라. 아무도 눈치채지 못하면 그것은 장식이고, 출력이 무너지면 그것은 load-bearing이며 그 무너짐이 역할을 증명한다.

## 4. Benefits for Testing and Iteration

> 모듈식 프롬프트의 이점은 개발 라이프사이클 전반에서 복리로 작용한다.

- 적용지침: 테스트에서 컴포넌트 분리는 어느 부분이 회귀(regression)의 원인인지 격리해준다 — 페르소나 모듈 변경 후 품질이 떨어지면 진단이 즉시 그곳을 가리키지만, 모놀리식에선 개별 문장을 이분 탐색하느라 수 시간이 걸린다.
- 적용지침: 평가 주도 반복(evaluation-driven iteration)이 용이해진다 — 컴포넌트를 격리 테스트하고 자동 테스트 스위트를 각 컴포넌트에 붙여, 형식 지시 변경이 점수를 높였는지 단지 문제를 다른 곳으로 옮겼는지 탐지한다.
- 협업에서 모듈식 프롬프트는 팀에 공유 어휘를 준다 — 리뷰어는 형식 블록에 얽히지 않고 페르소나가 옳은지 물을 수 있고, 디자이너는 작업 설명 재작성 없이 출력 스키마 변경을 제안할 수 있다.
- 적용지침: 버전 관리가 자연히 따라온다 — 모놀리식 프롬프트의 diff는 모든 편집이 뒤섞여 리뷰가 어렵지만, 모듈식 프롬프트의 diff는 변경을 특정 컴포넌트로 격리해 코드 리뷰를 의미 있게 만든다.
- 적용지침: 팀은 특정 모듈에 승인을 요구하고 나머지 시스템을 건드리지 않고 개별 컴포넌트를 롤백할 수 있으며, 소프트웨어용 코드 리뷰 도구와 신중한 리뷰의 문화적 습관을 프롬프트 설계로 전이할 수 있다.

## Summary (핵심 정리)

- 모놀리식 프롬프트는 변경 파급과 중복 유지보수로 취약하므로, 관심사 분리·단일 책임·재사용성·인터페이스 계약 원리로 모듈화하라.
- 형광펜 읽기 → 컴포넌트 추출 → 명명/치환 → 죽은 섹션 테스트의 리팩토링 절차로 프롬프트를 교체 가능한 부분으로 나눈다(DSPy, Mao et al. 2025 템플릿 슬롯).
- 모듈화는 회귀 격리, 평가 주도 반복, 컴포넌트 단위 버전 관리·롤백·코드 리뷰를 가능케 해 테스트·협업 효율을 크게 높인다.
