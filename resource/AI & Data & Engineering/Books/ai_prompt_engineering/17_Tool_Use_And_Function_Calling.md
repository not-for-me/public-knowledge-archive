# 17. Tool Use and Function Calling

## 챕터 개요 (3줄 요약)

- 에이전트는 텍스트 생성을 넘어 외부 세계와 상호작용할 수 있을 때 진정 유용해지며, 도구 사용(tool use, 함수 호출이라고도 함)은 언어 모델이 검색 엔진·API 같은 외부 능력을 호출하는 메커니즘이다.
- 도구를 명확히 기술하고 모델이 그중 선택하도록 돕는 프롬프팅 규율을 다루며, 프롬프트 설계가 모델이 유능한 운영자로 행동할지 혼란스러운 운영자로 행동할지를 결정하는 중심에 선다.
- 잘못된 호출을 최소화하는 도구 정의와 잘 형성된 호출을 낳는 구조 프롬프트를 설계하는 법을 배운다.

---

## 1. Defining Tools the Model Can Use

> 도구가 모델에 기술되는 방식이 도구의 기저 논리만큼 중요하다 — 코드의 함수 시그니처는 컴파일되면서도 자연어로는 절망적으로 모호할 수 있다. 모델이 도구 정의를 읽을 때 그것은 목적과 기대 행동을 몇 문장으로 전하는 작은 프롬프트다.

- 적용지침: Hsieh et al. (2023)는 도구 문서 품질이 zero-shot 환경 선택 정확도에 측정 가능한 효과를 가짐을 보였다 — 더 잘 쓰인 docstring이 더 신뢰할 만한 함수 호출을 낳는다. 도구 정의를 프롬프트의 핵심 표면으로 다뤄라.
- 적용지침: 이름은 수행 기능에 특정적이어야 한다 — `get_data`는 거의 쓸모없고(다른 여러 도구도 데이터 반환), `get_invoice_by_id`는 어떤 입력을 기대하고 어떤 질문을 향할지 즉시 알려준다.
- 적용지침: 설명은 도구가 무엇을 하고 무엇을 반환하는지 설명해야 하며, 각 파라미터에 기대 타입·허용 범위에 대한 짧은 메모를 달라. 기저 API가 필터를 지원하면 그 옵션을 설명에 각각 언제 적용되는지 안내와 함께 등장시켜라.
- 적용지침: Patil et al. (2024)는 밀집한 API 문서에 노출된 모델이 보지 못한 함수에 더 잘 일반화함을 보였다 — 철저한 정의가 즉각 작업을 넘어 전이됨을 시사한다.
- 적용지침: 반환 타입 계약(return type contract)도 같은 주목이 필요하다 — 모델이 출력을 소비할 것이므로 설명에 어떤 필드가 있고 무엇을 의미하는지 언급하라. "도구가 title·snippet 필드를 포함한 dict 리스트를 반환한다"는 한 줄만으로도 모델 기대를 고정하고 존재 안 하는 필드 접근 가능성을 줄인다.
- 적용지침: 도구 버전 관리(tool versioning)가 또 다른 조용한 함정이다 — 같은 도구가 필드 추가/이름 변경으로 갱신되면 시스템 프롬프트도 lockstep으로 갱신해야 하며, 안 그러면 모델이 더 이상 현실과 맞지 않는 문서 버전에 맞춘 호출을 낸다(stale 도구 설명이 디버깅 시간의 큰 원인).
- 적용지침: 흔한 실수 — 내부 문서에서 함수 시그니처를 복사하고 모델이 의도를 추론하리라 가정하는 것. 흔히 안 된다. 모델을 위해 쓰는 것은 제도적 맥락 없이, 후속 질문할 능력 없이 읽는 신중한 독자를 위해 쓰는 것이다. 노출하는 모든 도구에 언제 호출해야 하고 언제 다른 도구가 더 적절한지를 명명하는 사용 안내 한 줄을 더하라 — 이 작은 습관이 프로덕션 전 놀랍도록 많은 엣지 케이스를 잡는다.

## 2. Guiding Tool Selection

> 모델이 둘 이상의 도구에 접근하면 프롬프트가 각 단계마다 올바른 것을 고르도록 도와야 한다.

- 적용지침: 선택 실패는 몇 가지 알아볼 형태로 온다 — 특화 도구가 더 나은데 익숙한 도구로 기본 동작, 내부 지식으로 답할 질문에 불필요하게 도구 호출 등. Huang et al. (2024b)는 MetaTool 벤치마크로 이 행동을 측정해, 유능한 모델도 가용 함수가 어떻게 프레이밍되는지에 따라 선택 정확도에 상당한 변동을 보임을 발견했다.
- 적용지침: 명확화(disambiguation)가 돕는다 — 두 도구가 같은 요청에 그럴듯하게 봉사할 수 있으면, 각 설명에 이웃과 구별되는 점을 언급하라. `search_documents`는 공개 웹에 닿지 않음을 명시하고 `web_search`는 그 반대를 명시하라. 모델이 두 도구를 겹칠 것 같으면 가장 저렴한 수정은 그 차이를 양쪽 설명에 써넣는 것이다.
- 적용지침: 도구는 비자명한 에이전트에서 암묵적 위계를 형성한다 — 어떤 것은 싸고 빠르고 어떤 것은 느리고 비싸다. 비용 기울기(cost gradient)를 모델에 알리면 유용하게 행동이 바뀐다. "도구는 비싸니 충분한 경우 더 싼 대안을 선호하라"는 문장을 넣으면 낭비 호출이 줄고 월말 청구액이 눈에 띄게 작아진다.
- 적용지침 예시: "두 검색 도구에 접근할 수 있다 — search_internal_docs: 회사 지식 베이스를 질의(제품 사양·내부 정책에 사용), web_search: 공개 인터넷 질의(최근 뉴스·회사 외부 콘텐츠에 사용). 어떤 도구든 호출하기 전, 어느 것을 쓸지와 이유를 간단히 진술하라. 질문이 검색 없이 답할 수 있으면 직접 답하라." (Figure 3-6: 내부/공개 지식 소스 간 질의 라우팅 흐름)
- 적용지침: 이 패턴은 단순해 보여도 내 경험상 중복 호출의 큰 부분을 제거하고, 일이 잘못될 때 디버깅을 쉽게 하는 작은 감사 추적(audit trail)을 만든다.

## 3. Handling Tool Outputs

> 도구 출력을 직접 통제하는 규율은 출력 형식의 기초 위에 세워진다(Module 07) — 깔끔한 출력을 반환하는 도구가 원시 API 잔해를 반환하는 도구보다 모델이 해석하기 쉽다.

- 적용지침: 장황한 JSON·깊이 중첩된 필드는 컨텍스트를 소비하고 모델이 응답의 잘못된 부분에 집중할 가능성을 높인다. 도구 출력 처리의 기술은 모델이 행동에 필요한 것만 보도록 반환값을 형성하는 데 있다.
- 적용지침: 두 패턴이 지배한다 — (1) 경계에서의 요약(summarization at the boundary): 도구가 긴 문서·큰 결과 집합을 모델에 넘기기 전 압축. (2) 일관된 형태(consistent shape): 모든 성공 호출이 같은 순서로 같은 필드 집합 반환.
- 적용지침: 명시할 트레이드오프 — 경계에서의 요약은 토큰을 아끼고 집중을 높이지만 모델이 썼을 정보도 버린다. 검색 결과를 공격적으로 요약하면 에이전트가 깊이 중첩된 필드의 핵심 세부를 인용 못 할 수 있다. 수정책은 요약 옆에 작은 `raw_excerpt` 필드를 둬 필요 시 모델이 되돌아가게 하는 것 — 경계 설계는 일부 정보 손실을 받아들이는 것이다.
- 적용지침: 오류는 자체 형식이 필요하다 — 모델이 호출 성공 여부와 실패 시 할 일을 한눈에 알아야 한다. 흔한 패턴은 짧은 사람 판독 가능 오류 메시지와 재시도 가능성 힌트를 담은 status 필드 반환이다. 원리 — 도구 출력 자체가 프롬프트다. 미래 시스템 메시지에 나타나길 원치 않는 것은 경계에서 걸러내라.
- 적용지침 예시: "search_documents 결과를 요약할 때 다음 형식으로 출력하라 — Status: success 또는 failure / Top match: title과 한 문장 요약 / Confidence: high 또는 low(검색이 유용한 것을 안 반환하면 명시적으로 그렇게 말하고 대안 질의를 제안). 데이터 없는 성공도 짧은 reason 필드를 반환해야 하는데, 모델이 비어 있음을 일관되지 않게 해석하기 때문이다."

## 4. Failure Modes and Recovery

> 도구 사용은 특징적 방식으로 깨지며 각 붕괴엔 대응하는 프롬프트 수준 수정책이 있다 — 에이전트 설계 규율은 실패를 예상하고 그것을 담는 프롬프트를 쓰는 것을 포함한다(Figure 3-7).

- 적용지침: 첫 실패 양식은 환각 호출(hallucinated call) — 존재하지 않는 함수를 호출. 보통 불명확한 도구 정의나 가용 도구를 정확히 열거 안 한 시스템 프롬프트로 추적된다. 수정책은 가용 도구를 시스템 프롬프트에 명시적으로 나열하고 그 목록 밖 호출을 거부하도록 지시.
- 적용지침: 둘째 실패 양식은 잘못된 파라미터(malformed parameter) — 모델이 스키마를 오해할 때 발생. 도구 설명 안에 유효 호출의 짧은 예시를 더하면 보통 고쳐진다(모델은 강한 패턴 매처이며 잘 형성된 예시 한둘이 따를 템플릿이 된다). 파라미터 제약을 타입 시그니처 옆에 평이한 영어로 두는 것이 스키마만 의존하는 것보다 낫다.
- 적용지침: 반복 실패(repeated failure)는 진전 없이 예산을 소비해 특히 비싸다 — 간단한 가드 — 같은 도구가 유사 인자로 두 번 실패하면 멈추고 전략을 바꾸거나 보고하도록 지시.
- 적용지침: 탐지(detection)도 주목이 필요하다 — 루프가 실패를 일찍 잡을수록 복구가 싸다. 도구 호출과 파라미터를 루프 중 검토 가능하게 로깅하면, 특히 실패가 조용하거나 도구가 빈 콘텐츠로 success 상태를 반환할 때 도움이 된다. 어림 규칙 — 데이터 없이 success를 반환할 수 있는 도구는 짧은 reason 필드도 반환해야 한다(모델이 비어 있음을 일관되지 않게 해석하므로).
- 적용지침: 도구 비가용성(tool unavailability)이 가장 설계하기 쉽지만 실무에서 가장 자주 잊힌다 — 시스템 프롬프트가 도구가 없거나 unavailable 상태를 반환할 때 할 일을 기술해야 한다. 모든 에이전트 프롬프트에 차선 행동을 명명하는 폴백 단락을 두는 것이 권장되는데, 보통 한 능력이 오프라인이었다는 명확한 메모와 함께 부분 답을 반환하는 것이다(답이 부분적이어도 사용자는 뭔가 잘못됐음을 알아 이득).
- 핵심 통찰: 도구 사용은 프롬프트 엔지니어링이 시스템 설계와 만나는 곳이다 — 모델 행동은 어떤 도구가 존재하고 어떻게 기술되며 무엇을 반환하고 실패 시 무슨 일이 일어나는지로 형성된다. 각각이 설계 표면이며, 하나라도 부주의하게 다루면 데모 조건에선 유능해 보이나 프로덕션에서 무너지는 에이전트가 된다. 어떤 도구 설계도 실사용자와의 첫 접촉에서 완벽하지 않으니, 트랜스크립트를 검토하고 모델이 빗나가는 곳을 보며 그에 따라 도구 설명을 갱신하는 습관이 시간이 지나며 에이전트를 개선하는 가장 신뢰할 만한 방법이다.

## Summary (핵심 정리)

- 도구 정의를 핵심 프롬프트 표면으로 다뤄 특정적 이름·철저한 설명·파라미터 메모·반환 타입 계약을 쓰고, 버전 변경 시 시스템 프롬프트를 lockstep으로 갱신하라(Hsieh 2023, Patil 2024).
- 선택을 안내하려면 도구 차이를 양쪽 설명에 명시하고 비용 기울기를 알리며, 호출 전 어느 도구를 왜 쓸지 진술하게 하라(Huang 2024b MetaTool).
- 출력은 경계 요약·일관된 형태·status 오류 형식으로 형성하고, 환각 호출·잘못된 파라미터·반복 실패·도구 비가용성에 각각 명시적 프롬프트 수정책(열거·예시·재시도 가드·폴백)을 두라.
