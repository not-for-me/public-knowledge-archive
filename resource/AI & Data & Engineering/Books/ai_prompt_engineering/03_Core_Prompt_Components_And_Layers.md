# 03. Core Prompt Components and Layers

## 챕터 개요 (3줄 요약)

- 효과적인 프롬프트는 각기 역할이 다른 기능 구성요소들로 이루어지며, 이들을 세 가지 작업 계층 — 셋업(setup), 머티리얼(material), 출력 형성(output-shaping) — 으로 조직하는 계층 프레임워크를 제시한다.
- 프롬프트를 이 계층들의 스택(stack)으로 보면 각 계층을 독립적으로 편집하며 변경이 결과에 미치는 영향을 관찰할 수 있다(Figure 1-5).
- 셋업 계층은 역할·작업을, 머티리얼 계층은 컨텍스트·입력 데이터를, 출력 형성 계층은 형식·제약·톤·추론 지시를 담당한다.

---

## 1. The Setup Layer: Role and Task

> 셋업 계층은 모델이 첫 토큰을 내기 전에 "나는 누구인가, 무엇을 하는가" 두 질문에 답한다.

- 역할(role) 할당은 모델이 응답할 페르소나·입장을 지정하고, 작업 정의(task definition)는 요약인지 초안 작성인지 등 취할 정확한 행동을 명시하며, 역할이 작업과 맞을수록 성능이 좋아진다.
- 역할은 두 레지스터로 나뉜다 — role-as-flavor(표면 톤만 설정)와 role-as-genuine-frame(모델이 문제에 접근하는 방식·고려 증거·휴리스틱을 실제로 바꿈).
- 적용지침: 후자가 더 유용하다 — "검증되지 않은 가정을 찾는 신중한 리뷰어로 행동하라"가 "친근한 도우미로 행동하라"보다, 작업 정의가 같아도 눈에 띄게 다른 출력을 낸다(Salewski et al. 2023, in-context impersonation).
- 적용지침: 역할 특이성이 역할 분량보다 중요하다 — "사기 신호를 위해 경비 보고서를 평가하는 포렌식 회계사" 같은 짧고 정밀한 페르소나가, 8가지 성격과 허구적 배경을 나열한 긴 페르소나를 보통 능가한다.
- 적용지침: 역할을 작성할 때 두 가지를 인코딩하라 — 모델이 문제를 바라봐야 할 렌즈(lens)와 스스로 지켜야 할 품질 기준(standard of quality).
- 작업 정의는 예측 가능하게 실패한다 — "이것을 분석하라" 같은 모호한 작업은 "분석"의 의미 선택지가 너무 많아 일반적 응답을 부른다.
- 적용지침: 상충하는 작업(짧은 요약 + 포괄적 심층 분석 동시 요구)은 모델을 타협시켜 반쪽짜리 결과를 낳으므로, 산출물을 명명하는 선언적 동사 하나와 그 경계를 명시하라.

## 2. The Material Layer: Context and Input Data

> 머티리얼 계층은 모델이 추론할 내용을 제공하며, 컨텍스트(배경)와 입력 데이터(처리 대상)라는 두 기능을 담는다.

- 컨텍스트는 회사 스타일 가이드·청중 설명 같은 작업 프레이밍 배경이고, 입력 데이터는 요약할 이메일·검토할 코드 같은 구체적 처리 대상이다.
- 적용지침: 두 조각을 명확한 구분(demarcation) 없이 섞으면 모델이 지시와 내용의 경계를 몰라 원시 입력을 추가 명령으로 취급하는 흔한 오류가 발생한다.
- 적용지침: 입력 데이터를 명확히 레이블된 구분자 — XML 태그나 단순 섹션 헤더 — 로 감싸 운영 텍스트와 작업 내용 사이에 모델이 사용할 가시적 경계를 만들라.
- 예시 프롬프트 구조: Context에 "회사는 영국식 영어 철자를 쓰고 느낌표를 피한다", Input Data에 `<email>{email_text}</email>` 형태로 명시.
- 구분(demarcation)은 작은 구조적 결정이지만 효과는 크다 — 경계가 명확하면 모델이 무엇이 지시이고 무엇이 데이터인지 안정적으로 구분한다.

## 3. The Output-Shaping Layer: Format, Constraints, and Tone

> 모델이 누구이고 어떤 내용을 가졌는지 알고 나면, 이 지시 집합이 무엇이 나올지를 통제한다 — 형식은 구조를, 제약은 금지를, 톤은 목소리를 다스린다.

- 형식(format)은 구조를 지배한다 — 명명된 키를 가진 JSON 객체를 요청하는 것은 단일 문단 요약을 요청하는 것과 매우 다른 하방 행동을 낳는다.
- 적용지침: 형식 지시는 두 이유로 중요하다 — (1) 파싱 가능성(parseability): 예측 가능한 구조가 신뢰성 있는 파이프라인과 취약한 파이프라인을 가른다, (2) 추론 품질: 사소한 형식 변화도 LLM 벤치마크에서 수십 퍼센트포인트의 성능 변동을 낳는다(Sclar et al. 2024).
- 적용지침: 구조화 출력이 필요하면 엄격한 스키마에 단일 부정 제약(예: "JSON 밖에 어떤 텍스트도 내지 말 것")을 짝지어라 — 긍정적 목표와 명확한 경계를 동시에 줘 사후 정리 없이 기계 판독 가능한 출력을 얻는다(Module 07에서 심화).
- 적용지침: 제약(constraint)은 허용 출력 공간을 좁혀 작동한다 — "도입 인사말을 넣지 말 것" 같은 단일 부정 제약이, 긍정 지시를 재작성하는 것보다 부작용이 적게 원치 않는 행동 한 부류를 제거한다.
- 적용지침: 톤(tone)은 가장 가벼운 통제다 — "냉정하고 사실적인 어조로 작성하라" 같은 짧은 톤 한정자가 응답의 목소리와 컨텍스트가 요구하는 목소리 간 불일치를 해소한다.

## 4. The Output-Shaping Layer: Reasoning Directives

> 추론 지시(reasoning directives)는 모델이 최종 답을 내기 전에 어떻게 사고해야 하는지에 대한 지시로, 형식·톤과 같은 형성 계층에 있지만 답에 이르는 추론 과정을 표적으로 한다.

- 가장 많이 연구된 것은 chain-of-thought 프롬프팅이다(Module 08에서 심화) — 형식·톤이 답의 표면을 형성한다면, 추론 지시는 그에 이르는 생성 궤적을 바꾼다.
- 적용지침: 주어진 작업에 작동하는 가장 가벼운 버전부터 시작하라 — 무거운 추론 스캐폴드는 추론을 느리게 하고, 중간 사고 사슬이 빗나가면 더 자신 있게 틀린 답을 낼 수 있다.
- 적용지침: self-consistency 샘플링은 여러 추론 경로를 생성해 최종 답을 다수결로 집계하고(Module 09), self-critique는 초안을 쓰고 약점을 찾아 수정하게 한다(Module 12) — 둘 다 사고 과정 노출을 추론 개선 메커니즘으로 확장한다.
- 적용지침: 예시 — "최종 답을 쓰기 전에 계산을 단계별로 진행하고, 각 가정을 명시한 뒤, 최종 숫자를 별도 줄에 FINAL 접두로 제시하라."
- 적용지침: 추론 지시는 다른 계층과 비자명하게 상호작용한다 — 정밀한 역할이 이미 사고 방식을 심어두거나, 지나치게 엄격한 형식이 추론 공간을 압박할 수 있으니, 프롬프트를 계층화된 결정의 시퀀스로 읽고 상호작용을 관찰·튜닝하라.

---

## Summary (핵심 정리)

- 프롬프트를 셋업(역할·작업)·머티리얼(컨텍스트·입력)·출력 형성(형식·제약·톤·추론)의 세 계층 스택으로 보면 각 계층을 독립적으로 편집·관찰할 수 있다.
- 정밀한 역할, 구분자로 분리된 입력, JSON 스키마+부정 제약, 짧은 톤 한정자 같은 구체 지침이 신뢰성과 추론 품질을 함께 끌어올린다.
- 추론 지시는 가장 가벼운 버전부터 적용하고 계층 간 상호작용을 튜닝하는 것이 핵심이다.
