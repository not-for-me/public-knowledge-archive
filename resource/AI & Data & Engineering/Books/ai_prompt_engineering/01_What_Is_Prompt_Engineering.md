# 01. What Is Prompt Engineering?

## 챕터 개요 (3줄 요약)

- 프롬프트 엔지니어링은 대규모 언어 모델(LLM, Large Language Model)의 출력을 신뢰할 수 있고 유용하게 만들기 위해 입력을 체계적으로 설계·정제하는 학문 분야로, 응용 언어학과 컴퓨터 과학 사이에 위치한다.
- 단순한 지시 작성(prompt drafting), 프롬프트 해킹(prompt hacking), 전통적 소프트웨어 개발과는 구별되며, 동일 입력이 실행마다 다른 출력을 낼 수 있는 확률적 성격 때문에 응용 과학에 가깝다.
- 프롬프트 품질은 정확도·비용·지연시간·사용자 신뢰에 측정 가능한 영향을 미치므로, 관찰과 규율 있는 실험을 핵심 방법으로 삼는 엔지니어의 사고방식이 중요하다.

---

## 1. Defining Prompt Engineering

> 프롬프트는 모델이 작업을 수행하도록 주는 자연어 입력이며, 프롬프트 엔지니어링은 그 입력을 예측 가능하고 의도에 정렬되게 구성하는 체계적 실천이다.

- 프롬프트는 모델에게 작업을 위임하기 위한 자연어 입력이며, 엔지니어링은 결과 행동이 예측 가능하고 의도와 정렬되도록 입력을 구성하는 것이다.
- Liu et al. (2023)는 프롬프팅을 원래 작업 입력을 텍스트 템플릿으로 재구성해 모델이 채우고 다시 파싱하는 패러다임으로 규정했으며, 이는 NLP(Natural Language Processing)의 오랜 전통에 뿌리를 둔다.
- 적용지침: 입력이 대표 사례(representative cases)로 검증되기 전이면 "프롬프트 초안(drafting)"으로 간주하고, 출력을 관찰하며 조정하는 단계에 이르러야 "프롬프트 엔지니어링"으로 부른다.
- 인접 용어 구분이 핵심이다 — 프롬프트 작성(casual phrasing), 프롬프트 해킹(안전 정책 우회·시스템 프롬프트 추출 시도), 전통적 소프트웨어 개발(결정론적 출력)은 서로 다른 실천이다.
- 모델의 확률적 성격 때문에 같은 입력도 실행마다 출력 분포가 달라지므로, 여러 실행에 걸쳐 분포를 관찰하며 문구를 조정해야 한다.
- 핵심 어휘 고정: model(기반 언어 모델), system prompt(세션 페르소나·기본 규칙), user prompt(개별 요청), output(평가 대상 응답).

## 2. The Evolution of Engineered Prompts

> 지난 30년간 지배적이던 검색 질의(search query) 방식에서, 모델이 입력으로부터 직접 답을 생성하는 방식으로 인터페이스가 전환되며 명확성의 부담이 프롬프트로 이동했다.

- 과거에는 검색 질의가 주된 인터페이스였고, 키워드 몇 개로 순위화된 페이지 목록을 받는 방식이었으며 검색 시스템이 부정확함을 보정했다.
- 언어 모델은 이 관계를 역전시켜 입력으로부터 직접 답을 생성하므로, 명확성의 부담이 프롬프트 자체로 옮겨간다.
- Liu et al. (2023)는 레이블된 예시가 필요한 지도학습에서, 입력만 잘 구조화하면 단일 범용 모델이 여러 작업을 처리하는 프롬프팅으로의 전환을 추적했다.
- 적용지침: 검색식 입력("best running shoes 2025") 대신, 역할·출력 구조·평가 차원을 명시한 엔지니어링 프롬프트(예: "footwear analyst 역할로 2025 마라톤용 도로화 비교, 추천 2개로 제한, 무게(g)·스택 높이 보고")를 작성하라.
- 언어를 모델이 수행할 행동을 선택하는 "프로그래밍 표면(programming surface)"으로 다루면, 개별 문장을 수정·교체 가능한 컴포넌트로 보게 된다.

## 3. Why Prompt Engineering Matters

> 프롬프트 품질은 정확도·비용·지연시간·사용자 신뢰에 측정 가능한 하방 효과를 가지며, 작은 변화가 놀라운 차이를 만든다.

- Salinas & Morstatter (2024)는 입력 끝의 공백 하나 같은 아주 작은 교란(perturbation)도 모델 출력을 바꿀 수 있음을 11개 텍스트 분류 작업에서 보였다.
- 적용지침: 출력 형식을 명시하고 적대적 입력을 통제하면 정확도가 가장 쉽게 개선되며(Figure 1-1), 올바른 형식과 내용을 끌어내는 프롬프트는 후처리 부담을 줄인다.
- 비용은 토큰 사용량에 비례하고, 모호한 프롬프트는 재시도를 유발해 지연시간을 키우며, 사용자 신뢰는 "첫 시도"가 맞느냐에 달려 있다.
- 적용지침: 고객 서비스 분류처럼 닫힌 집합(closed set)에서 단일 레이블을 요청하고, 미지의 경우를 위한 폴백(fallback)을 두면 라우팅 가능한 구조화 출력을 안정적으로 얻는다.
- 프롬프트 품질은 모델 선택·평가 하네스(harness)와 상호작용하는 "승수(multiplier)"이며, 약한 프롬프트는 다른 곳의 이득을 가린다.
- 프롬프트를 자체 리뷰 프로세스와 성능 예산을 가진 시스템 구성요소로 취급하면 문제를 조기에 표면화할 수 있다.

## 4. The Skills and Mindset

> 효과적인 프롬프트 엔지니어는 언어를 분석적으로 다루고, 각 프롬프트를 대표 입력에 대해 검증할 가설로 취급한다.

- 자신의 지시에서 모호성을 알아채고, 사람과 다르게 해석할 수 있는 모델의 관점을 고려하는 분석적 언어 감각이 필요하다.
- 적용지침: 각 프롬프트를 가설로 보고 대표 입력에 대해 실험·수정하며, 단계별 추론(step-by-step reasoning, Module 08) 같은 기법으로 큰 성능 향상을 얻을 수 있다.
- 출력을 끈기 있게 읽는 인내심이 필요한데, 언어 모델의 실패 양식은 미묘하며 다수 예시를 검토한 뒤에야 드러나기 때문이다.
- 적용지침: 문구가 마음에 든다는 이유로 프롬프트를 옹호하게 되면, 그것을 신호로 삼아 한 걸음 물러나 지표(metrics)를 보라 — 개인 취향을 모델 행동 증거로 착각하지 말 것.
- 프롬프트는 부하 상태에서의 성능으로만 프로덕션 자격을 얻으며, 이 사고방식은 실제로 중요한 작업에 프롬프트를 쓰고 모델 반응을 관찰하는 의도적 연습으로 길러진다.

---

## Summary (핵심 정리)

- 프롬프트 엔지니어링은 언어를 프로그래밍 표면으로 다루어 LLM 출력을 신뢰 가능하게 만드는 체계적·실험적 실천이다.
- 토큰화·교란·형식 지정에 따라 정확도·비용·지연·신뢰가 크게 달라지므로, 출력 형식 명시와 닫힌 집합 레이블링 같은 구체 지침이 효과적이다.
- 핵심 역량은 모호성 탐지, 가설 기반 실험, 지표 중심의 객관적 평가이며, 이는 의도적 연습으로 길러진다.
