# 33. Automated Evaluation Methods

## 챕터 개요 (3줄 요약)

- 성공 기준이 정의되고 데이터셋이 조립되면 다음 질문은 출력을 규모로 어떻게 채점하느냐다 — 수동 검토는 수천 예시나 빠른 반복 사이클로 깔끔히 확장 안 되어, 자동 평가가 진지한 프롬프트 시스템의 핵심 인프라가 된다.
- 매 예시를 인간 검토 없이 채점하는 방법(결정론적 규칙 검사부터 개방형 응답을 다루는 모델 기반 판사까지)을 살피며, 각 방법의 강점·맹점·비용 프로필을 비교한다.
- 각 기준에 맞는 접근을 고르고 방법들을 실제 제품 품질과 정렬된 평가 스위트로 결합하는 법을 배운다.

---

## 1. Rule-Based and Programmatic Checks

> 가장 단순한 평가 방법은 결정론적이다 — 출력을 기대 패턴에 대조하는 코드를 돌려 이진·수치 점수를 반환한다(Figure 6-4: 하방 전달 전 스키마·regex 검사의 결정론적 검증 파이프라인).

- 적용지침: 형식 검증(format validation)이 표준 케이스 — 프롬프트가 특정 필드를 가진 JSON을 요청하면 파서가 성공·실패하고 파싱 실패한 출력을 사용자 도달 전 표시할 수 있다. 스키마 검증(Module 07)이 이를 확장해 필수 키·기대 타입이 계약과 맞는지 확인하고, 최신 스키마 라이브러리는 필드 안의 값 범위·regex 패턴도 강제할 수 있다.
- 적용지침: 이 검사의 매력은 속도와 신뢰다 — 밀리초 단위로 돌고 일반 컴퓨팅 외 비용이 거의 없다. 행동이 완전 투명해, 실패한 regex가 무엇이 잘못됐는지 정확히 알려줘 triage가 거의 사소해진다. 프로그래밍 검사가 CI(continuous-integration) 파이프라인에 자연히 끼어, 프롬프트 변경을 병합 전 참조 스위트에 대해 평가할 수 있고 더 무거운 지표 실행 전 놀라운 비율의 회귀를 잡는다.
- 적용지침: 비용은 커버리지 — 결정론적 검사는 출력의 기계적인 것을 채점한다. 생성 설명이 옳은지, 요약이 관련 아이디어를 포착하는지 말할 수 없다(그런 판단은 다른 방법이 필요). 형식을 완벽히 맞추는 그럴듯-하지만-틀린 출력도 놓치는데, 이는 구조화 시스템의 더 위험한 실패 양식 중 하나다.
- 적용지침: 구조화 출력 생성 벤치마크는 강한 모델도 미묘한 스키마 제약(특히 중첩 필드·열거 값 범위 주변)을 위반함을 보여, 결정론적 검사가 유능한 모델에도 유용함을 의미한다. 의미를 포함하는 기준엔 다른 도구가 필요하고 결정론적 스위트는 그 아래에서 baseline으로 계속 돌아야 한다.

## 2. Reference-Based Metrics

> 각 입력에 대한 정답을 생성·큐레이션할 수 있으면 참조 기반 지표(reference-based metric)가 가용해진다 — 모델 출력을 하나 이상의 참조 출력과 비교해 유사도 점수를 반환하며, 완전 기계적 검사와 인간식 판단 사이에 위치하고 작업에 "정답"의 옹호 가능한 개념이 있으면 유용하다.

- 적용지침: 가장 오래된 군은 string-overlap 지표 — BLEU는 기계 번역용으로 후보·참조 텍스트의 겹치는 n-gram을 brevity penalty로 가중해 센다. ROUGE는 요약용으로 recall을 강조해 참조 내용의 얼마가 후보에 나타나는지 묻는다. 둘 다 빠르고 재현 가능해 수년간 학술 벤치마크 표준이었다.
- 적용지침: 약점은 표면 겹침이 의미의 나쁜 대용이라는 것 — 다른 단어를 쓴 패러프레이즈 답이 옳아도 낮은 점수를, 진짜 이해 없는 복붙 답이 높은 점수를 받을 수 있다. 창의적·개방형 작업에선 인간 판단과의 상관이 0에 가까워질 수 있어 점수가 거의 노이즈다.
- 적용지침: 임베딩 기반 지표는 의미 유사도를 직접 타깃한다 — BERTScore는 사전학습 모델의 문맥 임베딩으로 후보·참조 토큰을 정렬해 유사도를 계산한다(Zhang et al. 2020). 문장 수준 임베딩 유사도는 텍스트당 단일 벡터 비교로 더 거친 수준에서 같은 일을 한다. 이들은 패러프레이즈를 잘 다루고 요약·번역 작업에서 인간 판단을 더 밀접히 추적하나, 모델 forward pass가 필요해 비용이 더 들고 절대 해석이 어렵다(0.87 값이 유사 예시의 baseline 보정 없이는 명확한 의미가 없음).
- 적용지침: 참조 기반 지표를 exact match(엄격)부터 embedding similarity(유연)까지의 슬라이딩 척도로 보라 — 척도 위 지점 선택은 정당한 답이 얼마나 다양하고 작업이 언어적 유연성을 얼마나 용인하는지에 의존해야 한다. 참조 기반 점수를 정답이 잘 정의되고 작업이 안정적 문구를 가질 때 유용한 신호로 다뤄라. 두 똑같이 좋은 답이 표면 형태에서 완전히 다를 수 있는 장문 개방형 출력엔 judge 기반 방법으로 옮겨라(참조 비교가 반복 루프를 오도하기 시작하므로).
- 적용지침: 참조 세트 품질이 많은 경우 지표 선택보다 중요하다 — 노이즈한 참조 세트는 어느 지표에서도 노이즈 점수를 내고, 여러 유효 답 스타일 중 하나만 포착한 참조 세트는 나머지를 체계적으로 페널티한다. 정답을 단일 주석자가 생성하면 둘째 사람이 표본을 검토해 참조가 채점에 baked되기 전 표류를 잡게 하라.

## 3. LLM-as-Judge Evaluation

> 기계적 채점에 저항하는 기준(도움됨·충실성)엔 언어 모델 자체를 평가자로 쓰는 것이 가장 실용적이다 — 판사 모델이 후보 출력을 원 입력과 함께 받아 루브릭을 주고 판정(선택적으로 짧은 정당화 포함)을 반환하게 요청한다.

- 적용지침: Zheng et al. (2023)가 MT-Bench·Chatbot Arena로 이 접근을 대중화해, 강한 판사 모델이 인간 선호와 약 80% pairwise 비교에서 동의함을 보였다(같은 비교의 인간 간 일치도 수준 근처)(Figure 6-5: order randomization·calibration이 있는 pairwise 출력 비교 LLM-as-Judge 설정).
- 적용지침: 판사 프롬프트는 좁게 범위 지정된 루브릭과 각 점수 수준이 어떤 모습인지 보이는 소수 보정 예시와 짝지을 때 가장 유용하다 — 모호한 루브릭은 모호한 판정을, 보정 예시를 안 본 판사는 가장 중요한 경계 케이스에서 자체 채점 행동을 발명한다.
- 적용지침: 알려진 실패 양식을 설계로 다뤄라 — position bias(두 후보 비교 시 등장 순서가 판정을 바꿈, 더 많은 후보 동시 평가 시 효과 증가), length·similarity 편향(긴 응답이 가치 없어도 선호, 자기 생성과 닮은 출력을 더 높게 평정).
- 적용지침 예시: "고객 지원 질문의 챗봇 답을 채점한다. Question: {question} Reference answer: {reference} Candidate answer: {candidate}. 후보를 다음으로 채점 — Faithfulness: 후보가 참조가 뒷받침하는 사실만 진술하는가? Helpfulness: 후보가 사용자의 실제 질문을 다루는가? JSON 반환: {"faithful": <true|false>, "helpful": <true|false>, "reason": "<한 짧은 문장>"}"
- 적용지침: 두 실천이 알려진 편향을 줄인다 — (1) 쌍 비교 시 각 비교를 양 순서로 돌려 점수 평균. (2) 판사를 판정 전 짧은 루브릭 근거 추론으로 제약(경계 케이스 일관성 향상).
- 적용지침: 가장 중요한 안전장치는 검증(validation) — held-out 평가 예시 부분집합에 인간 레이블을 모아 판사 점수와 인간 점수의 상관을 측정하라. 이 단계 없이는 판사 점수가 신경 쓰는 품질을 실제 추적한다는 증거가 없고, 대시보드에서 읽는 개선이 판사 모델의 임의 선호에 대한 최적화일 수 있다.
- 적용지침: 판사 모델 선택도 주의를 요한다 — 평가 대상 시스템보다 약한 판사는 자기가 탐지 못 하는 오류를 체계적으로 놓치고, 평가 대상과 같은 모델 패밀리의 판사는 self-preference를 통해 시스템의 맹점을 상속한다. 다른 패밀리의 강한 모델을 판사로 쓰고 몇 주마다 신선한 인간 레이블로 주기적 재검증하면 매우 신뢰할 결과를 얻는다.

## 4. Designing Evaluation Suites

> 진지한 시스템은 좀처럼 단일 지표에 의존하지 않는다 — 올바른 설정은 각 계층이 다른 것이 놓치는 것을 덮고 결과가 한눈에 읽는 작은 대시보드로 집계되는 보완 방법의 스위트다.

- 적용지침: 프로그래밍 검사가 형식·스키마 붕괴를 0 비용으로 잡고, 그 위에 참조 기반·판사 기반 점수가 의미 수준 기준을 다루며 판사가 개방형 출력 일 대부분을 짊어진다.
- 적용지침: 집계가 사람들 생각보다 중요하다 — 단일 가중 평균이 한 기준은 critical이고 다른 기준은 그 critical 하나가 충족된 후에만 중요할 때 심각한 문제를 가릴 수 있다. 안전·형식 준수에 hard gate를 적용하고 나머지 기준을 각각 사용자 인지 품질에 얼마나 기여하는지에 따라 가중하라. gate 위반 회귀는 평균과 무관하게 실패로, 가중 기준 회귀는 다음 릴리스 전 조사 필요한 경고로 친다.
- 적용지침: 자동 스위트의 가장 음흉한 실패는 metric gaming(지표 게이밍) — 시스템이 측정 가능한 대용을 최적화하면서 기저 품질이 조용히 멀어지는 현상(Goodhart's law: 측정이 목표가 되면 좋은 측정이길 멈춤). Karwowski et al. (2024)는 강화학습 환경에서 이 동역학을 경험적으로 연구해 proxy reward를 공격적으로 최적화하면 true reward에서 흔히 발산함을 보였고, 같은 패턴이 프롬프트 반복에 나타난다 — 장황한 답을 보상하는 판사가 실사용자가 간결을 선호해도 프롬프트 수정을 장황으로 끌고, 키워드 겹침을 보상하는 참조 지표가 출력을 점수는 잘 받지만 읽기 나쁜 정형 문구로 끈다.
- 적용지침: 비용(cost)이 명명할 마지막 설계 차원 — 프로그래밍 검사는 본질적으로 무료, 임베딩 모델 참조 지표는 저렴, 판사 호출이 비싼 부분으로 부주의한 스위트가 추론보다 평가에 더 쓸 수 있다. 실무 패턴 — 매 예시에 싼 검사를 돌리고 판사 평가는 샘플 부분집합과 싼 검사가 의심스럽다고 표시한 예시에 예약하라. 결과는 모델 예산을 태우지 않으면서 높은 커버리지를 주는 스위트다.
- 핵심 통찰: 잘 설계된 스위트는 단일 숫자보다 유용한 것을 준다 — 어느 품질 차원이 향상·회귀하는지 알려줘 다음 프롬프트 반복이 어디 집중할지 안내한다. 스위트 자체를 제품으로 다뤄 프롬프트와 함께 버전 관리하라. 그 습관이 자리잡으면 평가 계층이 의존하는 모든 것의 신뢰할 토대가 된다.

## Summary (핵심 정리)

- 결정론적 규칙·스키마 검사는 무료·빠르고 CI에 끼어 형식 회귀를 잡지만 의미·그럴듯-하지만-틀린 출력을 놓치므로 baseline으로 계속 돌려라.
- 참조 기반 지표는 string-overlap(BLEU/ROUGE)부터 임베딩(BERTScore)까지의 슬라이딩 척도이며(Zhang 2020), 정답이 안정적일 때 유용하고 장문 개방형엔 판사로 옮겨라.
- LLM-as-Judge는 좁은 루브릭·보정 예시·순서 무작위화·근거 추론으로 편향을 줄이고 다른 패밀리의 강한 판사를 인간 레이블로 검증하며(Zheng 2023), 스위트는 hard gate·비용 계층화로 구성하되 metric gaming(Goodhart, Karwowski 2024)을 경계하라.
