# 11. Decomposition Within a Single Prompt

## 챕터 개요 (3줄 요약)

- 한 번에 너무 많은 것을 요구하는 프롬프트는 서두르고 일관성 없는 결과를 내므로, 복잡한 요청을 모델이 하나씩 처리할 작은 조각으로 쪼개는 분해(decomposition)를 단일 프롬프트 안에서 다룬다.
- 프롬프트 자체를 미니 워크플로(mini-workflow)로 취급해 각 섹션이 모델을 특정 작업으로 안내한 뒤 다음으로 넘어가게 하며, 추가 모델 호출이나 외부 오케스트레이션 없이 그 워크플로를 구축한다.
- 자체 무게에 짓눌리는 프롬프트를 알아보고, 하위 작업들을 이어 붙여 모델이 한 단계에서 다음 단계로 유용한 컨텍스트를 전달하게 하는 법을 배운다.

---

## 1. Recognizing Tasks That Need Decomposition

> 프롬프트가 실패할 때 그 실패 양식이 프롬프트 자체보다 분해 필요성을 더 명확히 신호하는 경우가 많다.

- 모델이 요청의 절반만 답하고 나머지를 잊거나, 두 요구가 충돌하는 출력을 내며 어느 쪽도 만족 못 시키는 패턴은 모두 한 뿌리 — 프롬프트가 여러 가지를 동시에 요구해 모델이 그것들을 뭉뚱그린 것 — 을 가리킨다.
- 적용지침: Press et al. (2023)는 모델이 다단계 질문의 구성요소를 개별로 물으면 맞히지만 한 질의로 묶으면 틀리는 "compositionality gap(구성성 간극)"을 발견했다 — 프롬프트가 문제의 자연스러운 분해와 맞는 명시적 구조를 제공하면 간극이 좁아진다. 모델이 분명히 부분적으로는 할 수 있는 작업에서 실패하면 보통 구조적 해결(분해)이 정답이다.
- 적용지침: 과부하 프롬프트의 경고 신호는 초기 출력에서 보인다 — 목표가 뒤섞이면(유용할 만큼 간결하지도, 입장을 취할 만큼 날카롭지도 않은 "요약이자 설득" 응답), 또는 같은 프롬프트가 실행마다 눈에 띄게 다른 결과를 내면(모델이 어느 하위 작업을 우선할지 암묵적으로 선택 중) 분해가 필요하다.
- 적용지침: Radhakrishnan et al. (2023)는 명시적 분해가 multi-hop 질문 정확도를 높일 뿐 아니라 더 충실한(faithful) 추론 트레이스를 낳음을 보였다 — 모델이 내는 단계가 실제 밟은 단계와 대응하며, 작업을 건너뛰어 도달한 답은 우연히 맞아도 취약하므로 신뢰에 중요하다.
- 적용지침: 또 다른 증상은 표면상 멀쩡하지만 프롬프트의 어느 부분이 출력의 어느 부분을 낳았는지 말할 수 없는 결과다 — 지시-결과 관계를 추적할 수 없으면 의도를 갖고 프롬프트를 개선할 수 없다. 분해는 각 지시에 응답 속 고유한 자리를 줘 추적성을 복원한다.

## 2. Identifying Natural Sub-Tasks

> 프롬프트가 분해 대상으로 표시되면, 다음 동작은 이음새가 실제로 어디 떨어지는지 식별하는 것이다(Figure 2-12: 기능·시간·의존 분할).

- 적용지침: 기능적 분해(functional decomposition)는 작업 종류로 쪼갠다 — 데이터를 추출한 뒤 설명하라는 요청은 같은 프롬프트 안 두 개의 명확히 레이블된 패스가 된다.
- 적용지침: 시간적 시퀀싱(temporal sequencing)은 사건이 일어나야 할 순서로 쪼갠다 — 이전에 생성한 개요를 기반으로 단락을 작성하는 것처럼 출력이 이전 상태에 의존할 때 유용하다.
- 적용지침: 의존성 매핑(dependency mapping)은 어느 하위 작업이 어느 하위 작업을 먹이는지 식별해 순서를 정하는 보완 기법으로, 모델이 아직 듣지 못한 것을 추론하려 하지 않게 한다.
- 적용지침: Khot et al. (2023)는 분해된 프롬프팅이 기호 조작·다단계 추론 작업에서 단일 지시 프롬프트를 일관되게 능가함을 보였다 — 그들의 접근은 여러 호출에 걸쳐 형식화됐지만 교훈은 in-prompt에도 전이된다: 각 하위 작업이 구체적일수록 모델 성능이 좋아진다.
- 적용지침: 하위 작업이 충분히 구체적인지 보는 유용한 테스트는 치환 검사(substitution check) — 그 하위 작업을 프롬프트 나머지를 못 본 동료에게 넘겨도 유용한 결과가 나올지 물어라. 다른 세 문장에 비춰야만 말이 되면 너무 모호해 자체 완결적으로 만들어야 한다.
- 적용지침: Wang et al. (2023b)의 plan-and-solve 프롬프팅은 모델이 먼저 명시적 계획(하위 작업 목록)을 생성한 뒤 순서대로 실행하게 한다 — 산술·상식 벤치마크에서 기본 CoT 대비 큰 이득을 냈으며, 계획 단계가 문제의 자연 구조를 표면화하는 역할 덕분이다.
- 적용지침: 하위 작업에 이름을 붙이면 모델이 방향을 잡는다 — "Step 1: 관련 개체를 식별하라"가 일반적 전환보다 낫다(어떤 작업을 수행하고 성공이 어떤 모습인지 알려주므로). 하위 작업 레이블은 추가 비용이 작고 거의 항상 명확성으로 보상한다.

## 3. Sequencing and Connecting Sub-Tasks

> 하위 작업 식별은 절반의 일이고, 나머지 절반은 각 단계가 다음 단계가 필요로 하는 것을 전달하도록 순서를 정하고 경계를 넘는 연결 조직(connective tissue)을 쓰는 것이다.

- 적용지침: 순서가 나쁜 분해는 단일 밀집 프롬프트보다 나쁠 수 있다 — 모델이 초기 하위 작업을 이후 선택지를 닫아버리는 방식으로 완료할 수 있기 때문이다.
- 적용지침: 기본 원리는 정보 의존성이다 — 다른 것의 출력을 요하는 하위 작업은 그 뒤에 와야 하며, "방금 만든 목록을 사용해" 또는 "2단계의 요약을 기반으로" 같은 전환으로 의존성을 명시하라. 이 전환은 모델에게 어느 이전 내용을 끌어올지 알려주는 메모리 단서로 작동한다.
- 적용지침: Zhou et al. (2022)의 least-to-most prompting(모델이 먼저 문제를 분해한 뒤 이전 해를 명시적으로 참조하며 하위 문제를 순서대로 푸는 방식)은 CoT 단독보다 더 어려운 문제 인스턴스로 잘 일반화되며, 향상은 주로 명시적 참조에서 온다.
- 적용지침: 중간 요약(intermediate summary)도 유용하다 — 긴 출력을 내는 하위 작업 뒤에 다음으로 넘어가기 전 결과를 요약하라는 짧은 지시를 두면 이후 단계가 집중을 유지한다(그렇지 않으면 모델이 이전 텍스트 전체를 균일하게 가중해 가장 필요한 신호가 흐려진다). 두 하위 작업 사이에 한 줄 요약 지시를 더하는 것이 어느 하위 작업을 재작성하는 것보다 출력 품질에 더 기여하는 경우가 많다.
- 적용지침: 명시적 핸드오프(handoff)도 비슷한 역할을 한다 — 하위 작업이 다음 단계가 다룰 구조화 데이터를 내면, 핸드오프 형식을 프롬프트 안에 명시하라(Module 07). Wei et al. (2022)는 추론 단계를 일관된 형식으로 외부화하면 다단계 성능이 향상됨을 관찰했고, 같은 원리가 하위 작업 경계에 적용된다.
- 적용지침: 과분해(overdecomposition)의 비용도 있다 — 모든 하위 작업을 사소함 수준까지 쪼개면 프롬프트가 컨텍스트를 잡아먹고 응답을 느리게 하며 품질에 거의 기여 못 하는 미세 지시의 벽이 된다. 핵심은 올바른 입도(grain)를 찾는 것으로, 보통 각 하위 작업이 동료에게 메모로 쓸 단일한 일관된 사고에 대응하는 수준이다(그 아래로 가면 모델이 프롬프트의 큰 호를 잃는다).
- 적용지침 예시: "Step 1, 아래 제품 설명에서 타깃 청중을 식별하라. Step 2, 1단계에서 식별한 청중을 사용해 가장 관련 있는 가치 제안 두 개를 나열한 뒤 그것을 쓰는 한 단락 브리프를 작성하라. 작성 전, 청중과 가치 제안을 한 줄 요약으로 재진술하라."

## 4. Decomposition Versus Chaining

> 이 모듈의 기법은 전적으로 단일 모델 호출 안에 산다 — 프롬프트는 길고 구조화되지만 오케스트레이션이 프롬프트 텍스트 자체에서 일어난다. 멀티 프롬프트 체이닝(chaining)은 작업을 별도 호출로 나눠 한 호출 출력이 외부 코드를 거쳐 다음 입력을 먹인다(책 후반).

- 적용지침: 둘의 선택은 흔히 복잡성 문제로 오해되지만, 더 유용하게는 컨텍스트가 어디에 있어야 하는가의 문제다.
- 적용지침: in-prompt 분해는 전체 워크플로가 컨텍스트 윈도우에 편안히 들어가고 각 하위 작업이 이전 추론 사슬 전체를 보는 것이 이득일 때 적합하다 — 빠르고 단순하며 엔지니어링 오버헤드가 적다. 한계는 오류가 응답 내부에서 복리되고, 다음 단계 실행 전 중간 출력을 검사할 기회가 없다는 점이다.
- 적용지침: 체이닝은 중간 출력이 다음 단계 진행 전 검증을 요하거나, 전체 컨텍스트를 단일 프롬프트에 두면 실용 한계를 넘을 때 적합하다. Khot et al. (2023)는 이 구분을 모듈성의 하나로 봤다 — 체인 파이프라인은 각 하위 작업을 자체 계약을 가진 호출 가능 함수로, in-prompt 분해는 하위 작업을 단일 지시의 정렬된 섹션으로 다룬다. 둘 다 같은 기저 논리(일을 관리 가능한 조각으로 쪼개기)를 공유하며 경계가 강제되는 위치만 다르다.
- 적용지침: 실무 규칙 — 외부 오케스트레이션이 명백히 필요치 않은 작업은 in-prompt 분해로 시작하라. 구조화 프롬프트가 안정적 고품질 출력을 내면 체인의 추가 인프라가 불필요하고, 신중한 시퀀싱·명시적 핸드오프 뒤에도 출력이 불안정하면 그것이 체이닝으로 옮겨 각 하위 작업을 자체 호출로 격리하라는 신호다.
- 적용지침: 유용한 중간 경로 — 프롬프트를 텍스트 안에서 분해해 한 번 실행하고 검사용으로 각 레이블된 섹션을 손으로 검토한 뒤, 구조를 그대로 둘지 체인 파이프라인으로 승격할지 결정하라.
- 핵심 통찰: 특정 기법을 넘어선 더 깊은 요점은 분해가 프롬프트에 대한 사고 방식이라는 것이다 — 프롬프트를 명명된 단계와 단계 간 명시적 데이터 흐름을 가진 작은 프로그램으로 다루기 시작하면, 요청 자체가 읽기 쉬워져 모델을 안내하기 쉬워지고 프롬프트 품질이 전반적으로 개선된다.

## Summary (핵심 정리)

- 과부하 프롬프트(목표 충돌·일부 누락·실행 간 불일치)는 분해가 필요하다는 신호이며, 단일 프롬프트를 명명된 단계의 미니 워크플로로 다루면 정확도와 충실성이 함께 오른다(Press 2023, Radhakrishnan 2023).
- 기능·시간·의존 기준으로 하위 작업을 쪼개고 치환 검사로 구체성을 확보하며, 명시적 전환·중간 요약·핸드오프로 단계를 연결하되 과분해를 피하라(Khot 2023, Zhou 2022, Wang 2023b).
- in-prompt 분해는 컨텍스트가 한 호출에 들어가고 빠른 반복이 필요할 때, 체이닝은 중간 검증·컨텍스트 한계가 있을 때 적합하니 분해부터 시작해 필요 시 체인으로 승격하라.
