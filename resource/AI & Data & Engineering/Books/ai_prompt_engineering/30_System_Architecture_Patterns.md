# 30. System Architecture Patterns

## 챕터 개요 (3줄 요약)

- 앞선 모듈이 체인·검색 컴포넌트 같은 개별 빌딩 블록을 다뤘다면, 프로덕션 시스템은 이 요소들을 반복되는 문제 부류를 푸는 알아볼 만한 아키텍처 패턴으로 결합한다.
- 프롬프트 기반 애플리케이션 구축 시 숙련 실무자가 손을 뻗는 지배적 설계(Assistant, Workflow, Hybrid)를 적응·확장 가능한 참조 설계로 다루며, 어떤 패턴이 어떤 문제에 맞고 각각의 트레이드오프를 저울질하는 법을 배운다.
- 패턴 조합의 통합 지점(integration point)이 설계 위험 대부분을 운반하므로 그것을 명시 답변하는 것이 핵심이다.

---

## 1. The Assistant Pattern

> Assistant Pattern은 사용자와 모델이 지속 교환에 참여하는 대화형 시스템 아키텍처를 기술한다 — 헬프 센터의 챗봇과 에디터 속 코딩 코파일럿 모두 이 설계에 속한다.

- 적용지침: 이들을 묶는 것은 여러 턴에 걸쳐 일관된 행동을 유지하면서 매 단계 신선한 사용자 입력을 통합할 필요다. 정의 컴포넌트는 실행 대화를 추적하는 세션 매니저(session manager)와 고정 시스템 프롬프트를 동적 컨텍스트와 섞는 계층 지시 스택(layered instruction stack)이다. 컨트롤러가 외부 정보가 필요할 때 도구 호출을 다루고, 라우팅 계층이 필요 시 요청을 특화 프롬프트로 넘긴다(Figure 5-10).
- 적용지침: assistant 설계 시 무엇이 일정하고 무엇이 턴마다 변하는지 결정하는 데서 시작하라 — 시스템 프롬프트가 페르소나·안전 제약을 인코딩하고 세션 수명 동안 고정 유지된다. 대화 이력이 시간에 따라 커지고 대부분 프로덕션 시스템이 실행 컨텍스트가 모델 유효 한계에 다가가면 오래된 턴을 요약으로 압축한다. 일부 팀은 세션 간 지속되는 안정 사용자 선호 프로필을 붙여 어시스턴트에 닫힌 윈도우를 넘는 기억을 준다.
- 적용지침: 도구 사용은 어시스턴트가 모델이 구조화 요청(흔히 JSON 함수 호출)을 내고 외부 시스템이 결과 반환 전 실행하는 능력에 의존한다 — 오케스트레이션 코드가 사용자 턴과 도구 턴을 모델이 계속 읽는 단일 일관 트랜스크립트로 교차시킨다. 이 루프의 신뢰성은 가용 도구가 프롬프트에 얼마나 명확히 기술됐는지에 강하게 의존한다(Module 17).
- 적용지침: 세션이 수백 턴으로 늘면 아키텍처 질문은 사용자가 신경 쓰는 맥락을 잃지 않으면서 모델 유효 컨텍스트를 타이트하게 유지하는 법이다 — 일부 팀은 매 턴 갱신하는 구조화 사실 스토어(이름·핵심 결정을 작은 JSON에 포착)를 유지하고, 다음 프롬프트가 시스템 프롬프트와 최근 몇 턴을 그 사이 삽입된 구조화 스토어와 결합해, 매 단계 전체 트랜스크립트 재독 없이 안정적 사용자 그림을 모델에 준다.
- 적용지침: 실패 양식은 좀처럼 영리한 모델 실수가 아니다 — 흔한 문제는 어시스턴트에 한 번에 너무 많은 도구를 줘 프롬프트가 모델 어텐션을 희석하는 긴 카탈로그가 되는 것. 유용한 규율 — 활성 도구셋을 작게 유지하고 복잡성이 커지면 특화 요청을 전용 어시스턴트로 라우팅하라(두 좁은 전문가가 기저 작업이 형태에서 다를 때 하나의 넓은 제너럴리스트를 보통 능가).

## 2. The Workflow Pattern

> Workflow Pattern은 구조화 입력이 결정론적 파이프라인을 거쳐 정의된 출력으로 가는 작업 지향 아키텍처를 다룬다 — 영수증 업로드 후 필드 추출·검증, 긴 기사 요약 후 스타일 채점 등이 예다. 이 시스템은 신뢰성·재현성을 중시하고 구성 후 보통 인간 없이 돈다.

- 적용지침: 워크플로는 각각 무엇을 소비·생산하는지 명확한 계약을 가진 정렬된 단계 시퀀스다 — 일부 단계는 프롬프트 템플릿으로 모델을 호출하고, 일부는 파싱·검증 같은 결정론적 연산을 수행한다(Module 07). 모델이 여러 컴포넌트 중 하나로 다뤄지고 주변 코드가 재시도·일관성 검사를 책임진다.
- 적용지침: 워크플로는 작업이 잘 정의되고 반복적일 때 가장 유용하다(금융 문서 처리·임상 노트 정보 추출). 강점은 예측 가능성 — 각 단계를 격리 테스트할 수 있고 실패가 특정 단계에 국한되어, 한 단계가 오작동하면 진단 경로가 짧고 팀이 나머지 파이프라인을 교란 않고 한 프롬프트·한 파서를 갱신할 수 있다.
- 적용지침: 일부 워크플로는 중간 모델 출력에 기반한 조건부 분기를 포함한다 — 문서 분류기(Module 26)가 먼저 입력이 계약인지 인보이스인지 정하고 거기서 파이프라인이 갈린다. 분기는 표현력을 더하지만 테스트 부담을 키운다(시스템의 모든 도달 가능 경로가 자체 평가 세트 필요). 팀은 흔히 엄격한 선형 파이프라인으로 시작하고 단일 제너릭 프롬프트 비용이 너무 높을 때만 분기를 더한다.
- 적용지침 예시(인보이스 처리): 1단계 OCR(결정론적), 2단계 인식 텍스트를 vendor·total JSON 스키마를 명시한 프롬프트의 모델에 전달, 3단계 JSON을 스키마에 대해 검증하고 잘못된 출력을 큐로 재시도, 4단계 숫자 합계를 line-item 합과 교차 검증, 최종 단계 검증 결과를 회계 시스템에 기록 — 모델이 어려운 파싱을 다루고 주변 코드가 정확성을 강제(Figure 5-11: 결정론적 단계 + 모델 추출 단계의 하이브리드 파이프라인).
- 적용지침: 채팅에서 워크플로로 옮기면 지연 기대가 빠르게 재보정된다 — 워크플로는 밤새 돌 수 있어 그 여유가 라이브 대화에서 굼떠 보일 더 강한 모델·자기 검사 패스를 적용하기 쉽게 한다. 각 단계를 명시적 입출력 스키마로 쓰는 규율이 프로덕션 디버깅에도 보상한다(실패 실행이 알려진 단계를 거치는 단일 트레이스가 되어 어느 단계가 오작동했는지 보통 스스로 답함).

## 3. Hybrid and Composite Architectures

> 실제 프로덕션 시스템은 좀처럼 단일 패턴에 깔끔히 맞지 않는다.

- 적용지침: 코딩 코파일럿은 표면상 어시스턴트로 보이나, 도구 계층이 워크플로 단계를 호출하고 코드 저장소 검색이 지식 시스템 패턴을 빌린다. 상용 코딩 코파일럿 연구는 다양한 개발자 집단 전반에서 상당한 생산성 이득을 보고해 이 복합 설계의 규모 효과를 보여준다(Ziegler et al. 2024).
- 적용지침: 하이브리드 아키텍처는 단순 패턴이 한계에 부딪힐 때 등장한다 — 순수 어시스턴트는 단일 사용자 요청이 대화 중 재시도되면 안 되는 길고 결정론적 파이프라인을 트리거할 때 어려움을 겪고, 순수 워크플로는 사용자가 출력에 후속 질문하려 할 때 경직되게 느껴지며, 순수 지식 시스템은 사용자가 질문을 여러 차례 정제하려 할 때 생기 없게 느껴진다. 하이브리드 해법은 워크플로를 어시스턴트가 호출할 수 있는 도구로 다뤄, 워크플로가 배경에서 도는 동안 사용자가 결과에 대해 대화하게 하는 세션 토큰을 반환하는 것이다.
- 적용지침: 실무에서 등장하는 복합 패턴은 흔히 유기적으로 자란다 — 팀이 단일 사용 사례에 깨끗한 어시스턴트로 시작하고, 시간이 지나 같은 종류 분석의 반복 요청이 워크플로를 도구 호출 뒤로 깎아내야 함을 시사하며, 내부 문서 말뭉치가 커지며 지식 시스템이 또 다른 도구 호출 뒤로 미끄러져 들어온다. 최종 결과가 어떤 단일 교과서 다이어그램과도 거의 안 맞지만 기저 컴포넌트는 앞서 도입한 패턴을 내면화한 누구에게나 알아볼 만하다.
- 적용지침: 패턴을 조합할 때 통합 지점(integration point)이 설계 위험 대부분을 운반한다 — 상태가 어디 사는가(어시스턴트 세션 vs 워크플로 실행)? 워크플로가 대화로 진행을 어떻게 보고하는가? 검색이 워크플로 안에서 아무 유용한 것도 안 내면 어떻게 되는가? 이 질문들은 아키텍처 문서에서 명시 답변을 받아 마땅하며, 흔히 시스템이 실사용자로 스트레스 테스트될 때만 표면화된다.

## 4. Durable Composite Design

> 더 견고한 복합 설계는 각 컴포넌트가 단일하고 명확히 명명된 책임을 갖는 것이다.

- 적용지침: 시스템이 어시스턴트와 워크플로 중 어느 것이 한 상태 조각을 소유하는지 혼란스러워지면 디버깅이 고통스러워지고 작은 버그가 깨지기 쉬운 코드베이스로 누적된다.
- 적용지침: 보상하는 규율은 패턴 간 경계를 문서화된 스키마·오류 모드를 가진 integral interface로 다뤄, 각 조각이 이웃을 깨지 않고 진화할 수 있게 하는 것이다.
- 핵심 통찰: 패턴(Assistant·Workflow·Hybrid)을 참조 설계로 알아보되, 가장 신뢰할 시스템은 컴포넌트 책임이 명확하고 통합 지점이 명시적으로 설계된 것이다.

## Summary (핵심 정리)

- Assistant Pattern은 세션 매니저+계층 지시 스택+도구 호출로 다중 턴 일관성을 유지하며, 도구셋을 작게 유지하고 복잡성 증가 시 전용 어시스턴트로 라우팅하라(Figure 5-10).
- Workflow Pattern은 명확한 입출력 계약을 가진 결정론적 단계 시퀀스로 신뢰성·재현성을 중시하며, 격리 테스트·짧은 진단 경로가 강점이고 분기는 테스트 부담을 키운다.
- 실제 시스템은 하이브리드로 유기적으로 자라며(Ziegler 2024), 통합 지점(상태 위치·진행 보고·검색 실패 처리)이 설계 위험 대부분을 운반하므로 경계를 문서화된 스키마·오류 모드를 가진 인터페이스로 다뤄라.
