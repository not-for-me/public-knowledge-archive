# 04. Modeling and Refactoring Patterns

## 챕터 개요 (3줄 요약)

- 그래프 모델링의 고급 패턴인 하이퍼엣지(hyperedge), 시간 기반 버전 관리(time-based versioning), 시퀀스 표현(sequence)을 음악 도메인 예시로 다룬다.
- 사용 사례가 진화하면 그래프를 재구성(refactoring)해야 하므로, 관계 타입 변경·속성을 노드로·관계를 노드로 바꾸는 단계별 리팩터링 패턴을 실습한다.
- 각 패턴의 장단점을 가늠하고 언제 적용할지 판단하는 능력을 키우며, 후방 호환성(backward compatibility)을 유지하며 점진적으로 전환하는 방법을 보여준다.

---

## 1. Hyperedges: N-way Relationships

> 두 노드 사이의 단순 관계로 표현하기 어려운, 셋 이상의 엔티티가 얽힌 사실은 중간 노드(intermediate node)로 모델링한다.

- 예: 곡의 커버(cover) 기능 — 어떤 아티스트의 녹음이 커버인지, 그렇다면 원곡 아티스트가 누구인지를 추적해야 한다.
- 가장 단순한 방법은 `ARTIST` 관계에 "커버 여부"를 표시하는 것이지만, N-방향(N-way) 정보(아티스트·트랙·원곡 관계)를 담기에는 부족하다.
- 하이퍼엣지는 관계 자체를 노드로 승격(reify)해 여러 엔티티를 하나의 사실로 연결한다.
- 이를 통해 "원곡 → 커버들", "커버 → 원곡" 양방향 탐색이 깔끔해진다.

---

## 2. Time-Based Versioning

> 인프라·리테일 등 일부 도메인은 그래프를 시간에 따라 버전 관리하여 변화 이력을 추적해야 한다.

- 활용: 네트워크가 시간에 따라 어떻게 바뀌었는지, 행동 변화 추적, 과거 기록을 보존하면서 엔티티 갱신.
- 예시: 사용자의 "Favorites" 플레이리스트를 버전 관리해 시간에 따른 음악 취향 변화를 추적한다.
- 시간 감쇠(time decay)를 적용해 과거에 좋아했지만 지금은 덜한 아티스트/장르에 낮은 가중치를 부여, 추천 관련성을 높인다.
- 버전 노드를 시간 순으로 연결하면 특정 시점의 상태를 재구성할 수 있다.

---

## 3. Representing Sequences

> 순서가 중요한 데이터(플레이리스트의 트랙 순서 등)는 위치 속성보다 연결 리스트(linked list)로 표현하는 편이 효율적이다.

- PoC 모델은 `HAS_TRACK` 관계의 `position` 속성으로 순서를 표현했으나 단점이 드러났다.
- 마지막 트랙 찾기: 모든 `HAS_TRACK` 관계를 순회하며 `position`을 비교해야 해 비효율적이다.
- 대안: 트랙을 `NEXT` 류의 관계로 순차 연결한 연결 리스트로 만들면 첫/마지막/다음 트랙 탐색이 자연스럽다.
- 중간 노드(예: PlaylistTrack)를 두면 같은 트랙이 여러 플레이리스트에 다른 위치로 존재하는 경우도 표현된다.

```
Property-based:  (Playlist)-[:HAS_TRACK {position:1}]->(TrackA)
                 (Playlist)-[:HAS_TRACK {position:2}]->(TrackB)
Linked-list:     (Playlist)-[:FIRST]->(PT1)-[:NEXT]->(PT2)-[:NEXT]->(PT3)
                 PT1->TrackA, PT2->TrackB, PT3->TrackC
```

## 4. Refactoring Patterns

> 모델이 진화하면 후방 호환성을 유지하며 단계적으로 그래프를 재구성한다.

- 관계 타입 변경(change relationship type): Neo4j는 관계 타입 변경을 직접 지원하지 않으므로, 기존 `HAS_TRACK`을 두고 `ON_PLAYLIST`를 새로 만들어 병행하다가 쿼리 갱신 후 기존 관계를 삭제하는 다단계(Stage 1~3) 방식을 쓴다.
- 속성을 노드로(node from a property): 아티스트의 `genre` 속성을 별도 `Genre` 노드로 추출해 장르 간 순회와 공유를 가능하게 한다(`LOAD CSV`로 보강).
- 관계를 노드로(node from a relationship): `ON_PLAYLIST` 관계를 `PlaylistTrack` 중간 노드로 변환하고 연결 리스트로 이어, 시퀀스 패턴을 적용한다(Stage 1~4).
- 리팩터링은 현실적인 예제로 먼저 검증해 잠재 문제를 조기에 드러내는 것이 좋다.

---

## Summary (핵심 정리)

- 하이퍼엣지·시간 버전 관리·시퀀스 표현은 단순 관계로 담기 어려운 복잡한 사실을 중간 노드와 연결 리스트로 우아하게 모델링한다.
- Neo4j는 관계 타입 직접 변경을 지원하지 않으므로, 새 관계/노드를 병행 생성하고 쿼리 전환 후 정리하는 다단계 리팩터링으로 후방 호환성을 지킨다.
- 속성→노드, 관계→노드 리팩터링은 데이터의 공유성과 순회 효율을 높여 진화하는 사용 사례에 대응한다.
