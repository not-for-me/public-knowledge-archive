# 03. Revisiting Modeling Decisions

## 챕터 개요 (3줄 요약)

- 동일한 데이터를 그래프로 모델링하는 정답은 하나가 아니라 "상황에 따라 다르다(It Depends)" — 사용 사례·카디널리티·성능에 따라 선택이 달라진다.
- 속성 대 노드, 라벨 활용, 노드 팬아웃, 슈퍼노드, 관계 세분화, 한정/버킷/양방향 관계 등 핵심 모델링 트레이드오프를 음악 도메인 예시로 설명한다.
- 좋은 그래프 모델은 인간 직관성(human intuitiveness), 쿼리 단순성, 쿼리 성능을 동시에 만족시키며, 모델의 복잡도가 곧 쿼리의 복잡도로 이어진다.

---

## 1. It Depends & Principles of Modeling

> 트랙·앨범·아티스트·장르 데이터를 그래프로 모델링하는 방식은 여러 가지가 모두 유효하며, 핵심은 트레이드오프를 이해하는 것이다.

- 잘 튜닝된 그래프 모델의 특징: 인간 직관성(화이트보드 친화적), 쿼리 단순성, 쿼리 성능.
- Cypher는 표현력 있는 언어이며, 쿼리 단순성은 모델에서 직접 파생된다 — 불필요하게 복잡한 모델은 복잡한 쿼리와 인지 부담을 낳는다.
- 잘 설계된 모델은 합리적인 선택(속성/노드/라벨/관계)에 의해 뒷받침된다.
- 모델 선택은 주된 사용 사례(use case)와 카디널리티(cardinality)에 좌우된다.

---

## 2. Properties Versus Nodes

> 앨범 같은 엔티티를 노드로 둘지 속성으로 둘지는 그 값이 결과를 장식만 하는지, 아니면 탐색·조건에 쓰이는지에 달려 있다.

- 장식 속성(decorating property): 결과를 풍부하게 하지만 탐색이나 조건(predicate)에 쓰이지 않는 값은 속성으로 두는 편이 간단하다(예: 트랙 조회 시 함께 반환되는 앨범).
- 공통점으로 순회(Traversing Across Commonalities): "같은 앨범의 다른 트랙 찾기"처럼 공통 값으로 엔티티를 연결할 땐, 속성 비교는 전체 스캔이 필요해 비효율적이다.
- 이런 경우 공유 값을 노드로 만들어 관계로 연결하면 직접 순회가 가능해 성능이 좋다.
- 리스트 속성으로 공통값을 표현하면 교집합 검사가 필요해 쿼리가 더 복잡해진다.

## 3. Labels, Node Fanout, and Supernodes

> 범주형 변수는 속성·노드·라벨로 표현할 수 있으며, 모델은 팬아웃과 슈퍼노드 사이에서 균형을 잡아야 한다.

- 라벨로 개념 모델링(Modeling Concepts as Labels): 장르·국적·상태·성별 같은 범주형 변수(categorical variable, 값이 유한)는 속성/노드/라벨 중 선택할 수 있다.
- 노드 팬아웃(node fanout): 한 노드가 가진 연결 수. 모든 정보를 속성으로 압축(팬아웃 0)하거나 모든 속성을 노드로 분리(최대 팬아웃, RDF 유사)하는 양극단은 모두 부적합하다.
- 슈퍼노드(supernode): 같은 유형 대비 비정상적으로 많은 관계(수천~수백만)를 가진 노드로, 시각화 시 "헤어볼(hairball)" 현상을 유발한다.
- 일부 슈퍼노드는 예측 가능하므로 모델 설계 단계에서 회피할 수 있다.

```
Fanout=0:  (Track {title, album, genre, year, ...})   <- all properties
Fanout=max: (Track)-[:HAS_TITLE]->(Title), -[:HAS_GENRE]->(Genre), ...  <- RDF-like
Balanced:  (Track)-[:IN_ALBUM]->(Album), (Track)-[:HAS_GENRE]->(Genre)
```

## 4. Relationship Design Patterns

> 관계의 세분화 정도, 가중치 표현 방식, 고볼륨 관계의 버킷팅, 그리고 대칭 관계 처리는 모두 모델 성능에 직접 영향을 준다.

- 관계 세분화(relationship granularity): 범용 `HAS_TRACK`처럼 한 타입을 여러 맥락에 쓸지, 구체적 타입으로 나눌지는 주요 순회 패턴에 따라 결정한다.
- 한정 관계(qualified relationships): 관계 강도를 표현할 때 명시적 타입(`LOVES`/`HATES`)을 쓰거나, 범용 타입에 `level` 같은 속성을 둔다.
- 버킷 관계(bucketed relationships): `PLAYED` 카운터처럼 세분화가 모호하고 양이 매우 많은 관계는, 범위(예: 100~250회)별로 버킷팅하면 쿼리 효율이 좋아진다.
- 양방향 관계(bidirectional relationships): Neo4j의 모든 관계는 타입·방향·시작/끝 노드를 가지지만, 성능 저하 없이 양방향 순회가 가능하다.
- 대칭 관계(symmetric relationship)를 양방향으로 두 번 저장하는 것은 안티패턴이다 — `SIMILAR`처럼 의미가 동일하면 한 방향만 저장하고 양방향으로 질의한다.

---

## Summary (핵심 정리)

- 그래프 모델링에 단일 정답은 없으며, 사용 사례·카디널리티·성능 트레이드오프에 따라 속성/노드/라벨/관계를 선택한다.
- 모델의 직관성과 단순성이 곧 쿼리의 단순성과 성능으로 이어지므로, 팬아웃·슈퍼노드·관계 세분화를 신중히 설계해야 한다.
- 대칭 관계 중복 저장 같은 안티패턴을 피하고, 고볼륨 관계는 버킷팅으로 최적화한다.
