# 07. Search

## 챕터 개요 (3줄 요약)

- Neo4j에서 텍스트 검색을 구현하는 방법으로, 정확/부분 일치 검색과 TEXT 인덱스(트라이그램), 전문 검색(full-text) 인덱스를 비교한다.
- Apache Lucene 기반 전문 검색이 제공하는 멀티토큰·구문(phrase)·와일드카드·퍼지(fuzzy) 검색을 음악 트랙 예시로 다룬다.
- 토큰화(tokenization), 특수문자(해시태그/멘션), 식별자·IP 주소, 불용어(stopword), 그래프 패턴 결합 시 성능 같은 실전 인덱스 설계 고려사항을 설명한다.

---

## 1. What Is Search? & Searching for Data

> 검색은 질의(query)를 시스템에 던져 원하는 항목을 효율적으로 찾는 과정이며, 인덱스가 그 효율의 핵심이다.

- 비유: 레코드 가게에서 알파벳 분류대를 따라가 원하는 음반을 찾는 것이 일종의 검색이다.
- 정확 일치(`=`)만 사용하면 인덱스가 없을 때 쿼리가 멈출(spinning) 수 있다 — 적절한 인덱스가 필요하다.
- 부분 검색(partial searches): `CONTAINS`(어디든 포함), `STARTS WITH`(접두), `ENDS WITH`(접미)로 부분 일치를 찾는다.
- `=`는 정확 일치만 반환하므로 라이브 버전·리믹스 등은 부분 검색 연산자로 찾아야 한다.

---

## 2. TEXT Indexes (Trigram)

> TEXT 인덱스는 n-그램(trigram, n=3) 구조로 텍스트를 색인해 부분 검색을 빠르게 한다.

- n-그램(n-gram): 인접한 n개 글자의 순서 있는 시퀀스. Neo4j는 n=3을 사용해 흔히 트라이그램(trigram) 인덱스라 부른다.
- 예: "purple" → pur, urp, rpl, ple 로 분해되어 색인된다.
- 쿼리 입력도 트라이그램으로 분해해 매칭하므로 `CONTAINS` 류 부분 검색이 빨라진다.
- 더 빠른 조회를 위해 디스크 공간을 더 쓰는 트레이드오프가 있다.

```
TEXT index (trigram) of "purple":
  purple -> [pur][urp][rpl][ple]
query "rpl" matches "purple" via trigram lookup
```

---

## 3. Full-Text Indexes (Lucene)

> 전문 검색 인덱스는 텍스트를 토큰(보통 단어) 단위로 쪼개 색인하는 Apache Lucene의 역색인(inverted index) 기반으로, 유연하고 강력한 검색을 제공한다.

- 문자열 인덱스가 값 전체를 한 단위로 다루는 것과 달리, 토큰 기반 인덱스는 단어별로 따로 색인한다.
- 멀티토큰 검색(multitoken): `db.index.fulltext.queryNodes`로 "purple rain"을 검색하면 점수(score)와 함께 결과를 반환하나, "purple"만 든 결과도 섞일 수 있다.
- 구문 검색(phrase): 정확한 단어 순서로 검색한다.
- 와일드카드(wildcard) 검색: 접두/접미 패턴(`*`)으로 검색한다.
- 퍼지 검색(fuzzy): 철자 오류·근접 단어까지 허용해 검색한다.

---

## 4. Index and Query Considerations

> 분석기(analyzer)의 토큰화·불용어 처리 방식과 그래프 패턴 결합 성능을 이해해야 의도한 검색 결과를 얻는다.

- 토큰화(tokenization): 공백·구두점·비문자 경계로 텍스트를 토큰으로 분할하며, ASCII 문자 축소(collapsing)로 발음 구별 부호 등을 정규화한다.
- 특수문자(해시태그·멘션): 기본 분석기가 `#`, `@`를 경계로 처리하므로 별도 고려가 필요하다.
- 식별자·IP 주소: 기본 `standard-no-stopwords` 분석기는 `V-1234`를 `V`와 `1234`로 분해하고 단일 글자 토큰(`V`)은 색인하지 않아, `V-*` 검색이 실패할 수 있다.
- 불용어(stopwords): `and`, `the`, `is` 같은 흔한 단어를 제거해 관련성과 효율을 높이지만, 의도치 않게 검색 대상이 사라질 수 있다.
- 그래프 패턴과의 성능: Cypher 플래너는 그래프 패턴과 결합된 전문 검색에서 TEXT 인덱스만큼 잘 동작하지 않아, 복잡한 쿼리 설계가 필요하다.

---

## Summary (핵심 정리)

- 부분 검색에는 트라이그램 기반 TEXT 인덱스가, 멀티토큰·퍼지·와일드카드 같은 유연한 검색에는 Lucene 기반 전문 검색 인덱스가 적합하다.
- 분석기의 토큰화·불용어·특수문자 처리 방식을 이해하지 못하면 식별자나 IP 주소 검색이 실패할 수 있으므로 색인 설계 시 주의해야 한다.
- 전문 검색을 그래프 패턴과 결합할 때는 플래너 성능 한계를 고려해 쿼리를 신중히 구성한다.
