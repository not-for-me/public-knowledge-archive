# 03. Lists and Tuples

## 챕터 개요 (3줄 요약)

- 리스트(list)와 튜플(tuple)은 모두 배열(array) 계열 자료구조로, 내재된 순서(intrinsic ordering)를 알면 인덱스 조회가 O(1)로 이루어진다.
- 리스트는 동적 배열(가변, 리사이즈 가능)이고 튜플은 정적 배열(불변, 경량)이며, 리스트는 append 시 약 12.5% 과할당(overallocation)으로 메모리를 더 쓴다.
- 정렬된 데이터에서는 이진 탐색(binary search, O(log n))이 효율적이며, 올바른 자료구조를 선택해 유지하는 것이 고성능 코드의 핵심 원칙이다.

---

## 1. Arrays and O(1) Lookup (배열과 O(1) 조회)

> 시스템 메모리는 번호가 매겨진 버킷(bucket)들의 연속이며, 데이터가 어느 버킷에서 시작하는지 알면 임의 원소를 한 단계(O(1))에 찾을 수 있다.

- 리스트와 튜플은 배열 계열로, 원소들의 상대적 순서가 원소 값만큼 중요하다.
- Python은 데이터를 참조(reference)로 저장하므로 버킷에는 실제 데이터를 가리키는 정수 크기 포인터가 들어가, 어떤 타입이든 담을 수 있다.
- 배열 생성 시 커널에 N개의 연속 버킷을 요청해 메모리 블록을 할당받는다.
- 원소 i를 찾으려면 시작 버킷 M에서 M+i 위치를 읽으면 되므로, 배열 크기와 무관하게 O(1)이다.
- 이는 numpy 배열과 대비되는데, numpy는 정적 타입만 담아 더 효율적이다(6장 참조).

### Memory Layout (메모리 배치 개념도)

```
  array start = M
  index:   0     1     2     3     4     5
  bucket: [M]   [M+1] [M+2] [M+3] [M+4] [M+5]
           |
           v
        pointer -> actual object in memory
  lookup(i) = read bucket at (M + i)   --> O(1)
```

## 2. Searching — Linear vs Binary (선형 탐색 vs 이진 탐색)

> 순서를 모르는 배열에서 원소를 찾으려면 선형 탐색(O(n))이 필요하지만, 데이터를 정렬하면 이진 탐색으로 O(log n)에 찾을 수 있다.

- 선형 탐색(linear search)은 모든 원소를 순회하며 비교하므로 최악의 경우 O(n)이고, `list.index()`가 이 방식을 쓴다.
- 정렬된 데이터에서는 `__eq__`와 `__lt__` 매직 메서드로 비교가 이루어지며, `functools.total_ordering`으로 나머지 비교 함수를 자동 정의할 수 있다.
- Python의 리스트 정렬은 Tim sort를 쓰며 최선 O(n), 최악 O(n log n)이고, 삽입 정렬과 병합 정렬을 혼합한다.
- 이진 탐색(binary search)은 중앙값과 비교하며 절반씩 좁혀 평균 O(log n)에 원소를 찾는다.
- 데이터가 본질적으로 정렬돼 있다면 딕셔너리로 변환(O(n)) 후 조회하기보다 이진 탐색(O(log n))이 더 효율적이다.
- 표준 라이브러리 `bisect` 모듈은 정렬을 유지하며 삽입(`insort`)하고 최적화된 이진 탐색·근사값 찾기를 제공한다.

### 이진 탐색 예제 (원문 의도 유지, 새 예제로 재작성)

```python
def binary_search(haystack, needle):
    lo, hi = 0, len(haystack) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if haystack[mid] == needle:
            return mid
        elif haystack[mid] < needle:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1

# bisect로 정렬 유지하며 삽입 + 근사값 찾기
import bisect
data = []
for v in (734, 14, 496, 992, 265):
    bisect.insort(data, v)      # 항상 정렬 상태 유지
pos = bisect.bisect_left(data, 500)   # 500 이상 첫 위치
```

## 3. Lists Versus Tuples (리스트 vs 튜플)

> 둘 다 같은 배열 구조를 쓰지만, 리스트는 가변·리사이즈 가능한 동적 배열이고 튜플은 불변·경량의 정적 배열이며 런타임이 캐싱한다.

- 튜플은 "변하지 않는 하나의 대상의 여러 속성"(예: 전화번호, 다항식 계수)을 표현하기에 적합하다.
- 리스트는 "내용과 크기가 계속 변하는 동종 컬렉션"(예: 현재 독자 명단)에 적합하다.
- 둘 다 혼합 타입을 담을 수 있지만, 이는 오버헤드를 늘리고 최적화를 막으므로 동일 타입으로 강제하면(예: array, numpy) 빨라진다.
- 튜플의 불변성은 메모리 오버헤드가 적고 연산이 단순한 경량 구조를 만든다.
- 일반적(generic) 코드는 특정 문제용으로 설계된 코드보다 항상 느리다.

## 4. Lists as Dynamic Arrays (동적 배열로서의 리스트)

> 리스트에 append하면 Python은 향후 추가를 대비해 N+1개가 아니라 M(>N)개를 과할당(overallocation)하여 메모리 복사 횟수를 줄인다.

- 첫 append 시 더 큰 새 리스트를 만들어 기존 데이터를 복사하고 옛 리스트를 파괴한다.
- "한 번의 append는 여러 append의 시작"이라는 철학으로, 여분 공간을 미리 확보해 비싼 메모리 복사를 줄인다.
- Python 3.12의 과할당 공식은 `M = (N + (N >> 3) + 6) & ~3`로, 대략 12.5%를 추가 할당한다.
- 작은 리스트는 비율이 더 높아 1개일 때 4개, 9개일 때 16개를 할당한다.
- 과할당은 첫 append에만 발생하며, 리스트를 직접 생성하면 필요한 만큼만 할당된다.
- 작은 리스트가 백만 개면 과할당 오버헤드가 누적되어 상당한 메모리 낭비(예제에서 약 48MB)가 된다.

### 과할당 회피 패턴

```python
# list comprehension은 과할당됨
data_comp = [random_sample() for _ in range(1_000_000)]

# list()로 다시 감싸면 과할당 제거 → 메모리 절약
data_tight = list([random_sample() for _ in range(1_000_000)])

# 정적 데이터라면 tuple이 가장 경량
data_tuple = tuple(data_tight)
```

## 5. Tuples as Static Arrays & Caching (정적 배열과 캐싱)

> 튜플은 불변이라 리사이즈가 불가능하고 연결(concatenation)은 매번 새 튜플을 O(n)으로 복사 생성하지만, 여분 공간을 두지 않아 메모리를 절약한다.

- 1억 원소 리스트는 append로 만들면 약 104,391,068개 분량을 쓰지만, 같은 튜플은 정확히 1억 개 분량만 써 약 4.4% 절약한다.
- append 없이 만든 리스트도 리사이즈용 상태 정보 때문에 같은 데이터의 튜플보다 크다.
- 크기 0~20의 튜플은 사용이 끝나도 각 크기당 최대 2,000개까지 캐싱(freelist)되어 재사용된다.
- 이 캐싱 덕분에 새 튜플 생성 시 운영체제와 통신할 필요가 없어 매우 빠르다(리스트보다 약 7.6배 빠른 인스턴스화).
- CPython 3.12.2의 freelist 크기는 튜플이 40,000개로, 리스트(80)·딕셔너리(80) 등 다른 객체보다 압도적으로 크다.

## Summary (핵심 정리)

- 리스트와 튜플은 내재된 순서를 가진 데이터에 빠르고 저비용인 배열로, 순서를 알면 조회는 O(1), 정렬 시 탐색은 O(log n), 순서를 모르면 O(n) 선형 탐색이 필요하다.
- 리스트는 가변이지만 append 시 과할당으로 메모리를 더 쓰므로(필요 시 list() 재생성으로 완화), 정적 데이터에는 더 가볍고 빠르게 캐싱되는 튜플이 유리하다.
- 고성능의 핵심은 묻고자 하는 질문에 맞는 자료구조를 고르고 변환 비용 없이 일관되게 유지하는 것이며, 다음 장의 딕셔너리는 비순서 데이터의 탐색 문제를 오버헤드로 해결한다.
