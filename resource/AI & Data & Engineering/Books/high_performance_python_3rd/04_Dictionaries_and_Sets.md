# 04. Dictionaries and Sets

## 챕터 개요 (3줄 요약)

- 딕셔너리(dictionary)와 셋(set)은 내재된 순서가 없지만 해시 가능한(hashable) 키로 참조되는 데이터에 이상적이며, 해시 테이블 덕분에 조회·삽입이 O(1)이다.
- 이 O(1)은 오픈 어드레싱 해시 테이블(open address hash table)과 해시 함수로 달성되며, 메모리 오버헤드 증가와 해시 함수 품질 의존이라는 비용이 따른다.
- 해시 함수의 엔트로피(entropy)가 충돌(collision)을 좌우하므로, 나쁜 해시 함수는 조회를 O(n)까지 떨어뜨려 리스트보다 느려질 수 있다.

---

## 1. Why Dictionaries and Sets (왜 딕셔너리와 셋인가)

> 비순서 데이터를 고유 키로 참조할 때, 리스트의 O(n) 선형 탐색 대신 딕셔너리·셋은 임의 키로 O(1) 조회를 제공한다.

- 해시 가능한 타입은 `__hash__`와 `__eq__`(또는 `__cmp__`)를 구현하며, Python 기본 타입은 모두 이를 갖는다.
- 키(key)는 참조용 객체(보통 문자열)이고 값(value)이 데이터이며, 셋은 값 없이 고유 키만 모은 컬렉션이다.
- 전화번호부 예제에서 리스트는 전체를 훑어 O(n)이지만, 딕셔너리는 키로 직접 참조해 O(1)이다.
- 고유 이름 세기에서 리스트 방식은 내부 루프로 O(n log n)이지만, 셋의 `add`는 O(1)이라 전체가 O(n)이다.
- 1만 개 항목에서 셋 방식이 리스트 방식보다 약 560배, 10만 개에서는 약 4,300배 빨랐다.
- 셋은 합집합·교집합·차집합 같은 집합 연산(set operation)에 매우 유용하다.

### 리스트 vs 셋으로 고유값 세기 (원문 의도 유지, 새 예제)

```python
# 느림: 내부 루프로 O(n log n)
def count_unique_slow(names):
    seen = []
    for full in names:
        first = full.split(" ", 1)[0]
        if first not in seen:        # 매번 선형 탐색
            seen.append(first)
    return len(seen)

# 빠름: set.add는 O(1) → 전체 O(n)
def count_unique_fast(names):
    seen = set()
    for full in names:
        seen.add(full.split(" ", 1)[0])
    return len(seen)
```

## 2. How Hash Tables Work — Insert & Retrieve (해시 테이블 동작: 삽입과 조회)

> 키를 해시(hash)하고 마스크(mask)로 잘라 배열 인덱스로 변환한 뒤 해당 버킷에 저장하며, 충돌 시 프로빙(probing)으로 새 인덱스를 찾는다.

- 해시값을 마스크(예: 8버킷이면 `& 0b111`)로 잘라 할당된 버킷 수에 맞는 인덱스를 얻는다.
- 버킷이 비어 있으면 키와 값을 저장하고(조회 시 키로 검증), 키가 일치하면 이미 존재하는 것이다.
- 값이 다르면 충돌이므로, 상위 비트를 섞는 perturb 방식의 프로빙으로 다음 인덱스를 계산해 충돌을 분산시킨다.
- Python은 키/값을 별도 배열에 추가하고 해시 테이블엔 인덱스만 저장해 메모리를 30~95% 절약하며, 이 덕분에 3.7부터 삽입 순서가 보장된다.
- 조회도 같은 방식으로 인덱스를 만들어 키가 일치할 때까지 프로빙하며, 빈 버킷을 만나면 데이터가 없다고 결론짓는다.

### 프로빙 인덱스 시퀀스 (개념)

```
  i = hash(key) & mask          # 첫 인덱스
  loop:
    perturb >>= PERTURB_SHIFT   # 상위 비트 반영
    i = (i * 5 + perturb + 1) & mask
  # 빈 버킷이거나 키 일치까지 반복
```

## 3. Deletion & Resizing (삭제와 리사이즈)

> 삭제 시 NULL 대신 sentinel 값을 써서 충돌 체인을 보존하고, 테이블이 2/3 이상 차면 더 큰 2의 거듭제곱 크기로 리사이즈한다.

- 삭제된 버킷에 NULL을 쓰면 프로빙이 멈춰버리므로, "비었지만 뒤에 값이 있을 수 있음"을 뜻하는 특수 sentinel 값을 쓴다.
- 딕셔너리는 항상 2/3 미만(셋은 3/5 미만)으로 유지되며 크기는 항상 2의 거듭제곱이다.
- 최소 크기는 8이고, 2/3 이상 차면 현재 항목 수의 3배를 담을 최소 2의 거듭제곱으로 키운다.
- 리사이즈는 새 테이블 할당, 마스크 조정, 전 원소 재삽입(인덱스 재계산)이라 비싸지만, 가끔만 일어나 amortized O(1)이다.
- 리사이즈는 삽입 시에만 일어나므로, pop을 많이 한 딕셔너리에 새 원소를 추가하면 축소 리사이즈가 트리거되어 메모리를 절약할 수 있다.

## 4. Hash Functions and Entropy (해시 함수와 엔트로피)

> 해시 함수가 값을 고르게 분산할수록(엔트로피가 높을수록) 충돌이 줄며, 충돌이 많으면 조회가 O(n)까지 악화되어 리스트보다 느려진다.

- 정수의 해시는 자기 자신, 문자열·튜플은 내용 기반이며, 리스트는 값이 변할 수 있어 해시 불가능하다.
- 사용자 클래스의 기본 `__hash__`는 메모리 주소(id) 기반이라 같은 내용 객체도 다른 해시를 가지므로, 내용 기반 커스텀 해시가 필요할 수 있다.
- 엔트로피는 모든 해시값이 동등한 확률일 때 최대가 되며, 이를 최대화하는 함수를 이상적(ideal) 해시 함수라 한다.
- 무한 딕셔너리에서 정수 해시는 이상적이지만, 유한 딕셔너리에서는 마스크 때문에 일부 비트만 쓰여 충돌이 생긴다.
- 최악의 해시 함수(항상 같은 값 반환)는 조회를 약 54배 느리게 만들어 리스트보다도 느려진다.
- 대부분 커스텀 해시는 불필요하며, 객체의 핵심 속성을 튜플로 묶어 `hash(...)`에 넘기는 것으로 충분하다(문자열엔 SipHash 1-3 사용).

### 내용 기반 커스텀 해시 (원문 의도 유지, 새 예제)

```python
class Coordinate:
    def __init__(self, lat, lon):
        self.lat, self.lon = lat, lon

    def __hash__(self):
        # 메모리 주소가 아니라 내용으로 해시 → 동일 좌표는 동일 해시
        return hash((self.lat, self.lon))

    def __eq__(self, other):
        return self.lat == other.lat and self.lon == other.lon

# 같은 좌표는 set에서 하나로 취급됨
points = {Coordinate(1, 1), Coordinate(1, 1)}   # len == 1
```

## Summary (핵심 정리)

- 딕셔너리와 셋은 해시 테이블로 키 기반 O(1) 조회·삽입을 제공하지만, 메모리 오버헤드가 크고 성능이 해시 함수 품질에 크게 의존한다.
- 해시 테이블은 마스킹·프로빙으로 충돌을 처리하고 2/3 임계점에서 2의 거듭제곱으로 리사이즈하며, 삭제는 sentinel 값으로 충돌 체인을 보존한다.
- 좋은(고엔트로피) 해시 함수는 충돌을 최소화해 O(1)을 유지하지만 나쁜 해시는 O(n)으로 악화되므로, 커스텀 클래스를 키로 쓸 때는 내용 기반 해시를 신중히 정의하고 항상 프로파일링해야 한다.
