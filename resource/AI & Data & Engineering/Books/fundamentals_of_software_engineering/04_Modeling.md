# 04. Modeling

## 챕터 개요 (3줄 요약)
- 소프트웨어 모델(다이어그램)은 기술적 의도를 이해관계자에게 전달하는 의사소통 도구이며, 코드가 궁극적 진실의 원천이다.
- 컨텍스트, 컴포넌트, 클래스, 시퀀스, 배포, 데이터 모델 등 다양한 다이어그램이 있으며 청중과 목적에 맞게 선택한다.
- 단순하게 유지하고(Keep It Simple) 청중을 파악하며 색상에 주의하고 표준을 세우되, 다이어그램은 코드를 대체하지 않는다.

---

## 1. What Is Software Modeling and Why Do We Do It?
> 소프트웨어 모델링은 시스템을 이해/분석/소통하기 위한 추상적 표현을 만드는 과정이지만, 소프트웨어는 건물과 달리 쉽게 리팩터링할 수 있다.

- 소프트웨어 모델링은 시스템의 구조, 행동, 기능을 더 잘 이해하고 소통하기 위한 추상적 표현을 만드는 과정이다.
- 건물 청사진과 달리 소프트웨어는 적은 비용으로 빠르게 변경(refactoring)할 수 있어 다이어그램이 성공에 덜 결정적일 수 있다.
- 다이어그램은 컴파일되지 않으며, Jack Reeves의 "Code as Design"에 따르면 코드 자체가 궁극적 설계 산출물이다.
- 그럼에도 다이어그램은 맥락 제공, 복잡성 관리, 품질 속성(nonfunctional requirements) 예측에 유용하다.
- 시스템 학습, 신규 엔지니어 온보딩, 디버깅 시 도움이 되지만, 이는 다이어그램이 정확하고 최신일 때만 유효하다.
- 같은 다이어그램을 반복해서 그린다면 도구로 정형화하여 중앙에 저장하는 것이 좋다.

## 2. Which Diagrams Do You Need?
> 어떤 다이어그램을 언제 만들지는 "상황에 따라 다르며", 청중의 기술 수준에 따라 형식성(formal vs informal)을 조절한다.

- 어떤 다이어그램이 필요한지는 문제의 복잡도, 위험도, 예산, 비즈니스 중요도 등 상황에 따라 다르다.
- 모델은 형식적(formal, UML 등 기술 표기법) 또는 비형식적(informal)일 수 있다.
- 형식적일수록 청중이 더 기술적이어야 하므로 이해 가능한 사람의 범위가 좁아진다.
- 대부분의 엔지니어는 더 넓은 청중에게 도달하기 위해 UML 요소를 쓰되 비형식적 표기법을 선호한다.
- 단순한 것이 거의 항상 더 낫다(Simple is almost always better).

### UML(Unified Modeling Language)
- 1990년대 중반 여러 표기 체계를 통합하기 위해 개발되었고 2005년 ISO/IEC 19501 표준이 되었다.
- 채워진 다이아몬드 vs 빈 다이아몬드 같은 미묘한 차이를 청중이 모를 수 있어 비형식 표기를 선호한다.

### Context Diagrams
> 컨텍스트 다이어그램은 시스템 간 경계를 정의하는 가장 줌아웃된 최상위 뷰다.

- 시스템 또는 시스템 일부 간의 경계를 정의하고 환경 및 엔티티 상호작용을 보여준다.
- 아키텍트, 리더십, PM(Project Manager), 제품 책임자가 프로젝트 초기에 경계 합의를 위해 사용한다.

### Component Diagrams
> 컴포넌트 다이어그램은 런타임 시 시스템의 주요 요소와 상호작용/인터페이스를 보여준다.

- 시스템이 어떻게 함께 작동하는지 구조와 행동, 정보 흐름과 인터페이스를 보여준다.
- 개발자, 아키텍트, 운영 지원, DevOps 엔지니어가 사용하며 지식 전수와 온보딩에 유용하다.

### Class Diagrams
> 클래스 다이어그램은 클래스 간의 상속/합성 관계와 카디널리티를 보여준다.

- 상속(inheritance) 및 합성(composition) 관계와 카디널리티(cardinality)를 표현한다.
- 코드에서 추출되는 경우가 많고 금방 구식이 되며, 대형 시스템에서는 도메인 경계로 분할한다.
- 청중은 매우 기술적(엔지니어, 아키텍트, dev/ops)이다.

### Sequence Diagrams
> 시퀀스 다이어그램은 특정 흐름에 관여하는 엔티티 간의 상호작용 순서를 보여준다.

- 모든 엔티티가 아닌 특정 흐름에 관여하는 것만 보여주며, 아키텍처적으로 중요한 상호작용을 문서화한다.
- 코드에서 추출 가능하며 새 코드베이스 시작 시 유용하다. QA도 테스트 대상 이해에 활용한다.

### Deployment Diagrams
> 배포 다이어그램은 물리적 하드웨어 노드와 그 위에서 실행되는 소프트웨어의 런타임 뷰를 제공한다.

- 물리적 하드웨어 노드, 실행 소프트웨어, 연결 방식, 프로토콜, 카디널리티를 보여준다.
- 논리적/물리적으로 나뉘며 품질 속성(가용성, 확장성, 비즈니스 연속성) 검증에 사용된다.
- 조직은 표준 클라우드(AWS) 또는 온프레미스 참조 아키텍처를 두는 경우가 많다.
- 보안 다이어그램(security diagram)은 배포 다이어그램의 특수화로 HTTPS, LDAP 등 보안 메커니즘을 기술한다.

### Data Models
> 데이터 모델은 데이터 엔티티와 관계를 개념→논리→물리 수준으로 점진적으로 표현한다.

- 개념(conceptual) 모델은 최상위이고 정규화되지 않으며, 논리 모델은 정규형의 비즈니스 용어를 보여준다.
- 물리(physical) 데이터 모델은 데이터 타입 등 구현 세부사항을 보여준다.

```
Diagram Zoom Levels (most zoomed-out -> detailed):
  Context  -> system boundaries (high level)
  Component-> runtime elements & interfaces
  Class    -> class relationships
  Sequence -> interaction flow over time
  Deployment-> hardware nodes & software
  Data     -> conceptual -> logical -> physical
```

### Additional Diagrams
- Event storming: 색색의 스티키 노트로 도메인 이벤트/커맨드/액터를 탐색하는 워크숍 기법.
- Value stream mapping: 프로세스의 마찰/병목을 보기 위한 현재/미래 상태 분석.
- User story mapping: 스티키 노트와 스케치로 사용자 경험을 매핑.
- Disaster recovery models: 비즈니스 연속성 요구사항을 정의하는 상세 뷰.

## 3. Modeling Best Practices
> 모델은 강력하지만 통제하지 않으면 걷잡을 수 없어지므로, 단순함·청중·색상·표준의 원칙을 지킨다.

### Keep It Simple
- 다이어그램은 노이즈가 많아질 수 있으며, 충격을 주려는 과도한 다이어그램은 혼란만 전달한다.
- 모든 것을 한 다이어그램에 담지 말고 논리적 지점에서 분할하여 서로 참조하게 한다.

### Know Your Audience
- 코드 작성처럼 다이어그램을 소비하는 사람에 초점을 맞춘다.
- 다이어그램은 맥락(context)을 요구한다 — 청중이 도메인을 모르면 정교한 다이어그램도 평가받지 못한다.

### Be Careful with Your Color Choices
- 색상을 쓸 때는 반드시 범례(key)를 포함하고, 팀의 색상 관행을 모두가 공유한다고 가정하지 않는다.
- 누군가는 흑백 인쇄하거나 색맹일 수 있으므로 색상 외 대안을 둔다.

### Establish Standards and Templates
- 조직 내 표준 다이어그램과 색상 세트로 일관성을 확보하되, 지나치게 규정하지 말고 가감 여지를 둔다.

## 4. Tools
> 조직에 이미 도구가 있을 수 있지만, 메시지 전달이 핵심이며 도구는 표현을 돕는 수단일 뿐이다.

- OmniGraffle(Mac/iOS), Microsoft Visio(Windows): WYSIWYG 드래그앤드롭 다이어그램 도구.
- Mural, Miro, Lucidchart: 브라우저 기반 협업 화이트보드.
- Mermaid: Markdown 기반 텍스트로 다이어그램 생성, 문서 임베드에 인기.
- Structurizr(C4 모델 DSL), PlantUML(텍스트→UML): diagram-as-code 방식.
- IDE, 프레젠테이션 SW(PowerPoint/Keynote/Slides), AI 도구도 사용 가능(AI는 환각 주의).
- 모델에서 코드 생성(code generation)은 대부분 역사 속으로 사라졌으며, 다이어그램도 버전 관리해야 한다.
- 의심스러우면 단순하게 — 종이와 연필, 화이트보드도 훌륭한 도구다.

---

## Summary (핵심 정리)
- 모델과 다이어그램은 기술/비기술 이해관계자에게 핵심 개념을 전달하는 소프트웨어 엔지니어의 중요한 도구다.
- 컨텍스트~데이터 모델까지 청중과 목적에 맞는 다이어그램을 선택하고, 단순함·청중 이해·색상 주의·표준화 원칙을 지킨다.
- 다이어그램은 구식이 될 수 있으므로 코드를 궁극적 진실의 원천(source of truth)으로 삼고, 도구에 얽매이지 않는다.
