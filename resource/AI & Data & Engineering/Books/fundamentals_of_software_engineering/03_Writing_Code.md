# 03. Writing Code

## 챕터 개요 (3줄 요약)
- 좋은 코드란 컴퓨터가 아니라 사람이 읽을 수 있도록 작성된 코드이며, 프로그래밍은 본질적으로 의사소통 활동이다.
- 바퀴를 재발명하지 말고, 코드는 짧고 응집도 높게(high cohesion, low coupling) 작성하며, 영리한 코드보다 단순함을 추구해야 한다.
- 주석보다 테스트를 문서로 활용하고, 공감을 바탕으로 한 코드 리뷰와 새 언어 학습으로 지속적으로 성장한다.

---

## 1. Don't Reinvent the Wheel
> 새 코드를 작성하기 전에 이미 해결된 문제인지 먼저 확인하고, 가능한 라이브러리나 언어 기능을 활용하라.

- 개발자들은 이미 해결된 문제를 다시 푸느라 너무 많은 코드를 작성하는 경향이 있다.
- 작성 전 체크리스트: 현재 코드베이스, JDK(Java Development Kit), Spring Framework, Google Guava, Apache Commons, 기존/오픈소스 라이브러리 순으로 해결책 존재 여부를 확인한다.
- 모든 질문에 "아니오"라면 테스트와 함께 새 코드를 작성하고, 하나라도 "예"라면 기존 자산을 활용해 시간을 아낀다.
- 어제의 모범 사례가 오늘의 안티패턴이 될 수 있으므로 끊임없이 "왜?"를 질문하며 현상에 도전한다.
- 언어, 프레임워크, 기술은 계속 진화하므로 열린 관점으로 지속적으로 따라잡아야 한다.

```
Decision Tree: Should I write new code?
  In current codebase?      --yes--> reuse
  JDK does it?              --yes--> reuse
  Spring solves it?        --yes--> reuse
  Guava / Apache Commons?  --yes--> reuse
  Other libraries / OSS?   --yes--> reuse
  All NO --> write code (with tests!)
```

## 2. What Is Good Code?
> 좋은 코드는 주관적이지만 메트릭과 도구가 객관적 통찰을 제공하며, 맥락 없이 맹목적으로 따르면 안 된다.

- 좋은 코드와 나쁜 코드는 주관적이지만, 경험이 쌓이면 "냄새(smell)"로 감지할 수 있다.
- 순환 복잡도(cyclomatic complexity)는 코드 경로 수를 알려주어 리팩터링 대상을 식별하는 데 도움을 준다.
- PMD, SonarQube, JSHint, CodeScene 같은 소스 분석 도구로 버그와 나쁜 관행을 예방한다.
- 기존 프로젝트에 도구를 도입할 때는 모든 규칙을 켜지 말고, 팀이 합의한 소수 규칙부터 점진적으로 적용한다.
- 호손 효과(Hawthorne effect, 관찰되면 행동을 바꾸는 현상)를 활용해 코드 커버리지 통계를 공개하면 개선을 유도할 수 있다.
- 메트릭은 프로젝트 목표와 연결하고 추세(나아지는가/나빠지는가)에 집중하며 맹목적으로 따르지 않는다.

## 3. Less Is More
> 더 작은 코드베이스가 사람의 인지 부하를 줄이고 유지보수와 신규 개발자 온보딩을 쉽게 만든다.

- 코드가 적을수록 사람의 머리에 로드할 양이 줄어들며, 이것이 마이크로서비스 등장의 동력 중 하나다.
- 컴퓨터 과학 0번째 법칙: 높은 응집도, 낮은 결합도(high cohesion, low coupling).
- 응집도(cohesion)는 관련된 것들이 함께 있는 정도이고, 결합도(coupling)는 모듈 간 상호 의존성이다.
- 보일러플레이트 코드는 생성된 것이라도 프로젝트 수명 내내 짊어져야 하므로 피한다.
- 상속(inheritance)보다 합성(composition)을 선호하라 — 전기차 예시처럼 상속은 유연성이 떨어진다.
- 메서드는 한 가지 일만 하도록 짧게 작성하고, 이름에 접속사(and, or)가 들어가면 너무 많은 일을 한다는 신호다.

```
class Vehicle { Engine engine; ... }   // inheritance problem
class ElectricVehicle < Vehicle {
   // wait... EVs don't have engines!
}
=> Favor COMPOSITION over INHERITANCE
```

## 4. Write Code to Be Read
> 다음 개발자를 위해 의도(intent)와 구현(implementation)을 분리하고 신문 기사처럼 역피라미드 구조로 작성하라.

- 뒤따라올 개발자를 생각하며 코드에 골든 룰(Golden Rule)을 적용한다.
- 함수 이름만 읽고도 즉시 무엇을 하는지 이해할 수 있어야 하며, 이는 종종 한두 줄짜리 함수로 이어진다.
- 코드는 신문 기사의 역피라미드(inverted pyramid)처럼 작성해 독자가 원하는 깊이에서 멈출 수 있게 한다.
- 클래스 이름은 헤드라인, 메서드 시그니처는 본문 요약처럼 동작한다.
- 이름 짓기는 어렵다 — 시간을 들이고, 동료에게 묻고, 필요하면 "지비리시(gibberish) 게임"으로 임시 단어를 쓴 뒤 교체한다.
- IDE(Integrated Development Environment)의 후위 자동완성과 리팩터링 도구를 활용해 이름을 개선한다.

---

## 5. The Problem with Code Comments
> 주석은 DRY(Don't Repeat Yourself) 원칙을 위반하고 코드와 어긋나기 쉬우므로, 읽기 쉬운 코드가 더 낫다.

- 주석은 코드를 작성한 뒤 그 코드에 대해 다시 쓰는 것이므로 DRY 원칙을 위반한다.
- 코드를 수정하면서 주석을 갱신하지 않으면 주석이 코드와 점점 어긋나 유지보수 부담이 된다.
- 좋은 메서드/변수 이름이 주석보다 낫고, 주석이 필요하다고 느끼면 이름을 바꿔보라.
- 파일 상단의 변경 이력 블록은 노이즈이므로 SCM(Source Code Management) 도구에 맡긴다.
- 주석을 쓴다면 "무엇을"이 아니라 "왜"에 초점을 맞춘다 — 알고리즘 선택 이유 등.

## 6. Tests as Documentation
> 주석 대신 테스트를 작성하라 — 테스트는 프로덕션 코드와 함께 진화하는 실행 가능한 명세다.

- 테스트는 실행 가능한 명세(executable specification)로 코드와 함께 진화한다.
- 코드 작성과 동시에 테스트를 작성하면 자유롭게 리팩터링하고 품질에 대한 확신을 높일 수 있다.
- 기존 코드에 테스트를 추가하면 코드 동작에 대한 지식을 포착해 다른 개발자와 공유할 수 있다.
- 소비자 주도 계약(consumer-driven contracts)으로 서비스 동작을 전달하면서도 자유롭게 반복 개선할 수 있다.
- Spring Cloud Contract, Pact 등 다양한 플랫폼용 도구를 활용할 수 있다.

---

## 7. Avoid Clever Code
> 본질적 복잡성(essential)은 피할 수 없지만 우발적 복잡성(accidental)은 제거 대상이며, 의심스러우면 단순하게 유지하라.

- Fred Brooks의 "No Silver Bullet"은 복잡성을 본질적(essential) vs 우발적(accidental)으로 구분한다.
- 본질적 복잡성은 비즈니스 규칙 등 소프트웨어에 내재되어 제거할 수 없다.
- 우발적 복잡성은 개발자가 스스로 만든 불필요한 복잡성으로 적극 제거해야 한다.
- "혁신적"이라 주장하는 제품/프로세스에는 회의적이어야 한다(extraordinary claims require extraordinary evidence).
- Java의 중괄호 생략 if 문 같은 오류 유발 형태(error-prone forms)를 피하고 정적 분석 도구로 방지한다.

## 8. Code Reviews
> 코드 리뷰는 학습과 지식 공유의 기회이며, 비꼬지 말고 공감을 바탕으로 코드 리뷰 피라미드의 중요한 부분에 집중하라.

- 코드 리뷰는 배우고, 경험을 나누고, 지식을 사회화하는 훌륭한 방법이다.
- 비꼬거나 빈정대지 말고, 비판은 개인적으로 받아들여지기 쉬우므로 공감하며 전달한다.
- 코드 리뷰 피라미드(Code Review Pyramid) 하단의 스타일 이슈는 자동화하고, 컴퓨터가 못 잡는 것에 집중한다.
- 이름이 명확한지, 중복은 없는지, 로깅/추적/메트릭이 적절한지, 잘못된 추상화는 없는지를 본다.
- 체크박스 리뷰나 LFTM(Looks Fine To Me) 응답은 리뷰의 목적을 무효화한다.

### It Is Hard to Be Criticized
- 코드 리뷰는 코드에 관한 것이지 코더에 관한 것이 아니다 — 개인화하지 않는다.
- 비판은 칭찬으로 감싸면 더 받아들이기 쉽고, 겸손하게 도움이 되는 질문을 한다.
- 우려되는 점이 있으면 그룹이 아닌 일대일로 직접 대화한다.

### Fostering Trust
- 리뷰는 집단적 코드 소유(collective code ownership)와 팀 내 신뢰를 길러야 한다.
- "이주의 버그" 공유나 흥미로운 결함 토론이 강력한 학습 기법이 된다.

## 9. Learning New Languages
> 코드를 더 잘 작성하려면 코드를 써야 하며, 새 언어 학습은 외국어처럼 몰입을 통해 사고방식을 바꾼다.

- 실력 향상의 핵심은 몰입(immersion) — 새 언어로 실제 문제를 풀고 커뮤니티에 참여한다.
- 프로그래밍 언어는 도구일 뿐이며, 문제마다 적합한 언어가 다르다(예: 임베디드의 C).
- 연구에 따르면 수학 능력보다 언어 적성(language aptitude)이 프로그래밍 습득 속도를 더 잘 예측한다.
- 직장에서 쓰는 언어와 다른 패러다임(예: Ruby의 동적, Haskell의 함수형)을 골라 시야를 넓힌다.
- 언어를 많이 알수록 비교 대상이 늘어 새 언어 학습이 점점 쉬워진다.

---

## Summary (핵심 정리)
- 코드는 사람이 읽도록 작성하는 의사소통 활동이며, 바퀴를 재발명하지 말고 짧고 응집도 높은(high cohesion, low coupling) 코드를 지향한다.
- 영리한 코드와 오류 유발 형태를 피하고, 주석보다 테스트를 문서로 삼으며, 메트릭은 맥락과 함께 현명하게 사용한다.
- 공감 기반의 코드 리뷰와 꾸준한 새 언어 학습으로 평생 학습 과정으로서의 코딩 실력을 키운다.
