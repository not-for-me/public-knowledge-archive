# 01. Why Data Quality Deserves Attention—Now

## 챕터 개요 (3줄 요약)

- 데이터 다운타임(Data Downtime, 데이터가 누락/부정확/오류 상태인 기간)이 기업에 막대한 비용과 신뢰 손실을 초래하는 21세기 핵심 문제임을 제기한다.
- 데이터 품질(Data Quality)을 "데이터 라이프사이클 모든 단계에서의 데이터 건강 상태"로 정의하고, 왜 지금 최우선 과제가 되었는지 설명한다.
- 클라우드 이전, 데이터 소스 증가, 파이프라인 복잡화, 팀 분산화, 데이터 메시 등 데이터 품질 중요성을 끌어올린 산업 트렌드를 다룬다.

---

## 1. What Is Data Quality? (데이터 품질이란?)

> 데이터 품질은 데이터 라이프사이클의 어느 단계에서든 측정되는 데이터의 신뢰성·완전성·정확성, 즉 "데이터의 건강 상태"이다.

- 데이터 품질 개념 자체는 새롭지 않으나, 최근 수십 년에 걸쳐 측정 가능한 함수로 구체화되었다.
- "측정하지 못하면 관리할 수 없다"는 원칙에 따라, 높은 데이터 품질은 모든 견고한 분석 프로그램의 첫 단계이다.
- 데이터 품질은 데이터가 비즈니스 요구에 부합하는지를 이해하는 강력한 수단이다.
- 데이터 품질은 ingestion(수집) 이전, production(운영) 중, 또는 analysis(분석) 중 어느 단계에서도 영향을 받을 수 있다.
- "나쁜 데이터보다 데이터가 없는 게 낫다"는 격언이 통용되지만, 현실에서 항상 성립하지는 않는다.
- 데이터 정의를 명확히 하면 측정과 다운스트림 문제 예방이 훨씬 쉬워진다.

---

## 2. Framing the Current Moment (현재 시점의 맥락)

> 데이터가 단순 산출물을 넘어 금융 상품(financial commodity)이 되면서, 2020년대 들어 데이터 품질이 비즈니스의 최상위 우선순위가 되었다.

- 기업들은 데이터를 코드처럼 취급(treat data like code)하며 소프트웨어 엔지니어링 패러다임을 데이터 조직에 적용하기 시작했다.
- DevOps(Development Operations)의 베스트 프랙티스(SRE, CI/CD, 마이크로서비스)가 데이터 영역으로 확장되었다.
- 그 결과 DataOps가 등장했는데, 이는 자동화를 통해 데이터의 신뢰성·성능을 개선하고 데이터 사일로를 줄이는 과정이다.
- Intuit, Airbnb, Uber, Netflix 등은 2019년 이후 신뢰 가능한 데이터 확보를 위한 DataOps 실천을 적극 공유해왔다.
- 이들 거대 기술기업이 데이터 품질 중요성을 부각하면서, 모든 규모·산업의 기업이 이를 모방하기 시작했다.

---

## 3. Understanding the "Rise of Data Downtime" (데이터 다운타임 증가의 이해)

> 데이터 수익화와 정확도 요구가 커지면서, 데이터 다운타임을 유발하는 여러 구조적 요인을 이해할 필요가 있다.

### Migration to the cloud (클라우드 이전)
- 과거 사무실 지하에 있던 데이터 웨어하우스(구조화 데이터 저장·변환소)가 Redshift, Snowflake, BigQuery 등 클라우드로 이동했다.
- 데이터 레이크(비구조화 데이터 저장소)도 클라우드로 옮겨가며 유연성이 커졌고, 분석 기반 의사결정이 우선순위가 되었다.

### More data sources (데이터 소스 증가)
- 기업은 수십~수백 개의 내·외부 소스를 사용하며, 어느 하나라도 예고 없이 변경되면 의사결정용 데이터가 손상될 수 있다.

### Increasingly complex data pipelines (파이프라인 복잡화)
- 다단계 처리와 비자명한 의존성으로 파이프라인이 복잡해져, 한 데이터셋 변경이 의존 자산의 정확성을 해칠 수 있다.

### More specialized data teams (전문화된 데이터 팀)
- 데이터 분석가·과학자·엔지니어가 늘면서 협업 미스커뮤니케이션이 불가피해지고, 작은 변경이 파이프라인을 깨뜨릴 수 있다.

### Decentralized data teams (분산형 데이터 팀)
- 중앙 데이터 플랫폼 팀이 관리하되 분석팀이 비즈니스 전반에 분산된 탈중앙 아키텍처가 확산되어, 데이터 중복·누락·노후화가 발생하기 쉽다.

```
            +---------------------------+
            |  Central Data Platform    |
            |  (storage, catalog, ACL)  |
            +-------------+-------------+
                          |
        +-----------------+-----------------+
        |                 |                 |
   +---------+       +---------+       +---------+
   | Domain  |       | Domain  |       | Domain  |
   | Analyst |       | Analyst |       | Analyst |
   +---------+       +---------+       +---------+
   (distributed across functional teams)
```

## 4. Other Industry Trends Contributing to the Current Moment (현 시점을 만든 기타 산업 트렌드)

> 데이터 메시, 스트리밍 데이터, 데이터 레이크하우스 등 기술 혁신이 데이터 품질에 대한 관심을 높이고 있다.

### Data mesh (데이터 메시)
- Zhamak Dehghani가 정의한 사회기술적(sociotechnical) 패러다임으로, 도메인 지향·셀프서브 설계를 채택한다.
- 데이터를 "data-as-a-product"로 보고 각 도메인이 자체 파이프라인을 소유하며, 보편적 상호운용성 계층(interoperability layer)으로 연결한다.
- 데이터 메시는 데이터가 신뢰 가능할 때만 성공하며, 이는 테스트·모니터링·옵저버빌리티에 달려 있다.

```
   Domain A        Domain B        Domain C
  [data prod]     [data prod]     [data prod]
      |               |               |
  +------------------------------------------+
  | Universal Interoperability Layer         |
  | (shared syntax, standards, governance)   |
  +------------------------------------------+
```

### Streaming data (스트리밍 데이터)
- 연속적 데이터 흐름으로 실시간 인사이트를 만들지만, "움직이는 데이터"라 품질 검증 난이도가 높다.

### Rise of the data lakehouse (데이터 레이크하우스의 부상)
- 웨어하우스(구조화)와 레이크(비구조화)의 장점을 합친 모델로, 양쪽 경계가 좁아지며 파이프라인은 더 복잡해진다.

---

## Summary (핵심 정리)

- 클라우드, 분산 아키텍처/팀, 데이터 제품화(productization)가 데이터 리더에게 신뢰 가능한 데이터 확보 책임을 부여했다.
- 신뢰 가능한 데이터 달성은 단거리 경주가 아닌 마라톤이며, 파이프라인의 여러 단계를 아우른다.
- 데이터 품질 개선은 기술적 과제일 뿐 아니라 조직적·문화적 과제이기도 하다.
