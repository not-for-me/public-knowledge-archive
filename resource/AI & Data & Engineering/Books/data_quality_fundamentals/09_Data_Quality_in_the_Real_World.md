# 09. Data Quality in the Real World: Conversations and Case Studies

## 챕터 개요 (3줄 요약)

- 데이터 메시(data mesh), 클라우드 스택 여정, 지식그래프, 데이터 디스커버리 등 데이터 리더의 화두 5가지에서 데이터 품질의 역할을 다룬다.
- Zhamak Dehghani와의 대담으로 데이터 메시의 오해를 풀고, Kolibri Games의 5년 데이터 스택 진화 사례를 소개한다.
- 메타데이터를 비즈니스에 활용하는 법과 "언제 데이터 품질에 투자할지" 7가지 신호를 제시한다.

## 1. Building a Data Mesh for Greater Data Quality (데이터 메시 구축)

> 데이터 메시는 소프트웨어의 모놀리식→마이크로서비스 전환에 대응하는, 도메인 지향·셀프서브 데이터 플랫폼 아키텍처이다(Zhamak Dehghani 정의).

- Eric Evans의 도메인 주도 설계(domain-driven design)를 활용, "data as a product"로 각 도메인이 자체 파이프라인 소유.
- 3개 구성요소: 데이터 소스, 데이터 인프라, 도메인 지향 파이프라인 + 보편적 상호운용성·옵저버빌리티·거버넌스 계층.
- 도메인 데이터 소유자가 ingestion·cleaning·aggregation 책임, 중앙은 저장·카탈로그·접근제어 제공.
- 셀프서브: 도메인 무관 인프라 역량을 중앙 플랫폼으로 추출, 각 도메인이 커스텀 ETL 실행.
- 상호운용성: 포맷·거버넌스·디스커버리·메타데이터 표준화, 도메인 간 SLA·품질 보장.

### Why & 데이터 메시 점수
- 데이터 레이크의 한계(중앙 ETL의 통제력 부족, 다양한 변환 부하, 백로그된 데이터 팀) 해결.
- 데이터 메시 점수 = 데이터 소스 수 + 팀 규모 + 도메인 수 + 엔지니어링 병목(1-10) + 거버넌스 우선도(1-10).
  - 1-15: 불필요, 15-30: 일부 도입 권장, 30+: 데이터 메시 sweet spot.

```
   Domain A      Domain B      Domain C
  [pipeline]    [pipeline]    [pipeline]   (domain owners)
      |             |             |
  +-------------------------------------------+
  | Self-serve Infra + Universal Layer        |
  | (interoperability, observability,         |
  |  federated governance, standards)         |
  +-------------------------------------------+
```

## 2. A Conversation with Zhamak Dehghani (대담)

> 데이터 메시는 단일 기술이 아니라 분석 데이터를 수집·관리·공유하는 "사회기술적(socio-technical) 전환"이다.

- 단일 솔루션으로 못 만듦. 4대 요소: 데이터 소유권 분산, 장기 책임·제품 사고, 셀프서브 인프라, 연합 거버넌스(federated governance).
- 데이터 가상화(virtualization)와 다름 — 트랜잭션용 DB를 분석용으로 노출하는 건 나쁜 생각.
- 각 도메인이 별도 저장소를 갖는 게 아니라, 독립 스키마·접근권한·테넌시로 자율 배포(중앙 인프라 팀이 저장소 유지).
- 셀프서브 플랫폼이 곧 데이터 메시는 아님 — 메시용 플랫폼은 도메인 팀 자율성·제너럴리스트 권한 부여에 최적화.
- 모든 팀에 적합하진 않음(아직 nascent). 신뢰성 확장 문제를 겪는 엔지니어링 중심 조직이 적합. 톱다운 지원·1~3개 옹호 도메인 필요.
- 데이터 엔지니어-분석가 마찰을 줄임(보편적 거버넌스 표준으로 합의).

## 3. Case Study: Kolibri Games (5년 데이터 스택 진화)

> Berlin 모바일 게임사가 학생 아파트 스타트업(2016)에서 Ubisoft 인수(2020)까지 데이터 메시로 진화한 여정.

- First needs(2016): 기본 리포팅 목표, 3rd party 도구(Facebook Analytics, Firebase, GameAnalytics)만 사용 → 분산·불일치·투명성 부족.
- Performance Marketing(2017): AppsFlyer 추가, ROAS·LTV 계산 → 버전 관리·투명성 부재.
- Professionalize(2018): 데이터 엔지니어 채용, Azure 스택·중앙 웨어하우스, Power BI→Looker → SQL DB 한계.
- Data-Oriented(2019): Databricks·Snowflake 도입, 모니터링·A/B 테스트 → 직관 기반 의사결정 잔존.
- Data-Driven(2020): Ubisoft 인수, ML·데이터 SLA, ELT+dbt+Airflow, 임베디드 분석가 → 데이터 신뢰·확장 과제.
- Data Mesh(2021): 도메인 교차기능 팀, dbt·데이터 옵저버빌리티 투자, 엔드투엔드 리니지로 인시던트 해결 가속.

### 5가지 교훈
- 자체 데이터 스택 구축은 데이터 기반 역량을 부여하므로 가치가 있다.
- 데이터양·프로세스에 맞춰 기술 변경 시점을 이해해야 한다.
- 신뢰 확립엔 높은 옵저버빌리티가 필수(문제를 쉽게 인지).
- 고급 응용 전에 기본을 갖춰야 한다(분석가를 더 일찍 채용했어야).
- 데이터 기반 "문화" 확립이 올바른 기술 스택보다 더 중요할 때가 있다.

## 4. Making Metadata Work for the Business (메타데이터 활용)

> 메타데이터(데이터에 대한 데이터)도 맥락 없이는 무의미하다. 리니지는 비즈니스 응용에 연결될 때만 가치가 있다.

- 리니지의 실질 활용: (1)변경이 소비자에 미칠 영향 파악, (2)자산 깨짐 시 근본 원인 추적, (3)깨진 데이터 영향을 소비자에 소통.
- 메타데이터는 "이 데이터가 중요한가? 무엇을 의미하는가? 신뢰 가능한가? 누가 의존하는가?"에 답해야 한다.
- 자전거 탄 코끼리 비유: 사용 사례 없는 메타데이터는 인상적이나 무용. 핵심은 적시·적소 적용.

## 5. Unlocking the Value of Metadata with Data Discovery (디스커버리)

> 전통 데이터 카탈로그는 데이터 레이크·메시에서 한계를 보이며, 데이터 디스커버리(연합 카탈로그)가 이를 보완한다.

- 웨어하우스는 구조 강제로 카탈로그화 용이, 레이크는 비구조·분산으로 수동 거버넌스가 확장 안 됨.
- 전통 카탈로그는 수동·1차원적이라 진화하는 분산 자산을 못 따라감.
- 데이터 디스커버리(데이터 메시 기반 연합 카탈로그)의 보완점:
  - ML로 리니지 자동 추적·확장.
  - 실시간 데이터 건강 가시성("이상적 상태"가 아닌 현재 상태).
  - 리니지로 비즈니스 영향 파악.
  - 도메인 간 셀프서비스 디스커버리.
  - 레이크 전반 거버넌스·최적화(단일 진실 공급원 역할, 핵심/미사용 자산 식별).

## 6. Deciding When to Get Started with Data Quality (투자 시점)

> "언제 데이터 품질에 투자할까?"의 답은 "상황에 따라". 7가지 선행 지표가 투자 시점을 알려준다.

- 최근 클라우드로 마이그레이션함(신뢰·속도 유지 필요, AutoTrader UK 사례).
- 데이터 스택이 확장 중(소스·테이블·복잡도 증가, 대략 50개 테이블 이상이 가이드라인, Choozle table sprawl 사례).
- 데이터 팀이 성장 중("You're Using That Table?!" 문제 → 인증·리니지로 해결).
- 팀이 시간의 30% 이상을 firefighting에 소비(엔지니어가 혁신 대신 수정에 30~50% 소비).
- 데이터 소비자가 1년 전보다 많아짐(AutoTrader UK는 직원 50%+가 매월 Looker 사용).
- 셀프서비스 분석 모델로 전환 중(소비자가 신뢰해야 사용).
- 데이터가 고객 가치 제안의 핵심(Toast — 데이터가 고객 대면 제품, 핵심 제품 수준 신뢰성 요구).

## Summary (핵심 정리)

- 데이터 메시와 데이터 디스커버리 도구 등 고품질 달성의 핵심 기술·주제를 새로운 시각으로 조명했다.
- Kolibri Games는 스택을 처음부터 구축하며 자동화 솔루션·도메인 지향 접근으로 신뢰성을 사후 해결했다.
- 데이터 품질 투자의 7가지 신호를 제시했으며, 확장을 위한 데이터 전략 설계 시 이를 간과하지 말아야 한다.
