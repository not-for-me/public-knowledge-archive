# 10. Pioneering the Future of Reliable Data Systems

## 챕터 개요 (3줄 요약)

- 데이터는 비즈니스의 가장 경쟁력 있는 통화가 되었으나, 수집만으론 부족하고 신뢰가 필수임을 강조하며 책을 마무리한다.
- 데이터 품질 투자를 정당화하기 위한 데이터 다운타임 노동 비용 계산법을 제시한다.
- 웨어하우스·레이크 융합, 새로운 데이터 직무, 자동화, 분산 환경·데이터 도메인의 부상 등 4대 미래 트렌드를 예측한다.

---

## 1. Be Proactive, Not Reactive (반응이 아닌 선제적 대응)

> 데이터 신뢰성은 사후 대응이 아닌 사전 설계로 다뤄야 하며, 그 가치는 측정될 때 비로소 정당화된다.

- Unity Technologies(2022.5): stale 데이터가 광고 수익화 도구를 분기 이상 망쳐 주가 36% 폭락, $110M 손실.
- 데이터 다운타임: DDT = N × (TTD + TTR) (N=인시던트 수, TTD=탐지 시간, TTR=해결 시간).
- "돈이 떠나갈 때"야 좋은 데이터의 가치가 명확해짐 → 첫걸음은 재무 영향 측정.

### 연간 데이터 다운타임 노동 비용 계산
- 다운타임 시간 = (테이블 수 / 15) × (TTD + TTR), 최대 8,760시간(연간 총 시간).
- 성숙도별 TTD+TTR: 낮음 8시간, 평균 6시간, 높음 4시간.
- 비용 = 데이터 엔지니어 수 × 1,804(연 근무시간) × $62(시급) × 품질 대응 시간 비율.
- 비율: 낮은 성숙도 50%, 평균 35%, 높은 성숙도 20%.

```
   Example: 5,000 tables, avg maturity, 5 engineers, TTD+TTR=8h
   Downtime = (5000/15) * 8 = 2,664 hours/year
   Cost     = 5 * 1804 * $62 * 35% = ~$279,620 labor/year
   (기회비용은 미포함)
```

## 2. Predictions for the Future (미래 예측 4가지)

> 데이터 접근의 민주화가 진행되며 데이터 품질 해결 방식도 진화한다.

### Data Warehouses and Lakes Will Merge (웨어하우스·레이크 융합)
- AWS·Snowflake·Google·Databricks가 양쪽 장점을 결합(레이크하우스)하며 경계가 흐려짐.
- 품질은 웨어하우스가 유지 쉬움(스키마·볼륨·신선도 추적). 레이크는 유연하나 복잡도·깨짐 여지 큼.
- 융합은 표준화를 요구하여 중앙집중식 품질·옵저버빌리티의 문을 엶(ACID, Delta Lake).
- 단, 더 많은 사용자 → 중복·오류·downstream 소동 증가라는 복잡성도 동반.

### Emergence of New Roles (새로운 직무 등장)
- 전문화가 시장 성숙의 신호. 향후 10년 부상할 직무:
  - Data Product Manager: 데이터 제품 라이프사이클·로드맵·이해관계자 관리.
  - Analytics Engineer(dbt Labs 대중화): 엔지니어와 분석가 사이, 변환·모델링 담당.
  - Data Reliability Engineer: 옵저버빌리티·테스트로 회복력 있는 스택 구축(DevOps 역량).
  - Data Designer: 분석가와 협업해 BI 시각화로 데이터 스토리텔링.
- 팀·이해관계자 증가 → "보는 눈이 많아져" 이슈가 더 드러남. DRE 채용만으로 해결 안 됨.

### Rise of Automation (자동화의 부상)
- 자동화가 적용될 영역: 하드코딩 파이프라인 자동 수집, 단위 테스트·오케스트레이션 체크(서킷 브레이커), 스테이징→프로덕션 이전 자동화, 근본원인분석(RCA) 자동화, 문서화·카탈로깅·디스커버리 자동화.
- 문서화가 자동화되지 않으면 일어나지 않음("You're using that table?!" 문제).

### More Distributed Environments & Data Domains (분산 환경·데이터 도메인)
- 데이터 메시 등 도메인 지향 소유는 잠재력(빠른 접근·민주화)이 크나 복잡성도 큼.
- 한 도메인의 스키마 변경이 다른 도메인의 fire drill 유발 가능 → 기술·문화적 규율 필요.
- 보편적 연합 계층(federation layer)이라도 특정 계약·전용 API 준수가 복잡성을 낳음 → 교차조직 채택 가능성을 신중히 검토.

## So Where Do We Go from Here? (앞으로의 방향)

> 데이터 품질 달성은 더 쉬워지면서도 더 어려워질 것이다.

- 복잡한 시스템·대용량 데이터는 복잡성을, 기술 혁신은 자동화·예방 능력을 키운다.
- 어떻게 보든 데이터 신뢰성 추구는 가장 초보적인 데이터 팀에게도 필수 요건(table stakes)이 될 것이다.
- 데이터 리더들이 품질을 데이터 성숙도의 척도로 측정하고, 본서의 기술·접근으로 더 신뢰 가능한 시스템을 구축하길 기대한다.

---

## Summary (핵심 정리)

- 데이터 신뢰는 구축하기 어렵고 잃기 쉬우며, 투자 정당화는 재무 영향을 측정할 때 가능하다.
- 데이터 다운타임 노동 비용 계산으로 선제적 데이터 품질 투자의 근거를 마련할 수 있다.
- 웨어하우스·레이크 융합, 직무 전문화, 자동화, 분산 도메인이라는 4대 트렌드가 데이터 품질의 미래를 형성한다.
