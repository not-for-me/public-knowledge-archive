# 04. Bringing Data Consumers and Generators Closer Together

## 챕터 개요 (3줄 요약)

- 데이터 컨트랙트 도입의 핵심 목표는 데이터 소비자와 생성자를 가깝게 만들어 역할·책임을 명확히 하는 것이다.
- 책임을 생성자로 좌측 이동(shift-left)시켜 데이터 품질 문제를 원천에서 한 번만 고치도록 한다.
- 종종 간과되는 제품 엔지니어링팀(서비스)도 핵심 소비자이며, 데이터 진화(evolution)는 마찰(friction)을 두어 신중히 관리한다.

---

## 1. Who Is a Consumer, and Who Is a Generator?

> 소비자와 생성자는 각각 여러 페르소나(persona)로 나뉘며, 신뢰 가능한 데이터를 위해 서로의 기대를 이해해야 한다.

### Data consumers

- 데이터 소비자는 데이터를 소비해 행동·의사결정을 하는 사람·팀·서비스다.
- 소비자 페르소나: 소프트웨어 엔지니어(제품팀), 데이터 과학자, 데이터/BI/분석 엔지니어, 데이터 분석가·비즈니스 사용자.
- 각 페르소나는 고유한 접근 패턴(저지연 API, 과거 데이터, SQL 인터페이스, BI 툴 등)을 가진다.
- 소비자는 구조·문서·시맨틱, 신뢰성·성능, 소유권·지원 수준을 알아야 하며 이것이 곧 데이터 컨트랙트다.
- 소비자는 컨트랙트를 소유하지 않지만 요구사항을 명확히 전달해 컨트랙트 형성에 핵심 역할을 한다.

### Data generators

- 데이터 생성자는 후속 소비를 위해 데이터를 생성하는 사람·서비스다.
- 생성자 페르소나: 소프트웨어 엔지니어(제품팀), 데이터/BI/분석 엔지니어, 서드파티 서비스.
- 생성자는 데이터를 가장 잘 알고 품질·신뢰성을 개선할 수 있는 유일한 주체다.
- 항상 트레이드오프(형식·성능 vs. 비용·서비스 영향)가 존재하므로 생성자가 컨트랙트를 소유해야 한다.
- 생성자는 조직 정책에 따라 데이터 분류·접근 관리·보존 기간 관리 책임도 진다.

## 2. Assigning Responsibility and Accountability

> 책임을 데이터 엔지니어링팀에서 생성자(원천)로 좌측 이동시켜 데이터 품질을 사전 예방적(proactive)으로 관리한다.

- 기존에는 생성에 관여하지 않은 데이터 엔지니어링팀이 신뢰성 책임을 떠안아 반응적(reactive)이었다.
- 원천에서만 정확성·신뢰성을 완전히 통제할 수 있다: 원천에 없으면 다운스트림 어디에도 없다.
- 책임 좌측 이동은 인시던트 빈도와 영향을 줄여 전체 비용을 낮춘다.
- 원시 데이터 소비 시(Figure 4.1): 무효 데이터가 모든 파이프라인에 영향을 줘 중복 수정이 필요하다.
- 컨트랙트 소비 시(Figure 4.2): 생성자가 한 곳에서 수정하면 모든 다운스트림이 혜택을 받는다.
- 소비자에게도 책임이 있다: 투자에 대한 보답으로 영향력 있는 데이터 제품을 제공해야 한다.

```
Raw consumption:   [bad data] -> pipe1 fix / pipe2 fix / pipe3 fix   (duplicated effort)
Contract:          [generator fixes once] -> all consumers benefit
```

## 3. Feeding Data Back to the Product Teams

> 제품팀과 그들의 서비스는 조직에서 가장 크고 중요한 데이터 소비자이며, 컨트랙트는 '데이터를 위한 API'로 작동한다.

- 서비스는 데이터를 입력받아 처리·행동 후 새 데이터를 출력하므로 다른 소비자와 동일한 기대를 가진다.
- 동기 통신(synchronous): HTTP 기반 API가 인터페이스와 기대치를 정의한다.
- 비동기 통신(asynchronous): Apache Kafka, Google Cloud Pub/Sub 같은 메시지 브로커를 통한 이벤트 기반 아키텍처가 늘고 있다.
- 데이터 컨트랙트는 동일한 목표(신뢰성·기대치 설정)를 가져 '데이터를 위한 API'로 불린다.
- GoCardless에서는 서비스 간 비동기 통신의 약 80%가 데이터 컨트랙트로 인터페이스를 정의한다.
- 이렇게 개방된 데이터는 더 많은 핵심 제품에 쓰여 경쟁 차별화와 방어력(defensibility)을 높이는 자산이 된다.

```
Service A --(data contract = API for data)--> Message Broker --> Service B / Data Team
```

## 4. Managing the Evolution of Data

> 데이터는 조직처럼 진화하지만 핵심 모델은 안정적이므로, 변경 유형에 따라 적절한 마찰을 두어 관리한다.

- 핵심 데이터 제품은 공개 API처럼 수년간 거의 바뀌지 않으므로 진화 시 약간의 마찰은 바람직하다.
- 비파괴 변경(non-breaking change): 새 필드 추가 등 기존 소비자에게 영향 없음 → 마찰 최소.
- 파괴 변경(breaking change): 필드 제거·대규모 리팩터 등 소비자 변경 필요 → 충분한 마찰과 사전 통보 필요.
- 파괴 변경에는 문서화된 마이그레이션 경로가 있어야 하며, 두 컨트랙트를 일정 기간 병행 운영할 수 있다.
- 대규모 조직에서는 데이터 리니지(data lineage) 도구로 다운스트림 소비자를 파악해 마이그레이션을 계획한다.
- 마이그레이션 투자 수준은 데이터셋의 중요도와 사전에 설정한 기대치(SLO 등)에 따라 결정된다.

---

## Summary (핵심 정리)

- 소비자와 생성자의 역할·책임·기대를 명확히 정의해 두 그룹이 가깝게 협업하도록 함으로써 데이터 접근성과 품질을 높인다.
- 제품팀은 핵심 소비자이며, 명확한 기대치와 함께 데이터를 되먹임하면 차별화된 데이터 주도 제품을 만들 수 있다.
- 데이터 진화에는 중요도에 비례한 마찰을 두며, 다음 장에서는 컨트랙트에 데이터 거버넌스를 내장하는 방법을 다룬다.
