# 03. How to Get Adoption in Your Organization

## 챕터 개요 (3줄 요약)

- 데이터 컨트랙트는 기술 아키텍처일 뿐 아니라 조직의 데이터 문화를 바꾸는 '도구(vessel)'이며, 문화 변화가 도입 성공의 핵심이다.
- 데이터 생성자(generator)와 소비자(consumer) 두 역할이 모두 변화해야 하며, 소비자는 데이터 가치를 명확히 설명해야 한다.
- 원시 테이블이 아닌 '데이터 제품(data product)'을 컨트랙트 인터페이스로 제공해 병목을 제거하고 신뢰를 구축한다.

---

## 1. Using Data Contracts to Change an Organization

> 데이터 컨트랙트의 본질은 기술 구현보다 조직의 데이터 문화 변화를 촉진하는 데 있으며, 생성자와 소비자 모두의 변화를 요구한다.

- 두 핵심 역할: 데이터 생성자(데이터를 만드는 제품 엔지니어링팀, 데이터 소유자)와 데이터 소비자(데이터 엔지니어·분석가·과학자 등).
- 생성자는 더 큰 자율성과 책임을 부여받는 '새로운 역할'을 맡게 되어 상당한 변화를 겪는다.
- 솔루션 설계 전 가능한 한 일찍 이해관계자를 참여시켜 주인의식과 buy-in을 확보해야 한다.
- 모든 우려(concern)를 진심으로 받아들이고 공감(empathy)으로 대응해 신뢰 관계를 쌓는다.
- 소비자도 변해야 한다: 필요한 데이터를 명확한 이유와 함께 '요청'하는 새로운 기술을 익혀야 한다.
- 생성자와 소비자는 데이터로 가치를 창출하는 파트너 관계가 된다.

---

## 2. Articulating the Value of Your Data

> 책임을 생성자(제품팀)로 좌측 이동(shift-left)시키려면, 소비자가 데이터가 만드는 비즈니스 가치를 설득력 있게 설명해야 한다.

- '왜(why)' 투자하는지 명확히 하지 못하면 생성자는 최소한의 노력만 하게 된다.
- 청중에 따라 가치 전달 방식이 다르다: 리더에게는 전사 전략·의사결정 개선·예측을 강조한다.
- 빠른 의사결정(실시간 자동화)과 ML(Machine Learning) 기반 제품의 매출 목표로 가치를 표현할 수 있다.
- 달러로 측정이 어려우면 뉴스레터·발표로 데이터 주도 성과를 가시화한다.
- 비즈니스 문제에서 출발해 거꾸로 작업하고, 중요도·가치 순으로 우선순위를 매긴다.
- 고가치 영역에서 먼저 개념을 증명(prove the concept)해 데이터 컨트랙트의 영향을 입증한다.

## 3. Building Data Products

> 원천 시스템의 원시 테이블 대신 '소비를 위해 설계된 고품질 데이터 제품'을 만들어 병목을 제거한다.

### What is a data product?

- 데이터 제품은 타인의 소비를 위해 설계된 고품질 데이터셋으로, 신뢰하고 구축할 수 있어야 한다.
- 단일 비즈니스 도메인으로 범위가 한정되며, 그 도메인 팀이 소유·공급한다.
- 자체로 가치가 있고, 발견 가능(discoverable)·접근 가능(accessible)하며 안정적 인터페이스와 문서를 갖춘다.
- 중앙 데이터팀만이 아니라 데이터를 소유한 도메인이 제품을 생산해 책임의 좌측 이동을 촉진한다.
- 탈중앙화된 데이터 제품은 병목을 없애고 다크 데이터(dark data)를 활용 가능하게 한다.

### Adopting & designing a data product

- 데이터 제품 마인드셋: "누구를 위한 것인가, 요구·기대는 무엇인가"를 먼저 이해한다.
- 데이터 공급망(supply chain)은 짧고 효율적으로 유지해 어떤 팀도 병목이 되지 않게 한다.
- Data Product Manager 역할을 두어 소비자 요구를 이해하고 팀과 협업하도록 할 수 있다.
- 데이터 제품은 원천 DB의 내부 모델이 아닌 비즈니스 엔티티/도메인 기준으로 설계한다.
- 컨트랙트는 버전 관리되며, 무중단·무손실 마이그레이션 경로를 제공한다.

```
Source System ---explicitly produces---> [Data Product (contract interface)] ---> Consumers
   (domain owns the product, shift-left of responsibility, no central bottleneck)
```

## 4. Walking Through an Example of a Data Product

> e커머스 할인(discount) 데이터를 사례로, 원시 DB 모델 위에 쌓은 복잡한 파이프라인을 데이터 제품으로 전환하는 과정을 보여준다.

- 문제 상황: 할인 모델이 여러 번 바뀌어 products·discounts 테이블에 분산되고, CDC(Change Data Capture)로 그대로 복제되었다.
- 소비자는 800줄 넘는 CASE/IFNULL SQL로 비즈니스 로직을 떠안아 비싸고 깨지기 쉬운 파이프라인을 운영한다.
- 업스트림 변경 알림이 없어 사용자가 먼저 오류를 발견하고 신뢰를 잃는다.
- 해결: 생성자가 새 인터페이스(order_events)로 데이터를 명시적으로 공급해 내부 모델과 소비자를 디커플링한다.
- 컨트랙트는 YAML로 표현되며 owner, description, version, fields(스키마)를 포함한다.
- 합의된 SLO 예시: 완전성(completeness) 100%, 적시성(timeliness) 60분, 가용성(availability) 95%.
- 결과: 파이프라인이 단순·저렴·신뢰성 향상되고, 사용자가 데이터를 신뢰해 더 나은 의사결정으로 이어진다.

```
slos:
  completeness_percent: 100
  timeliness_mins: 60
  availability_percent: 95
  lakehouse_path: order_events
```

## Summary (핵심 정리)

- 데이터 컨트랙트는 단순 기술이 아니라 더 나은 비즈니스 성과를 위해 데이터에 투자하는 문화 변화의 매개체다.
- 소비자는 데이터 가치를 명확히 설명하고, 생성자는 소비자 요구를 충족하는 데이터 제품을 만들며 책임을 더 진다.
- 다음 장에서는 생성자와 소비자를 더 가깝게 만드는 방법을 다룬다.
