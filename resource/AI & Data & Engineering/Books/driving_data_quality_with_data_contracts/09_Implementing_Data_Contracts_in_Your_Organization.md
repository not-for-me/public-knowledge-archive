# 09. Implementing Data Contracts in Your Organization

## 챕터 개요 (3줄 요약)

- 강력한 사용 사례로 개념 증명(POC)을 하고, 이를 지원하는 최소 데이터 컨트랙트 구현(MVP)부터 시작한다.
- 나머지 데이터 자산을 컨트랙트로 마이그레이션하고, 데이터 카탈로그·데이터 리니지로 발견 가능성을 확보한다.
- 데이터 제품·연합형 거버넌스를 정착시켜 성숙한 데이터 컨트랙트 기반 데이터 문화를 구축한다.

---

## 1. Getting Started with Data Contracts

> 핵심 목표를 정하고 사용 사례로 POC(Proof of Concept)를 수행하며, 세 가지 최소 툴링(정의·인터페이스·쓰기)으로 MVP(Minimum Viable Product)를 구축한다.

- 첫 단계는 핵심 목표 설정: 파이프라인 신뢰성 향상, 데이터 접근성·핵심 애플리케이션 활용 등.
- POC에 적합한 사용 사례와 함께 생성자·소비자 등 적절한 인력·리소스를 확보한다.
- 툴링 설계 시 탈중앙화·소유권 목표를 지원하고 생성자가 셀프서브할 수 있어야 한다.
- 최소 구현에 필요한 세 부분: 컨트랙트 정의, 소비자용 인터페이스 프로비저닝, 생성자의 데이터 쓰기.

### The ability to define a data contract

- 정의 언어는 생성자에게 익숙한 것을 선택(GoCardless는 Jsonnet, 없으면 YAML 추천).
- 저장 위치도 생성자가 이미 있는 곳(예: 인프라 코드 저장소 또는 생성자 코드 저장소)이 좋다.
- 최소 캡처 항목: 이름, 소유자, 스키마, 인터페이스 위치 + 버전 번호, 개인 데이터 분류.
- 컨트랙트 정의 자체도 버전 관리해 전체 마이그레이션 없이 정의의 파괴 변경을 가능하게 한다.

### Provisioning the interface & writing data

- 인터페이스는 보통 웨어하우스/레이크하우스 테이블이나 Kafka/Pub/Sub 토픽이며 기존 플랫폼을 활용한다.
- 생성자가 인터페이스 배포를 셀프서브로 하되, 툴링이 휴먼 에러를 방지한다.
- 데이터 쓰기: 직접 쓰기가 가장 쉬우나, 소스 시스템과의 일관성이 필요하면 트랜잭셔널 아웃박스(transactional outbox) 패턴을 쓴다.
- 아웃박스 패턴은 내부 모델과 outbox 이벤트를 동일 트랜잭션에서 써서 일관성을 보장한다.
- 대안으로 CDC(Change Data Capture) 기반 머티리얼라이즈드 뷰가 있으나 장기적으로는 비권장이다.

```
Service --(same DB transaction)--> [internal models] + [outbox table]
outbox --(later)--> [data contract interface] --> Consumers
```

## 2. Migrating to Data Contracts

> POC 이후 나머지 데이터 자산을 마이그레이션하며, 속도와 제품팀 로드맵의 균형을 맞춘다.

- 접근법 ① 핵심 소비자(데이터/분석 엔지니어, 데이터 과학자)가 워킹그룹으로 중요 데이터셋 우선순위를 정한다.
- 접근법 ② 마이그레이션 마감일 설정(단, 생성자가 목표에 공감하지 않으면 형식적 전환에 그칠 위험).
- 공유 소유권 데이터셋은 우선 데이터를 생성한 서비스의 소유 팀에 책임을 할당한다.
- 진행률·인시던트 감소·ETL 비용 절감 등을 측정하고 뉴스레터로 영향을 정기 공유한다.
- 거버넌스 자동화, 백업, 스키마 진화 마이그레이션 등 셀프서브 툴링을 계속 반복 개선한다.
- 데이터 카탈로그·데이터 리니지 같은 중앙 서비스를 추가하고, 데이터 인프라팀 구성을 정당화한다.

---

## 3. Discovering Data Contracts

> 소유권 분산이 커질수록 데이터 카탈로그와 데이터 리니지(data lineage)가 발견·접근에 필수가 된다.

### Data catalog

- 데이터 카탈로그는 도메인·소유자와 무관하게 모든 데이터 제품을 검색·발견하는 중앙 인벤토리다.
- 소유자, 문서, 포맷·스키마, 기대치(SLO 등), 저장 위치, 접근 제어, 컴플라이언스 등 컨텍스트를 제공한다.
- 컨트랙트로 카탈로그를 채워 데이터를 격리(isolated)하되 사일로화(siloed)되지 않게 하고 셀프서브를 촉진한다.

### Data lineage

- 데이터 리니지는 데이터 제품의 상·하류 관계와 ML/리포트 활용까지 추적한다.
- 영향 분석(스키마 진화·인시던트 영향), 컴플라이언스, 소비자 신뢰 확보에 유용하다.
- 탈중앙화로 소유권이 늘고 공급망 단계마다 소유자가 바뀌므로, 리니지로 각 단계 소유자를 쉽게 찾는다.
- 많은 도구가 카탈로그 기능도 포함해 두 요구를 한 도구로 충족할 수 있다.

## 4. Building a Mature Data Contracts-Backed Data Culture

> 데이터 컨트랙트를 조직 전략·아키텍처의 명시적 부분으로 삼고, 데이터 제품·조직구조·거버넌스를 성숙시킨다.

- 모든 유용한 데이터를 컨트랙트로 제공하고, 데이터 제품 매니저를 두어 데이터 제품을 책임진다.
- 도메인별 데이터 제품이 많아지면 도메인에 임베드된 전문 데이터 엔지니어링팀(cross-functional team)을 구성한다.
- 임베드된 엔지니어는 도메인 모델링으로 레이크하우스에 최적화된 데이터 제품을 만들어 소비 비용을 줄인다.
- 거버넌스는 중앙에서 정책을 정하고 책임은 로컬 생성자에게 두되, 툴링으로 대부분 자동화한다.
- Chapter 2의 네 원칙(합의된 인터페이스·기대치·거버넌스·명시적 품질 데이터 생성)을 충족해 진정한 데이터 주도 조직이 된다.

---

## Summary (핵심 정리)

- 적합한 사용 사례로 개념을 증명하고 MVP 툴링(정의·인터페이스·쓰기)으로 시작한 뒤 나머지 데이터셋을 마이그레이션한다.
- 소유권 분산에 대응해 데이터 카탈로그와 데이터 리니지로 발견 가능성·접근성을 유지한다.
- 데이터 제품·조직구조·자동화된 거버넌스를 성숙시켜 데이터 컨트랙트 기반 문화를 완성하고, 다음 장에서 실무를 다룬다.
